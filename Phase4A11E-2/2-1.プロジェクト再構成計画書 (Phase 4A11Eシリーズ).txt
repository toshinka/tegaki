プロジェクト再構成計画書 (Phase 4A11Eシリーズ)
🎯 計画の最終目標
AIとの協業効率の最大化: 曖昧な指示をなくし、AIが正確なコードを生成できる環境を構築する。

保守性と拡張性の飛躍的向上: 機能追加や仕様変更に強く、バグの少ない堅牢なアーキテクチャを実現する。

段階的な移行: アプリケーションが動作不能になる期間を最小限に抑え、安全にリファクタリングを遂行する。

🗺️ 全体ロードマップ
リファクタリングは、**「設計定義 → 骨格構築 → 機能移行」**の順で、依存関係の根幹から段階的に進めます。

Phase 1: 設計定義
工程ID: 4A11E-1
目的: 新アーキテクチャにおける各クラス・モジュールの役割と責任範囲を明確に文書化する。これが今後のすべての作業の基礎となる。

作業内容:

【統合版】プロジェクト改善計画に基づき、主要な新クラス（WebGLRenderer, CanvasViewport, LayerStore, ToolStoreなど）の責務を定義する。

各クラスが持つべき主要なプロパティ（状態）とメソッド（操作）をリストアップする。

クラス間の基本的な連携方法（データの流れ）をダイアグラムや文章で記述する。

成果物:

クラス責務定義書: 各クラスの役割、メソッド、プロパティを詳細に記述したMarkdownドキュメント。

参照資料:

【統合版】プロジェクト改善計画：現状の課題と新アーキテクチャ提案 改訂版.txt

Phase 2: 骨格構築
工程ID: 4A11E-2
目的: 新しいディレクトリ構造を物理的に作成し、空のファイルを配置する。コードを移行するための準備を整える。

作業内容:

srcディレクトリ内に、app, engine, features, ui, data, utilsの各ディレクトリを作成する。

features内にlayers, tools, historyなどのサブディレクトリを作成する。

4A11E-1で定義した各クラスに対応する空のJavaScriptファイルを作成し、配置する。(例: src/engine/WebGLRenderer.js)

成果物:

提案されたアーキテクチャ通りの、空ファイルが配置された新しいディレクトリツリー。

Phase 3: エンジン移行
工程ID: 4A11E-3
目的: アプリケーションの心臓部である描画エンジンとビューポート管理を、新しいクラスに移行する。

作業内容:

旧webgl-engine.js, rendering-bridge.jsのロジックを、新しいengine/WebGLRenderer.jsに移植・再構成する。twgl.jsを全面的に活用する。

旧canvas-manager.jsの拡大・縮小・回転ロジックを、新しいengine/CanvasViewport.jsに移植する。

旧drawing-engine.jsの入力処理ロジックを、新しいfeatures/canvas/CanvasInteraction.jsに移植する。

成果物:

WebGLによるレイヤー合成と表示機能を持つWebGLRendererクラス。

ズーム・パン・回転機能を持つCanvasViewportクラス。

ポインタイベントを処理するCanvasInteractionクラス。

(以降のフェーズも同様に、目的、作業内容、成果物を定義して進めます。ここでは代表的なものを記載しました)


Phase 4: 状態管理導入 (Store/Actionsパターンの実装)
工程ID: 4A11E-4
目的: アプリケーション内に散在していた状態（現在のツール、アクティブレイヤー等）を、状態管理を一元的に担うStoreモジュールに集約する。これにより、誰が状態を変更したかが明確になり、データの流れが予測可能になる。

作業内容:

状態管理の基盤作成: 状態の変更を監視（購読）できるシンプルな仕組み（Observerパターン）を実装する。Storeが変更されたら、それを購読しているモジュールに通知が飛ぶようにする。

Storeの作成:

features/layers/LayerStore.js: レイヤーのリスト、レイヤーの表示/非表示、アクティブレイヤーIDなどの状態を管理する。

features/tools/ToolStore.js: 現在のツール、ペンサイズ、色、不透明度などの状態を管理する。

features/history/HistoryStore.js: Undo/Redoのための状態履歴を管理する。

Actionsの作成:

Storeの状態を変更するための唯一の窓口となるActionsファイル群を作成する。(例: LayerActions.js, ToolActions.js)

LayerActions.jsにはaddLayer(), setActiveLayer(layerId)のような関数を定義する。これらの関数が呼ばれると、対応するLayerStoreの状態が更新される。

成果物:

アプリケーションの状態を一元管理するLayerStore, ToolStore, HistoryStoreモジュール。

状態を変更するための操作を定義したLayerActions, ToolActionsモジュール。

状態の変更を検知するための基本的な購読メカニズム。

Phase 5: 機能移行 (新アーキテクチャへのロジック移植)
工程ID: 4A11E-5
目的: Phase 3で作成した新エンジンと、Phase 4で作成した状態管理システムを使い、アプリケーションのコア機能（描画、レイヤー操作など）を新しい設計思想で再実装する。

作業内容:

描画ロジックの再構築:

features/canvas/CanvasInteraction.jsでユーザーの描画入力（ポインターイベント）を受け取る。

入力が完了したら、直接キャンバスに描画するのではなく、ToolActions.drawStroke(points)のようなアクションを呼び出す。

DrawingEngineはToolStoreを監視しており、drawStrokeアクションによってToolStoreの状態が変化したら、その情報（座標、ペン設定）を元にWebGLRendererに描画を指示する。

レイヤー操作ロジックの再構築:

レイヤーの追加、削除、順序変更などの操作を、すべてLayerActionsを介して行うように書き換える。

WebGLRendererはLayerStoreを監視し、レイヤーリストに変更があれば自動的に再描画する。

Undo/Redo機能の実装:

描画やレイヤー操作が完了するたびに、HistoryActions.saveState()を呼び出し、現在のアプリケーション状態をHistoryStoreに保存する。

成果物:

Store/Actionsパターンに基づいて動作する、新しい描画ロジックとレイヤー管理ロジック。

（この時点ではUIからの操作はできないが）コード上でActionsを呼び出せば、描画やレイヤー操作が正しく実行される状態。

Phase 6: UI連携 (アプリケーションの完成)
工程ID: 4A11E-6
目的: HTMLのUI要素（ボタン、スライダー等）と、新しい状態管理システムを接続する。これにより、ユーザー操作がアプリケーションに反映され、リファクタリングが完了する。

作業内容:

UIコントローラーの実装: ui/UIController.jsを作成し、UIとアプリケーションロジックの間の「通訳」の役割を担わせる。

UIからのイベント発信:

UIControllerで、HTMLの全UI要素にイベントリスナーを登録する。

ユーザーがボタンをクリックしたり、スライダーを動かしたりしたら、対応するActionsを呼び出す。(例: ペンツールボタンクリック → ToolActions.setCurrentTool('pen'))

UIへの状態反映:

UIControllerは、すべてのStoreを監視（購読）する。

Storeの状態が変化したら、UIControllerはそれを受け取り、HTMLの表示を更新する。(例: LayerStoreにレイヤーが追加されたら、レイヤーリストのDOMを生成して表示する)

最終クリーンアップ:

すべての機能が新アーキテクチャ上で動作することを確認した後、旧coreディレクトリ、旧managerファイル群をすべて削除する。

成果物:

新アーキテクチャ上で完全に動作する、インタラクティブなお絵かきツール。

UIの表示がアプリケーションの状態と常に同期している状態。

旧構造のファイルが完全に排除された、クリーンでメンテナンス性の高いプロジェクト。

# 指示書: 4A11E-1 - 新アーキテクチャの責務定義

## 依頼内容
添付の「【統合版】プロジェクト改善計画」ドキュメントに基づき、新しいアーキテクチャで採用する主要なクラスとモジュールの「責務定義書」を作成してください。

## 成果物フォーマット
- Markdown形式で出力してください。
- 各クラスについて、以下の項目を記述してください。
    - **クラス名**: (例: `WebGLRenderer`)
    - **配置パス**: (例: `src/engine/WebGLRenderer.js`)
    - **責務 (一言で)**: このクラスが持つ唯一の責任を簡潔に記述してください。
    - **主要プロパティ**: クラスが保持すべき主要な状態やデータをリストアップしてください。(型も明記)
    - **主要メソッド**: クラスが提供すべき主要な操作をリストアップし、各メソッドの役割、主要な引数、戻り値を記述してください。
    - **連携する主要クラス**: このクラスが直接やり取りする他の主要なクラスをリストアップしてください。

## 対象クラスリスト
以下のクラスについて、上記のフォーマットで定義を作成してください。

### engine
- `WebGLRenderer`
- `CanvasViewport`
- `DrawingEngine`

### features
- `LayerStore`
- `LayerActions`
- `ToolStore`
- `ToolActions`
- `HistoryStore`
- `CanvasInteraction`
- `CoordinateSystem`

### ui
- `UIController`

### data
- `StorageService`

## 補足
- このドキュメントは、今後の開発における「設計図」そのものになります。
- 各クラスの責務は、「単一責任の原則」を厳格に守るように定義してください。
- メソッド名は、その操作内容が明確にわかるように命名してください。
