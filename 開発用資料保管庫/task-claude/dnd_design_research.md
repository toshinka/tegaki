# Claude向け調査依頼 — レイヤーパネル/アルバム独自D&D設計

## 目的

Tegaki のレイヤーパネル移動、フォルダへの入れ込み、アルバム並べ替えを、より滑らかで破綻しにくい独自D&Dへ移行できるか調査する。

今回は実装しない。後続Phaseで Codex / Gemini が安全に実装へ進めるよう、現状分析、設計案、段階実装計画、リスク、検証手順を Markdown でまとめる。

## 前提

- 対象リポジトリは `D:\GitHub\tegaki`。
- 実装対象は原則 `tegaki_work/`。
- `PastFiles/` は参照専用。編集・削除しない。
- 現在の描画方式は PixiJS のラスター焼き込み方式。D&D調査では描画パイプラインへ手を入れない。
- レイヤー構造、保存形式、履歴形式を壊さないことを最優先にする。
- 実装コードの変更は不要。調査結果の Markdown 作成のみ。

## 調査してほしい内容

### 1. 現状整理

- レイヤーパネルの並べ替え、フォルダ入れ込み、フォルダ外出しが現在どのファイル・どの関数で扱われているか。
- アルバムカードの並べ替えが現在どのファイル・どの関数で扱われているか。
- SortableJS など外部ライブラリを使っている場合、その責務と限界。
- 現在すでに成立している操作と、操作感・安定性に課題がありそうな操作。

### 2. 独自D&D設計案

Pointer Events ベースの独自D&Dを採用する場合の設計案を作る。

考慮してほしい方針:

- ドラッグ中に実DOMやレイヤーデータを即変更しない。
- ドラッグ中は半透明プレビュー、挿入位置マーカー、フォルダ吸い込み候補だけを表示する。
- ドロップ時に一回だけデータ更新する。
- マウス、ペン、タッチで破綻しにくい。
- スクロール、右サイドバー、QAP、設定ポップアップと競合しにくい。

整理してほしい操作:

- 同階層内の並べ替え。
- フォルダへの入れ込み。
- フォルダ外への出し。
- 閉じたフォルダ上での扱い。
- フォルダそのものの移動。
- 複数選択レイヤーの扱い。初回実装で含めるべきか、後回しにすべきか。

### 3. アルバムD&Dへの横展開

- レイヤーパネル独自D&DとアルバムカードD&Dを共通化できるか。
- 共通化するなら、共通にできる部分と個別にすべき部分。
- 先にレイヤーパネル専用で作り、後からアルバムへ横展開する方が安全か。

### 4. 段階実装案

後続AIが作業しやすいよう、Phase分割案を出す。

例:

1. Phase A: レイヤー同階層並べ替えだけを独自D&Dへ置換。
2. Phase B: フォルダ入れ込み・外出しを追加。
3. Phase C: 閉じたフォルダ、スクロール、ペン/タッチ操作の調整。
4. Phase D: アルバムD&Dへ横展開。

各Phaseについて、触るファイル、触らないファイル、確認項目、失敗時の戻し方を書く。

### 5. リスク洗い出し

以下を具体的に書く。

- 壊れやすい既存契約。
- 保存/復元に影響する可能性。
- 履歴へ積むべきか、UI操作として履歴対象外にするべきか。
- フォルダ親子関係が破綻するケース。
- 背景レイヤー、フォルダ、通常レイヤー、クリッピングレイヤーの扱い。
- D&D中にレイヤー削除、表示切替、名前変更が起きた場合の扱い。

### 6. 検証手順

後続実装時に使えるチェックリストを作る。

最低限含める:

- マウス操作。
- ペン操作。
- タッチ操作。
- レイヤー数が多い状態。
- フォルダが開いている状態。
- フォルダが閉じている状態。
- 背景レイヤー付近へのドロップ。
- クリッピング中レイヤーの移動。
- Undo/Redo や保存/読込後に順序が破綻していないか。
