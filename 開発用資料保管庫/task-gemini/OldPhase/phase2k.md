# Phase 2k — キャンバス上限 / カメラ拡縮範囲の安全確認

> Phase 2j でストローク品質の緊急問題は一区切り。
> Phase 2k は、キャンバス片辺上限 2500px とカメラ拡縮範囲を、現行ラスター構造を壊さずに小さく扱うフェーズ。

---

## 背景

オーナーから以下の追加要望がある。

- キャンバスサイズ変更の片辺上限を広げたい。
- 当初 3000px 案があったが、漫画原稿サイズ 1700x2400 程度を一つの目安とし、2500px でもよい。
- キャンバス操作での拡縮倍率をもう少し広げたい。

Phase 2j の結果、縮小時の丸の歪みは入力点密度や手ブレ補正/後補正の別問題として扱う。
線補正は設定 popup のタブ化やブラシ設定整理と一緒に後続で扱う。

---

## 目的

- リサイズUIの上限値と、その参照元を確認する。
- 片辺 2500px にした場合の RenderTexture / 履歴 / サムネイル / 保存読み込みへの影響を整理する。
- カメラの `minScale / maxScale` を少し広げる場合の候補値と影響範囲を整理する。
- 初回は安全な最小差分で済むか判断できる状態にする。

---

## Gemini に任せる範囲

Gemini は初回、棚卸しと記録に限定する。

- 関係ファイルの読み取り。
- `rg` による上限値、ズーム値、リサイズ経路の検索。
- `PROGRESS.md` への調査結果追記。
- 低リスクなコメント/記録整理。

Gemini は独自判断で `LayerSystem`、`HistoryManager`、保存形式、描画パイプラインを変更しない。

## 調査観点

1. **キャンバス上限**
   - 現在の片辺上限がどこで定義されているか。
   - 2500px 化が `resize-popup.js` だけで済むか。
   - プリセットの扱いを変える必要があるか。

2. **メモリと履歴**
   - 2500x2500 の RenderTexture がレイヤーごとに増えることを前提に、履歴スナップショットで負荷が増える箇所を整理する。
   - 上限変更だけで履歴破損や保存破損が起きる可能性があるか。

3. **カメラ拡縮**
   - `TEGAKI_CONFIG.camera.minScale / maxScale` の現在値。
   - ホイール、ドラッグ、リセット、ショートカットで同じ値を参照しているか。
   - 候補値を広げるなら、まずどの程度が安全か。

4. **やらないこと**
   - 内部2倍化。
   - `renderer.resolution` 変更。
   - WebGL2 / WebGPU / SDF / MSDF 復活。
   - 線補正や後補正の実装。
   - 設定 popup タブ化。

---

## Phase 2k 初回の完了条件

- 2500px 上限化の最小変更点が `PROGRESS.md` に整理されている。
- カメラ拡縮範囲の候補値とリスクが整理されている。
- 実装候補が Gemini でできる低リスク修正 / Codex判断が必要な修正に分かれている。
- コード変更した場合のみ `npm.cmd run build` が成功している。

---

## Codex 判断へ戻すもの

- `LayerSystem.resizeLayerTextures()` のロジック変更。
- 履歴スナップショットや保存形式の変更。
- サムネイル生成の構造変更。
- 2500px を超える上限、または 3000px 採用。
- カメラ拡縮による座標ズレ修正。
