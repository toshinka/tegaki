# Phase 3i — 色操作の基礎（スポイトMVPとカラー導線整理）

## 概要

Phase 3h でバケツ系は実用状態になったため、次は色操作の基礎を小さく整える。
QAP のカラースロットはすでにあるので、まずはキャンバス上の色を拾って現在色へ反映するスポイト MVP を優先候補にする。

カラーサークルは UI 面積と操作導線が重いため、いきなり大きく実装しない。
まず QAP の余白を作り、スポイトを接続し、その後に「カラースロットと同時に使える折りたたみ式カラーサークル」を第一候補として検討する。

## 優先順

### 1. スポイトMVPの現状調査

- [ ] 既存の色取得 API、ピクセル抽出、座標変換経路を `rg` で確認する
- [ ] `brushSettings` の現在色変更イベントと QAP カラースロット更新経路を確認する
- [ ] 背景レイヤー、透明ピクセル、表示中レイヤー合成から拾うか、アクティブレイヤーから拾うかを整理する

方針:

- 最初は「表示中の描画結果から色を拾う」を第一候補にする。
- 完全透明を拾った場合は、現在色を変えないか、背景色を拾うかを実装前に判断する。
- 保存形式、レイヤー構造、履歴には触れない。

### 2. スポイトツールMVP

- [ ] ツールモード `eyedropper` を追加するか、一時ショートカットとして扱うかを決める
- [ ] キャンバスクリック位置の色を取得し、現在色へ反映する
- [ ] QAP の現在カラースロット表示と同期する
- [ ] ステータス表示に「スポイト」と出るようにする

候補操作:

- 基本は `I` キーなどのショートカット主体。
- QAP の COLOR スロット切り替えの横に小さなスポイトSVGを置く。
- 色取得後に元のツールへ戻るか、スポイトに留まるかは実装前に確認する。

### 3. QAP スリム化の下準備

- [ ] COLOR スロットの小型化を検討する
- [ ] COLOR スロットの横に、スロット切り替えとスポイトアイコンを横置きできるか確認する
- [ ] SIZE と OPACITY の余白を詰める
- [ ] SIZE / OPACITY スライダー自体を小型化する必要があるか確認する
- [ ] 「スロット選択後、サイズ/不透明度変更で自動保存」の説明文を撤去する

方針:

- まずは COLOR スロット小型化と SIZE / OPACITY の余白圧縮から始める。
- ツールボタンやペンスロットの小型化は、必要になった場合の次段階にする。
- `COLOR` / `TOOL` などのセクション見出し削除は、さらに面積が足りない場合の後続候補にする。

### 4. カラーサークル導線の整理

- [ ] QAP 内へ入れる場合の面積を見積もる
- [ ] カラースロットとタブ切り替えにする案を検討する
- [ ] 折りたたみ式でカラースロットと同時利用できる案を検討する
- [ ] QUICK 見出し付近にタブまたはタブアイコンを置けるか確認する
- [ ] 既存カラースロットとの役割分担を整理する

注意:

- QAP はすでに密度が高い。いきなり大きなカラーサークルを入れない。
- 先にスポイトで「色を取る」導線を固める。
- カラーサークルはカラースロットと同時に見たい場面が多いため、タブ切り替えより折りたたみ式を第一候補にする。
- カラーサークル本体は Phase 3i 内で無理に完了させなくてもよい。

## 確認事項

- [ ] `npm.cmd run build` が成功するか
- [ ] クリック位置と取得色が視覚的に一致するか
- [ ] QAP のカラースロット、サイズ/不透明度スロット、バケツ系設定が退行しないか
- [ ] QAP の縦幅が増えすぎず、低めの画面でも主要操作が見切れないか
- [ ] ペン、消しゴム、バケツ、投げ縄塗りの基本操作が退行しないか

## 禁止事項

- WebGPU / SDF / MSDF / JFA 経路へ触れない。
- QAP 全体の大規模再設計を同時に行わない。
- 保存形式、履歴形式、レイヤー構造を変更しない。
- 色管理の高度化（ICC、カラープロファイル等）へ広げない。
