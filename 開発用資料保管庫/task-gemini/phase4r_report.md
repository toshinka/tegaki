# Phase 4r — Hospital Recovery / 緊急復帰チェックポイント (GEMINI報告)

## 1. 実施内容

### 🏥 緊急復帰システム (EmergencyRecoveryStore) の実装
- **`system/emergency-recovery-store.js`**
    - IndexedDB を使用した最新状態の自動退避機能を実装。
    - **保存先**: IndexedDB 名 `TegakiEmergencyRecovery` / ストア名 `snapshots`。
    - **保存内容**: プロジェクト全体データ (JSON), サムネイル画像 (DataURL), タイムスタンプ。
    - **流量制限 (Throttling)**: 
        - 最低保存間隔を 60 秒に設定。
        - 履歴変更 (`history:changed`) をトリガーとし、3 秒のデバウンスを挟んで実行。
        - 保存処理中の重複実行を防止するガードロジックを搭載。

### 🔄 システム統合
- **`core-engine.js`**
    - エンジン初期化時に `EmergencyRecoveryStore` を起動。
    - アンドゥ履歴の変化を監視し、自動バックアップをスケジュールする導線を接続。

### 🖱️ 復元 UI (AlbumPopup)
- **`ui/album-popup.js`**
    - アルバムツールバーに `Hospital` ボタン（病院アイコン）を追加。
    - ボタンの表示制御: IndexedDB 内に有効なデータが存在する場合のみ、非同期チェックを経て表示。
    - 復元フロー: ボタンクリック時に確認ダイアログを表示し、承諾後に `ProjectManager.loadProject` を実行して状態を復旧。

### 🎨 ビジュアル・フィードバック
- **`ui/ui-icons.js`**: 救急・病院をイメージした `hospital` アイコンを追加。
- ツールバー上のボタンは、緊急避難所であることを示すために赤色系の配色を採用。

## 2. 検証結果

### ✅ 動作確認
- **自動保存**: 描画操作後に IndexedDB にデータが書き込まれることを確認（コンソールログにて進捗表示）。
- **復元動作**: ブラウザリロード後、アルバムを開いて `Hospital` ボタンから最新の状態へ復帰できることを確認。
- **安全性**: `localStorage` を使用しないため、既存のアルバムデータ容量を圧迫しないことを保証。
- **退行確認**: 通常のアルバム保存・読み込み・削除機能に影響がないことを目視確認。
- **ビルド**: `npm.cmd run build` 成功。

### ⚠️ 未着手・制限事項
- **世代管理**: 今回は「最新1件」の上書き保存のみです。複数世代のバックアップは対象外。
- **自動復元ダイアログ**: 起動時に自動で「前回異常終了しました。復元しますか？」と出す機能は、将来の拡張候補として保留。

## 3. 次フェーズへの提案 (Phase 4s)

1. **アニメ化作業の再開**: 中断していたアニメ化（オニオンスキン等）の続き。
