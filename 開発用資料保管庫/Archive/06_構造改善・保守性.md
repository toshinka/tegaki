# 構造改善・保守性ロードマップ

更新日: 2026-06-20

状態: Phase 5eで監査・局所修正を完了。

## 目的

新機能を止めて全面リファクタリングするのではなく、実装時に確認された技術負債を小さい監査と局所修正へ分割する。
対象はEventBus契約、global依存、UI/CSS、popup、History、通常LayerとCAFの描画境界。

## 現行実装との照合結果

### EventBus

- event名literalと `TegakiEventBus.EVENTS` が混在している。
- 定数化だけではpayload不一致や送受信漏れを防げない。
- 旧 `animation-system.js` のeventにも現行listenerが残るため、名称だけでdead eventとは判定できない。
- 全件一括置換より、変更対象eventごとにemit / on / payload / ownerを確認する方が安全。

### popup・UI style

- Phase 5bでoverlay mount helperが追加され、stacking context問題は解消済み。
- `style.left/top` 等には動的配置として必要なものがある一方、色・背景・固定寸法等の静的styleも残る。
- BasePopup classを先行導入するより、mount helperと共通CSS classの契約を先に安定させる。

### Layer・CAF描画境界

- CAFは独立した描画engineを持たず、animation working Layerを介して既存DrawingEngine / LayerSystemへ接続している。
- TimelineModel / ClipAsset / DrawingSnapshotが保存正本で、working Layerはアダプター。
- `IDrawingTarget` の先行導入は、この境界を再構成して回帰範囲を広げる。
- Phase 5c / 5dで変形・selectionの重複が具体化した時点で、必要最小APIを抽出する。

### History

- 現行Historyは既に `{ name, do, undo, byteSize?, meta? }` のcommand objectを受け取る。
- Phase 5bで件数・推定メモリ上限とCAF Raster差分履歴が導入済み。
- 抽象classと具象classの追加は必須ではない。先にcommand契約、meta、復元対象、byteSize計上を揃える。

### file header

- 長いheaderを全fileへ追加すると、被依存一覧などが実装より早く陳腐化する。
- import / export / `rg` の検索結果を正本とし、headerは責務・公開API・主要eventが必要なfileへ限定する。

## 優先度

### A. 不具合と同時に直す

- パレット外の文字色、背景色、border色。
- popupのmount先・stacking context不整合。
- 同一eventのpayload不一致。
- History commandのbyteSize未計上による容量制御漏れ。
- static styleの重複。

### B. Phase 5eで監査する

1. EventBusの送受信・payload・owner一覧。
2. `window.*` 登録とimport依存の役割重複。
3. static inline styleとCSS classの重複。
4. popup mount helperの適用漏れ。
5. History commandの命名、meta、byteSize、復元対象。
6. Phase 5c / 5d後のtransform・selection・CAF描画target重複。

### C. 証拠が出るまで行わない

- EventBus literalの全件機械置換。
- `window.*` の一括削除。
- 全popupのBasePopup継承化。
- `IDrawingTarget` 基底classの先行導入。
- `HistoryCommand` class階層への全面移行。
- CSS classの全件rename。

## 段階的な実施方法

### Step 1 — 棚卸し

- 対象を1 subsystemへ限定する。
- event、global、inline style、History commandを `rg` で抽出する。
- 「未使用の疑い」と「未使用を確認済み」を区別する。

### Step 2 — 契約の記録

- eventは名前、payload、emit元、listener、削除条件を記録する。
- popupはmount先、position責務、z-index、close / drag契約を記録する。
- Historyは対象ID、before / after、byteSize、Undo時の編集文脈復帰を記録する。

### Step 3 — 小分け修正

- 1回の変更で1契約または1componentを扱う。
- 既存挙動を維持し、互換classやadapterを必要な期間だけ残す。
- 使用箇所が0であることを確認してから旧event・global・classを削除する。

### Step 4 — 回帰確認

- 通常描画とCAF描画。
- Layer / Folder / CAF内部Layerの操作。
- Undo / Redo。
- popupの重なり、drag、close。
- project / album復元とexport。

## 完了条件

- 改修対象の責務と契約が実コードに一致する。
- 未使用と証明できない旧経路を推測で削除していない。
- 新しい抽象化が、実在する重複を減らしている。
- CSS変更がパレットとスタイルガイドへ適合する。
- buildと対象操作の回帰確認が完了する。
