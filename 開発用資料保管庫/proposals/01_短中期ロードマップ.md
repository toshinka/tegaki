### Animation / transform

- Phase 5wから5z6でposition / scale / rotation / anchor、opacity、Clip blend、cubic-bezier sampling、preset、Segment Easing Editorを既存 `transformKeyframes` 正本へ段階実装した。
- Segment Easing Editorは1区間の進捗曲線、Motion GraphはClip全体のparameter実値、Motion PathはCanvas上のXY経路として分ける。Canvas、数値 / marker、graphの各編集面から同じkey正本を操作し、UI固有のsample列を保存しない。
- Motion Graphはread-only viewer、preset拡張、既存key編集、curveを保つ途中点追加の順で検証する。詳細は `10_Motion_Graph・Easing・Motion_Path設計.md` を正本とする。
- Motion確定後の候補として、元Clipを保持したまま整数FrameのCAF列へ展開するbake操作を検討する。ペン加筆は展開後の各コマへ行う明示workflowを優先する。
- 滑らかな時計針等はClipだけのFPS値では解決せず、作画露出は整数Frameのまま、より高いproject出力sample rateでMotionだけsubframe samplingする構造を先に検証する。
- Ctrl複数選択CAFのDURATION一括増減は、同じdeltaを原子的に適用するTegaki側候補とする。Groupの位置・間隔を含む時間比例retimingは引き続き動画WEBUI側に置く。総Frames不足時はFRAMES増加を案内する。
- 簡易warp / morphは少数control pointと自動weightを優先し、密な手打ちmeshから始めない。
- Bone、Perform記録、physicsはwarpとkeyframe安定後まで保留する。
- 詳細は `09_変形アニメーション・メッシュ・GPU画材ロードマップ.md` を正本とする。

### 変形・欄外Raster・Resizeの後続

- 通常モードのFolder cardをLAYER TRANSFORM対象にし、Folder配下の全表示Raster / 子Folderを一つの操作単位として変形する要望を再監査する。既存資料の「通常/CAF Folder V変形済み」と実機報告が一致しないため、UI入口、対象収集、preview、確定、1 History、nested Folderの二重適用を最初に確認する。
- Ultra監査で、未確定V変形の初回CAF capture、compositorのtransform前crop、CAF mergeのunion bounds欠落を分離した。Phase 5z7でcapture / composite順だけを修正する。
- Canvas-only Resizeが全RenderTextureを新Project寸法へ破壊的に焼き直す経路、CAF DrawingSnapshot正本とworking adapterの分裂、Cameraの無条件再中心化を確認した。Phase 5z8で非破壊bounds / anchor transaction、adapter再同期、Camera view分離として扱う。
- content / both Resizeはresize前sourceを一度だけ取得するAffine transactionへ変更し、HistoryはSnapshot ID単位でdedupeして事前memory見積もりを行う。詳細と再現matrixは `11_Animation_Table欄外Raster・Canvas_Resize整合性監査.md` を正本とする。

### 長期候補: 別動画WEBUIとの連携

- Tegaki完成後に、After Effects的な動画編集を別ウィンドウ・別WEBUIとして構築する案を保留する。Animation Tableを分離するより、描画と動画編集のUI密度・操作契約を互いに独立させやすい。
- TegakiからはCAFまたはCAF Groupを、動画側からは使用clipをCAFへ分解可能な交換形式で受け渡す。初期段階は明示的な送信・取込とし、同時編集や自動同期を前提にしない。
- Group時間比率伸縮は動画側のkeyframe/retiming機能として扱い、Tegaki側Groupへは持ち込まない。動画側から戻す時に、整数FrameへbakeしたCAF列として変換する案を優先する。
- 難度は、片方向のCAF素材送信が中、Group構造を保った往復が高、動画clipから編集可能なCAF・Layerへ可逆分解する往復が非常に高い。主な難所はAsset/Snapshot参照、FPS・時間丸め、effect/transformのbake、色・alpha、変更競合、version互換である。
- 最初の実証は `manifest + raster frame sequence + timing + stable asset id` 程度に絞り、同一ブラウザ内では `BroadcastChannel` または明示ファイル受け渡しをtransport候補とする。保存形式のversioningをUI連携より先に決める。
