# 短中期ロードマップ

更新日: 2026-07-13

## 優先順位

### 現行: Phase 5w - Clip transform keyframe基盤

- 既存 `ClipInstance.transform` / `transformKeyframes` を正本とし、新しい運動モデルを重複実装しない。
- Slice 0でkeyframe schema、Frame基準、hold / linear補間を固定し、compositorをsample結果へ接続する。
- position / scale / rotationを先行する。opacity、色変化、easingはsamplingとround-trip安定後に分ける。
- mesh、bone、physics、WebGPUはPhase 5wへ混ぜない。

### 完了: Phase 5r - Animation Table表示操作

1. Lane個別の表示・非表示
   - Lane行にeye iconを追加する。
   - Playback Scopeのinclude checkboxとは別stateにする。
   - `LaneModel.visible` を保存・復元し、preview、再生、animation exportで同じ判定を使う。
   - CAF内部Layerのvisibility正本は変更しない。
2. Table閉状態のTimeline onion
   - Layer Panelの既存Lane onionの隣へTimeline onion buttonを置く。
   - Lane onionとTimeline onionは独立してON/OFFできる。
   - Timeline onionは前後同数、0から4の循環。前Frameは暖色、後Frameは青系の標準ghost色を維持する。
   - button配色はLaneをふたば系、Timelineを橙系として識別し、過度に鮮やかな同時点灯を避ける。

### 完了: Phase 5s - Animation Table基本操作

- ホイール: 縦Lane scroll、Shift+wheel: 横Timeline scroll、Ctrl+wheel: Timeline zoom。
- Lane追加・削除・並べ替えの保存・Undo境界を監査する。
- FPSごとの秒境界表示を追加する。
- Layer Panel上部のFrame表示を、未使用時は隠し、CAF利用時は小さく整理する。

### 完了: Phase 5t - CAF複数選択と一括移動

- Slice 1: Ctrl+clickでCAFを加算・解除選択する。選択集合はUI stateとし、単体の `selectedCelId` を編集対象として維持する。
- Slice 2: 選択集合のdragを1つのTimeline構造Historyとして移動する。
- 選択集合はUI stateとし、ClipAsset正本や内部Layer選択と混同しない。
- drag開始時に選択CAFの相対Lane/frame差を固定し、dropを1つのTimeline構造Historyにする。
- 重なり、範囲外、別Lane、duration付きCAF、linked CAFの衝突規則を先に決める。
- Windows風drag marqueeは未実装。空白領域からのdrag、pan、CAF移動の競合を避ける別Phase候補とする。

### 完了: Phase 5u - CAF複数payloadとCAF Group基盤

- 既存単体CAF clipboardのClipAsset共有/複製、History、save/load契約を最初に監査する。
- 直接の複数copy/pasteとCAF Groupのcopy/pasteを、同じ相対Lane/frame payloadと原子的一括配置へ載せる。
- CAF GroupはClipAsset内部FolderやLane Folderとは別のTimeline構造とする。
- folder iconは2件以上選択時に表示し、Lane×Frame占有範囲が1つの連結成分にならない場合は薄くdisabled表示する。
- Group作成/解除、移動、copy/paste、member削除、Undo/Redo、save/loadを段階実装する。
- Tegaki側のCAF Groupは移動・複製の操作単位に限定し、時間比率伸縮は原則実装しない。発想は将来の動画WEBUI側retiming候補へ送る。

### 完了: Phase 5v - Folder clipping

- Phase 5iのCAF `clippingMode` 3状態契約回帰を最初に修復する。
- CAF Folder clippingをpreview、onion、Frame compositor、History、save/loadへ接続する。
- 通常Folder targetは同一parent直下Raster sourceへ限定した実装から安全性を確認する。
- 通常/CAF Folderのvisibilityを配下全体へ揃え、Layer直下のFolderを配下alphaによるclip sourceとして扱う。
- Folder add/multiply/overlayは配下をまとめたgroup結果へ適用する。正確なライブgroup合成にdirty group RenderTextureが必要な場合は、Visibility / clipping完了後の低優先度Sliceとする。
- Timeline zoom controlをheaderへ移し、大量Frame時の水平scrollbar競合を解消した。33%まで縮小可能とし、60%未満はFrame番号を省略して秒境界を維持する。

### 長期候補: 別動画WEBUIとの連携

- Tegaki完成後に、After Effects的な動画編集を別ウィンドウ・別WEBUIとして構築する案を保留する。Animation Tableを分離するより、描画と動画編集のUI密度・操作契約を互いに独立させやすい。
- TegakiからはCAFまたはCAF Groupを、動画側からは使用clipをCAFへ分解可能な交換形式で受け渡す。初期段階は明示的な送信・取込とし、同時編集や自動同期を前提にしない。
- Group時間比率伸縮は動画側のkeyframe/retiming機能として扱い、Tegaki側Groupへは持ち込まない。動画側から戻す時に、整数FrameへbakeしたCAF列として変換する案を優先する。
- 難度は、片方向のCAF素材送信が中、Group構造を保った往復が高、動画clipから編集可能なCAF・Layerへ可逆分解する往復が非常に高い。主な難所はAsset/Snapshot参照、FPS・時間丸め、effect/transformのbake、色・alpha、変更競合、version互換である。
- 最初の実証は `manifest + raster frame sequence + timing + stable asset id` 程度に絞り、同一ブラウザ内では `BroadcastChannel` または明示ファイル受け渡しをtransport候補とする。保存形式のversioningをUI連携より先に決める。
