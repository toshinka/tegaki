# 無限キャンバス / 欄外ラスター保持 設計

更新日: 2026-06-28

## 目的

Phase 5pの目的は、400x400などのProject frame外へ移動したRaster内容を失わないこと。

Phase 5nでLayer全体の `V` 変形入口が通常Layer / CAF working Layerへ接続され、Phase 5oで外部画像import / clipboard画像貼り付けが可能になった。これにより、画像素材やアニメ用の前後フレーム素材を一時的に欄外へ逃がす操作が現実的になった。一方で現行実装はLayerごとのRenderTextureをProject frame寸法へ固定しており、変形確定や画像合成時にframe外へ出たpixelが切り捨てられる。

このPhaseでは、Project frame、保存/export範囲、カメラの基準寸法は維持しつつ、Layer内部のRaster保持範囲だけを可変化する。これは「真の無限Texture」ではなく、「有限の表示フレームに対して、Layer内容の保存矩形を必要に応じて広げる」設計である。

## 用語

### Project frame

- `TEGAKI_CONFIG.canvas.width / height` で表される作品の基準矩形。
- PNG / APNG / GIF / WebP / PSD出力、Timeline frame、背景Layer、リサイズUIの基準。
- Phase 5pではProject frameを無限化しない。

### Canvas / world coordinates

- 現行の描画座標、selection bounds、Layer変形中心が扱う座標。
- 基本的にProject frame左上を `(0, 0)` とする。
- 無限キャンバス導入後も、Project frame外の座標は負値または `width/height` 超過として扱える。

### Raster bounds

LayerのRenderTextureがProject coordinates上のどこに存在するかを表す矩形。

```js
{
  x: 0,
  y: 0,
  width: 400,
  height: 400
}
```

既存Layerはすべて上記と同等として扱う。移動や貼り付けで欄外が必要になった時だけ、`x/y` が負値または正のoffsetになり、`width/height` がProject frameより大きくなる。

### Visible frame crop

保存・出力・Frame compositorが最終的に描画するProject frame範囲。Raster boundsが大きくても、通常のPNG出力はProject frameでcropされる。欄外データはProject保存やLayer/CAF snapshotには残るが、通常出力画像へは写らない。

## 現行把握

### 固定RenderTexture前提

- `system/data-models.js`
  - `LayerModel.initializeTexture(width, height)` は指定寸法のRenderTextureとSpriteを作るだけで、原点情報を持たない。
- `system/layer-system.js`
  - 初期背景Layerと初期Raster Layerは `config.canvas.width / height` でtextureを作る。
  - `createLayer()` も新規LayerをProject frame寸法で作る。
  - `createLayerRasterSnapshot(layer)` はRenderTexture全体を抽出し、`{ layerId, width, height, pixels }` を返す。origin情報はない。
  - `restoreLayerRasterSnapshot(snapshot)` は `snapshot.width / height` と同寸法のRenderTextureへ復元する。Sprite位置は固定原点扱い。
  - `confirmLayerTransform()` の整数平行移動は `translateRgbaPixels()` で同じ幅高のbuffer内だけをshiftするため、外へ出たpixelは失われる。
  - 回転・拡縮・複合変形は `bakeTransform()` で同じRenderTextureへ焼き戻すため、同様にclipされる。
- `system/image-importer.js`
  - `beforeSnapshot.width / height` をtargetとして、外部画像を中央配置し、同じsnapshot寸法へ合成する。
- `system/pixel-selection-system.js`
  - selection bounds、copy/paste、floating transformの多くが `0..renderTexture.width/height` または `0..canvasWidth/Height` へclampされる。
- `system/animation/animation-data-model.js`
  - `DrawingSnapshotModel` は `width / height / pixels` のみでoriginを持たない。
- `ui/animation-table-popup.js`
  - CAF working Layerのcapture / restore、ClipAsset internal Layer preview、`_createRasterSnapshotCompat()` が幅高だけのsnapshotを前提にしている。
- `system/project-manager.js`
  - Project保存はLayerのRenderTextureをPNG dataURLへ変換するが、originを保存しない。
  - Project読込は新規LayerをProject frame寸法で作り、そのRenderTextureへ画像を流し込む。
- `system/animation/timeline-frame-compositor.js`
  - normal Layer snapshotやCAF snapshotを `(0,0)` に描画する。

### すでに維持するべき境界

- 本番描画はPixiJS RenderTextureへのライブラスター焼き込み。
- 背景Layerは特殊な不透明Layerで、通常Layerと同じ無限化対象にしない。
- 通常LayerとCAF internal LayerはUIを共有しても、保存正本とHistoryを混同しない。
- WebGPU、SDF/MSDF、WebGL2 Mesh、DPR 2倍化、内部2倍RenderTextureはPhase 5pへ混ぜない。

## 採用方針

### 方針A: Project frame固定 + Layer raster bounds可変

Phase 5pはこの方針を採用する。

- Project frameは固定矩形として残す。
- 各Raster Layerは `rasterBounds` を持つ。
- Layer Spriteは `rasterBounds.x / y` に配置し、RenderTextureは `rasterBounds.width / height` を持つ。
- Snapshot、History、Project保存、CAF DrawingSnapshotはboundsを保持する。
- 出力系はProject frameでcropする。
- 旧Project / 旧Snapshot / 旧Historyはboundsなしを `{ x: 0, y: 0, width, height }` として読む。

この方針なら、まず「欄外に逃がした素材が消えない」ことを解決できる。大規模なカメラUI、タイルエンジン、WebGPU化を待たなくてよい。

### 採用しない方針

#### 真の無限Texture

1枚の巨大RenderTextureを無限に広げる設計は採用しない。WebGLのmax texture size、VRAM、History byteSize、CAF多枚数運用に対して危険が大きい。

#### Tiled canvas

タイル分割は長期的には妥当だが、Phase 5pの初手にはしない。描画、selection、fill、clipping、export、CAF previewの全経路が一度に複雑化する。

#### Project frame自体の無限化

Project frameを可変・無限にすると、export、Timeline frame、背景、リサイズ、Album thumbnail、既存保存形式の意味が崩れる。Phase 5pの主目的は「欄外データ保持」であり、出力範囲の再設計ではない。

## データ契約

### RasterBounds

```js
const DEFAULT_RASTER_BOUNDS = {
  x: 0,
  y: 0,
  width: TEGAKI_CONFIG.canvas.width,
  height: TEGAKI_CONFIG.canvas.height
};
```

不変条件:

- `x/y/width/height` は整数pixel。
- `width/height >= 1`。
- `pixels.length === width * height * 4`。
- `width/height` はRenderTexture寸法と一致する。
- `rasterBounds.width/height` とsnapshot `width/height` は当面ミラーする。
- boundsなしの旧データは `{ x: 0, y: 0, width: snapshot.width, height: snapshot.height }` とみなす。

### LayerModel

通常Raster Layerは次を持つ。

```js
layerData.rasterBounds = { x, y, width, height };
```

背景Layer:

- Project frame固定。
- `rasterBounds` を持たせてもよいが、常にProject frameと同値。
- 変形、import、selection pasteによるbounds拡張対象外。

Folder:

- pixelを持たないため `rasterBounds` は持たない。
- 将来、配下Layerのunion boundsを表示用に計算することはあり得るが、保存正本にはしない。

### LayerRasterSnapshot

`LayerSystem.createLayerRasterSnapshot()` はboundsを返す。

```js
{
  layerId,
  width,
  height,
  rasterBounds: { x, y, width, height },
  pixels,
  pathsData,
  paths
}
```

旧呼び出し互換のため `width/height` は残す。新規実装では `normalizeRasterSnapshot(snapshot, projectFrame)` を経由し、boundsなしsnapshotを必ず補正してから扱う。

### DrawingSnapshotModel

CAFの保存正本もboundsを持つ。

```js
new DrawingSnapshotModel({
  width,
  height,
  rasterBounds: { x, y, width, height },
  pixels,
  isBlank
});
```

`serialize()` は `rasterBounds` を保存する。旧Projectのsnapshotは読込時にbounds補完する。

### ClipInstance.rasterSnapshot

`rasterSnapshot` は互換用だが、まだ多くのpreview / history / paste経路が参照するため、`DrawingSnapshotModel` と同じboundsを持たせる。

```js
{
  id,
  drawingSnapshotId,
  width,
  height,
  rasterBounds,
  pixels,
  isBlank,
  updatedAt
}
```

## Rendering契約

### Layer containerの役割

- Layer containerはProject coordinate spaceに残す。
- 通常確定状態ではLayer containerのposition / rotation / scale / pivotを初期値へ戻す。
- `layerData.layerSprite.position` を `rasterBounds.x/y` にする。
- `layerData.layerSprite.texture` は `rasterBounds.width/height` のRenderTextureを参照する。

### 変形previewの役割

- V変形中だけLayer containerまたはLayerTransformがpreview transformを持つ。
- confirm時にRaster boundsへ焼き込み、preview transformはリセットする。
- cancel時は開始時のtransform / bounds / pixelsへ戻す。

### 描画座標

現行のstroke pointはProject coordinatesとして扱われている。Raster bounds導入後、RenderTextureへ描く時は次へ変換する。

```js
rasterX = projectX - rasterBounds.x;
rasterY = projectY - rasterBounds.y;
```

BrushCore、FillTool、Eraser、Airbrush、Blur、selection制限など、RenderTextureをtargetにする経路はこの変換を共有helperへ寄せる。

推奨helper:

- `LayerSystem.getRasterBounds(layer)`
- `LayerSystem.normalizeRasterBounds(bounds, fallbackWidth, fallbackHeight)`
- `LayerSystem.projectToRasterPoint(layer, x, y)`
- `LayerSystem.rasterToProjectPoint(layer, x, y)`
- `LayerSystem.ensureRasterBoundsForProjectRect(layer, rect, options)`
- `LayerSystem.restoreLayerRasterSnapshot(snapshot, { preserveLayerId: true })`
- `LayerSystem.createBlankLayerRasterSnapshot(layerId, bounds)`

## 操作別設計

### Layer作成

- 新規Raster LayerはProject frameと同じboundsで開始する。
- 初期描画の挙動は変えない。
- 背景LayerはProject frame固定。

### 画像import / clipboard画像貼り付け

Phase 5oの入口を維持し、初期配置はProject frame中央を基本にする。

Phase 5pで変える点:

- 合成先は現在のsnapshot寸法ではなく、`union(currentRasterBounds, placementRect)`。
- placementRectはProject coordinatesで持つ。
- imageがProject frameより大きい場合の自動縮小はPhase 5o互換として維持する。原寸importや配置UIは別slice候補。
- Undo / Redoはboundsごと復元する。
- CAF working Layerは通常Historyへ記録せず、現行どおりCAF raster Historyへ流す。

### VキーLayer全体変形

最優先対象。

#### 整数平行移動

現在は `translateRgbaPixels()` で同じbuffer内shiftを行う。Phase 5pでは、純粋な整数平行移動はpixelを動かさず、`rasterBounds.x/y += dx/dy` を基本にする。

利点:

- 欄外へ出てもpixelを失わない。
- 再サンプリングが発生しない。
- 大きいLayerでもpixel copyを避けられる。

注意:

- Historyには移動前後のboundsが必要。
- `layerSprite.position` 更新とthumbnail更新が必要。
- 既存のPhase 5h非再サンプリング品質を維持する。

#### 非整数移動 / 拡縮 / 回転 / flip

一般変形は次の順で扱う。

1. 変形後のRaster boundsを計算する。
2. 必要boundsを上限内へ丸める。
3. 新しいRenderTextureへ、旧Spriteを変形行列 + offset補正付きで1回だけ描画する。
4. Layer containerのpreview transformをリセットする。
5. `rasterBounds` とsnapshotを更新する。

このsliceは整数平行移動よりリスクが高い。最初の実装では整数移動だけを無損失化し、拡縮・回転は現行1回bakeを維持してもよい。ただし設計上は一般変形もbounds拡張へ接続できるようにする。

### Selection / floating selection

Selection stateはProject coordinatesのまま維持する。

必要変更:

- `_extractRegion(snapshot, bounds)` はProject boundsをRaster boundsへ変換してから切り出す。
- `_clearRegion()` も同様にProject -> Raster変換する。
- selection pasteはProject frame内へclampしない。paste先rectをProject coordinatesで持ち、Layer boundsを必要に応じて広げる。
- Folder selection clipboardは各LayerのRaster boundsとProject boundsを区別する。

初期sliceではLayer全体Vと画像importを先に通し、selectionは監査と互換維持から始める。

### Fill / eyedropper / reference all layers

Fillはboundary snapshot上の探索座標が重要になる。

- active Layerだけ参照する場合: click座標をRaster boundsへ変換して探索する。
- all layers参照の場合: Project frame cropのcomposite snapshotを使う現行設計を維持し、欄外までは探索対象にしない。
- 欄外までfill探索する水準はPhase 5pの主目的外。

EyedropperはProject frame内サンプリングを維持する。欄外表示・欄外サンプリングは後続候補。

### Clipping / inverse clipping

Clippingはbounds差の影響が大きい。

現行はclipped Layerのcontainer内へsource LayerのRenderTextureをmask Spriteとして置く。bounds導入後は、mask Spriteにもsource rasterBounds位置を反映する必要がある。

最低条件:

- source Layerとtarget Layerのboundsが違っても、Project coordinates上のalpha位置でmaskされる。
- inverse clippingも同じProject coordinatesで判定する。
- source Layerが欄外にある場合、欄外alphaもtargetの欄外内容をmaskできる。ただしProject frame出力時はcropされる。

### CAF working Layer / DrawingSnapshot

CAF internal Layerも通常Raster Layerと同じbounds契約を持つ。

必要変更:

- `_syncClipAssetToWorkingLayers()` はDrawingSnapshotのboundsをworking Layerへ復元する。
- `_captureDrawingLayerToSelectedClip()` はworking LayerのboundsをDrawingSnapshotへ保存する。
- `drawing:stroke-completed` 経由のCAF raster Historyはboundsごとbefore / afterを持つ。
- CAF copy / paste、ClipAsset duplicate、internal Layer duplicateはboundsもcloneする。
- Animation Table previewとFrame compositorはsnapshotをProject frame上の `rasterBounds.x/y` に描画する。

### Save / Load

Project JSONはversionを上げる候補。

通常Layer:

```js
{
  id,
  name,
  visible,
  opacity,
  blendMode,
  clippingMode,
  parentId,
  rasterBounds: { x, y, width, height },
  image
}
```

- `image` はRenderTexture全体、つまり `rasterBounds.width/height` のPNG dataURL。
- boundsなしの旧Projectは `{ x: 0, y: 0, width: canvas.width, height: canvas.height }` として読む。
- 読込時はLayer作成後にbounds寸法のRenderTextureへ差し替え、Sprite位置をboundsへ合わせてからimageを復元する。

CAF:

- `DrawingSnapshotModel.serialize()` に `rasterBounds` を追加する。
- 旧snapshotはbounds補完する。
- `projectData.animation` のmigrationはTimelineModel生成時に行い、呼び出し側で黙って部分補正しない。

### Export / thumbnail

出力:

- PNG / WebP / APNG / GIF / WebM / Frame compositorはProject frameでcropする。
- 欄外pixelは保存正本には残るが、通常出力へは含めない。
- 「全boundsを書き出す」機能は別候補。

Layer / CAF thumbnail:

- Project frame crop thumbnailと、Layer content fit thumbnailのどちらを使うかを用途ごとに決める。
- Layer panel上の内部Layer previewはcontent fitの方が欄外データに気付きやすい。
- Frame preview / Album thumbnail / Export previewはProject frame cropを維持する。

## 上限とメモリ

無限キャンバスは、実装上は「必要矩形を広げる」だけであり、上限が必要。

初期上限案:

- `maxTextureSize` は実行時にrenderer / WebGL capabilityから取得する。
- `TEGAKI_CONFIG.canvas.maxSize` を超えるProject frameは既に禁止されているが、rasterBoundsはProject frameより大きくなり得る。
- 1 Layerあたりの初期soft capを `min(maxTextureSize, 4096)` 程度から始め、実機計測で調整する。
- pixel byteは `width * height * 4` としてHistory byteSizeとCAF memory profilerへ反映する。
- cap超過時は黙ってclipせず、操作を拒否するか、確認toastを出す。

重要:

- Phase 5pでは巨大タイル退避、IndexedDB差分保存、Worker圧縮を先行導入しない。
- Phase 5k / 5lで入れたメモリ・保存負荷対策を壊さない。

## Phase 5p Slice設計

### Slice 1 — 固定前提の監査とcontract追加

目的:

- まだ挙動を変えず、bounds契約とhelperを足す。
- 旧snapshotを正規化して扱えるようにする。

対象:

- `system/raster-bounds.js` 新規候補。
- `system/data-models.js`
- `system/layer-system.js`
- `system/animation/animation-data-model.js`
- `ui/animation-table-popup.js`

実施:

- `normalizeRasterBounds()`、`normalizeRasterSnapshot()`、`projectFrameBounds()`、`unionRasterBounds()`、`translateRasterBounds()` を追加する。
- LayerModelへ `rasterBounds` を追加する。
- `createLayerRasterSnapshot()` がboundsを返す。
- `restoreLayerRasterSnapshot()` がboundsなしsnapshotも読める。
- まだ既定boundsはProject frame固定で、ユーザー挙動を変えない。
- debug時だけ、boundsとRenderTexture寸法の不一致を警告する。

受け入れ:

- 既存のpen / eraser / airbrush / image import / V変形 / CAF working Layerが従来通り動く。
- 旧Project保存復元が変わらない。
- node --check、build成功。

### Slice 2 — 通常Layerの整数V移動をbounds commit化

目的:

- 欄外へ純粋移動したpixelを失わない最小価値を出す。

対象:

- `system/layer-system.js`
- `system/layer-transform.js`
- `system/raster-translation.js`
- `system/project-manager.js`

実施:

- `resolveIntegerTranslation()` がtrueの時、`translateRgbaPixels()` ではなくbounds origin更新を使う。
- `layerSprite.position` をboundsへ同期する。
- Historyはbefore / after snapshotにboundsを含める。
- Project保存復元で通常Layerのboundsを保存・復元する。
- ExportはProject frame cropを維持する。

固定確認:

- 400x400、通常Raster Layerへ短い線を描く。
- `V` -> ArrowRightを多数回、またはtransform入力で +450px 移動 -> confirm。
- 見えなくなった内容がUndoで戻り、Redoで再び欄外へ移動する。
- さらに -450px 移動して戻した時、元の線が欠けない。
- 保存復元後も戻せる。

### Slice 3 — 画像importとLayer複製/結合のbounds対応

目的:

- Phase 5oで入った外部画像入口を無限Layer契約へ接続する。

対象:

- `system/image-importer.js`
- `system/layer-system.js`
- `system/project-manager.js`

実施:

- import合成はcurrent boundsとplacement rectのunionへ描く。
- 複製はboundsとSprite位置をcloneする。
- merge downはProject coordinates上で合成し、bottom Layerのboundsを必要に応じて拡張する。
- clear/delete drawingsはboundsを維持するか、Project frameへ戻すかを仕様化する。初期はbounds維持が安全。

固定確認:

- 画像を貼る -> Vで欄外へ移動 -> 戻す。
- Layer duplicate後もboundsが維持される。
- merge downで欄外部分が欠けない。

### Slice 4 — CAF DrawingSnapshot / working Layer対応

目的:

- Animation Table / CAF working Layerでも同じ欄外保持を実現する。

対象:

- `system/animation/animation-data-model.js`
- `ui/animation-table-popup.js`
- `system/animation/timeline-frame-compositor.js`
- `system/animation/caf-memory-profiler.js`

実施:

- `DrawingSnapshotModel` にboundsを保存する。
- working Layer restore / captureがboundsを往復する。
- CAF raster Historyがbounds込みsnapshotを保持する。
- ClipAsset duplicate / internal Layer duplicate / CAF copy pasteでboundsをcloneする。
- Frame compositorはsnapshot canvasを `rasterBounds.x/y` に描く。

固定確認:

- Animation TableでCAFを作る。
- CAF internal Layerへpen描画または画像貼り付け。
- `V` で欄外へ移動しconfirm。
- 次FrameへCAF copy / pasteしてから戻しても欠けない。
- Undo / Redo、保存復元、Frame出力でProject frame cropが維持される。

### Slice 5 — drawing / selection / fillのProject-to-Raster変換

目的:

- bounds originが0でないLayerへ追加描画しても位置がずれないようにする。

対象:

- `system/drawing/brush-core.js`
- `system/drawing/drawing-engine.js`
- `system/drawing/fill-tool.js`
- `system/pixel-selection-system.js`
- `system/drawing/eyedropper-tool.js`

実施:

- stroke描画targetへ渡す点をProject coordinatesからRaster coordinatesへ変換する。
- stroke範囲が現bounds外へ出る場合は、stroke前またはstroke中にboundsを拡張する。
- selection extract / clear / pasteはProject boundsをRaster boundsへ変換する。
- fillはactive Layer参照時だけRaster bounds対応し、all layers参照はProject frame cropを維持する。

固定確認:

- 欄外から戻したLayerへ追加strokeしても既存絵とずれない。
- bounds.x/yが負値のLayerでもselection copy / cut / pasteが成立する。
- Fillがbounds originに影響されない。

### Slice 6 — Clipping / preview / thumbnail回帰

目的:

- 通常・逆clipping、Layer panel、CAF previewをbounds差で壊さない。

対象:

- `system/layer-system.js`
- `ui/layer-panel-renderer.js`
- `ui/animation-table-popup.js`
- `system/animation/timeline-frame-compositor.js`

実施:

- clipping mask spriteへsource bounds位置を反映する。
- Layer thumbnailは用途に応じてProject crop / content fitを使い分ける。
- CAF internal Layer previewはbounds位置つきで描く。

固定確認:

- source Layerとclipped Layerが異なるboundsでもmask位置が合う。
- inverse clippingも位置が合う。
- Thumbnail / Animation preview / Export previewが破綻しない。

### Slice 7 — 仕上げと操作補助

目的:

- ユーザーが欄外データを見失った時に回収できる。

候補:

- 「アクティブLayerをProject frame中央へ戻す」既存resetとの関係整理。
- 「Layer内容へ表示を合わせる」camera helper。
- 欄外データがあるLayerへ小さなbadge表示。

このsliceは必須ではない。データ保持と保存復元が先。

## 固定テスト行列

### 通常Layer

- 400x400 / pen 10px / opacity 100%。

## 失敗しやすい点

### 1. Sprite位置だけ変えて描画座標を変え忘れる

`layerSprite.position = rasterBounds.x/y` だけでは表示は合うが、追加strokeやfillがずれる。RenderTextureへ描くすべての経路でProject -> Raster変換が必要。

### 2. boundsなし旧データを場当たり的に補完する

呼び出し元ごとに補完すると、History / CAF / Project loadで差が出る。正規化helperへ集約する。

### 3. Exportを全boundsへ広げてしまう

Phase 5pの目的は欄外保持であり、通常出力はProject frame cropを維持する。ここを変えるとアニメ出力とAlbum thumbnailの意味が変わる。

### 4. clipping maskの座標空間を混同する

mask spriteはtarget Layer container内に置かれるため、source bounds位置をProject coordinatesとして反映する必要がある。

### 5. CAF working Layerを通常Layer Historyへ記録する

Phase 5n / 5oで分離した境界を維持する。CAF上のbounds変更はCAF raster Historyへ入れる。

### 6. 無制限に巨大RenderTextureを作る

「無限」はUI上の概念であり、実装は有限bounds。必ずmax texture sizeとHistory byteSizeを確認する。
