# フォルダ合成・クリッピング調査

作成日: 2026-06-30

## 結論

Folder clippingはPhase 5vで段階実装する。Folder blendの完全なgroup合成は低優先度で棚上げする。

理由は、単純なUI解放では済まず、通常Layerのライブ表示、CAF working Layer、CAF preview / onion、Timeline export、thumbnail、History / save-loadの全経路で「フォルダを一度groupとして合成してから親へ置く」契約を共有する必要があるため。

現在の満足度が高く、フォルダopacity、通常/CAFフォルダV変形、CAF preview / onion / exportのフォルダblend基礎は既に入っている。次に進むなら、ここをPhase化せず proposal に残し、後で合成パイプラインを触るPhaseが来た時にまとめて扱うのが安全。

## 外部仕様から採る契約

Adobe Photoshop公式のLayer opacity / blendingでは、Layer groupの既定はPass Throughで、groupに別blend modeを指定すると、group内部を先に合成し、その合成結果を一枚の画像として親へblendする、と説明されている。

TegakiではPass Through modeを新設せず、次の契約へ寄せる:

- 子Layer / 子Folderのblend modeは変更しない。
- Folder自身のopacity / blend modeは、配下をgroupとしてまとめた後に親へ適用する。
- 子がmultiply、親Folderがaddなら、子のmultiplyをgroup内で処理し、そのgroup出力をaddで親へ合成する。
- Folder設定を変えても、子Layerの設定値を書き換えない。

Adobe Photoshop公式のclipping mask説明では、base layerの非透明部分が上側Layerの表示を決め、複数Layerを連続してclipping maskへ含められる。Tegakiでは既存の「同じ親内の直下アイテムをsourceにする」方式を維持し、Folderをsource / targetにできるなら次の契約にする:

- Folderがclip sourceの場合、そのFolder配下の可視Rasterをgroupとして合成したalphaをmaskにする。
- Folderがclip targetの場合、そのFolder配下をgroupとして合成し、そのgroup出力全体をsource alphaでclipする。
- clippingは同一parent内の直下きょうだいだけを参照する。親Folder境界は越えない。
- sourceが非表示、存在しない、または有効alphaを持たない場合、targetは非表示相当になる。
- 逆クリッピングは、source group alphaを `destination-out` 相当で扱う。

参照:

- Adobe Photoshop User Guide: Layer opacity and blending  
  https://helpx.adobe.com/photoshop/using/layer-opacity-blending.html
- Adobe Photoshop User Guide: Reveal layers with clipping masks  
  https://helpx.adobe.com/photoshop/using/revealing-layers-clipping-masks.html

## 現行コード確認

### 通常Layer / Folder

`tegaki_work/system/layer-system.js`

- `setLayerBlendMode(layerIndex, blendMode)` はFolderを拒否しない。`layer.layerData.blendMode` へ値は保存できる。
- `setLayerClippingMode(layerIndex, clippingMode)` は `isBackground || isFolder` を拒否している。
- `refreshClippingMasks()` は `isBackground || isFolder || !isClippingEnabled(data)` をskipしている。
- `_findClippingSourceLayer()` は同じparentの直下候補がFolderなら `null` を返す。
- 現在の通常clippingは「source RasterTextureをmask Spriteとしてtarget layerSpriteへ貼る」構造で、Folder group outputをmask Sprite化する経路がない。

`tegaki_work/ui/layer-panel-renderer.js`

- `_createClipStatusIcon()` はFolderではtoggle不可。
- 属性popupはFolderの場合 `toggle-clipping` ボタンを出さない。
- blend selectはFolderにも出せる状態。

`tegaki_work/system/data-models.js` / `tegaki_work/system/project-manager.js`

- `LayerModel` は `clipping` / `clippingMode` / `blendMode` を保持・serializeできる。
- Project保存復元もFolderの `blendMode` / `clippingMode` を保持できる形に近い。
- ただしUIとLayerSystemがFolder clippingを拒否しているため、通常Folder clippingは実運用経路に入っていない。

### CAF internal Layer / Folder

`tegaki_work/system/animation/animation-data-model.js`

- `ClipAssetInternalLayerModel` はFolderでも `blendMode` / `clipping` をserializeする。
- `toggleClipAssetInternalLayerClipping()` 自体はFolderを拒否していない。

`tegaki_work/ui/animation-table-popup.js`

- CAF preview / onion側の `_renderInternalLayerPreviewGroup()` はFolderをPixi `Container` として再帰描画し、Folder own opacity / blendをContainerへ適用している。
- `_createInternalClippedSnapshot()` / `_createInternalMaskSprites()` はclip sourceがFolderなら配下Raster descendantsをmask sourceにできる。
- `setInternalLayerAttributesFromExternal()` は `attributes.clipping` について `layer.type !== 'folder'` の時だけ適用するため、UI経由でFolder clippingはまだ入らない。
- `_syncClipAssetToWorkingLayers()` はCAF内部Rasterを既存LayerSystemのworking Layerへflatten同期する。opacityは祖先Folder分をeffective alphaにできるが、Folder blendはworking Layerへ継承しない方針コメントがある。
- つまりCAFのpreview / onion / exportはgroup寄り、編集用working Layer表示はflat adapter寄りで、完全WYSIWYGへ寄せるには追加設計が必要。

### Export compositor

`tegaki_work/system/animation/timeline-frame-compositor.js`

- `_renderNormalLayerGroup()` / `_renderAssetLayerGroup()` はFolder配下をCanvas2Dのgroup canvasへ描いてから、Folder own opacity / blendを親ctxへ適用する。
- `_applyClippingMask()` はCAF asset側でsourceがFolderなら配下Raster descendantsをmask canvasへ描ける。
- 現状のexport compositorは、後続実装の参照コードとして最も近い。ただしライブ表示のPixi RenderTexture / mask / cacheとは別世界なので、このまま移植はできない。

## なぜ重いか

### 1. Folder blendとFolder clippingは同じ「group output」が必要

Folder opacityだけなら子Layerへeffective alphaを掛ける近似で済む場合がある。

Folder blend / Folder clippingは違う。正しい結果には次が必要:

1. Folder配下を、子Layerのopacity / blend / clipping / visibility / rasterBoundsを反映して一枚のgroupへ描く。
2. Folder自身のclippingがあれば、そのgroup全体へmaskを掛ける。
3. Folder自身のopacity / blendを使って、親groupまたはrootへ合成する。

子LayerへFolder blendを継承させるだけでは、Photoshop型のgroup合成と一致しない。

### 2. 通常Layerのライブ表示が現在の最大コスト

通常Layerのclippingは、source RasterTextureをmask Spriteにする設計。Folder source / Folder targetを入れるには、Folder配下をRenderTextureへレンダリングしたgroup textureが必要になる。

候補:

- Folderごとにdirty cache付きRenderTextureを作る。
- Folderが非default設定を持つ時だけgroup RenderTexture化する。
- Folder childrenを通常表示から隠し、group Spriteで置き換える。
- 選択、V変形、thumbnail、hit判定、描画中のactive Layer表示と同期する。

これは「UIのボタンを有効化する」ではなく、LayerSystemの表示構造に近い。

### 3. CAF working Layerは構造上flat adapter

CAFの正本は `TimelineModel / ClipAsset / DrawingSnapshot` で、編集時はanimation working Layerへ展開して既存描画engineへ接続している。

Folder blendを正確に見せるには、working Layer表示にもFolder group出力を重ねるか、選択中CAF previewを常に正本表示として使う必要がある。しかしworking Layerは描画入力先でもあるため、完全なgroup置換はペン入力、bucket、selection制限、V変形、History境界へ影響する。

このため、CAFでは次の割り切りが現実的:

- 編集中のworking Layerはflat adapterのまま。
- preview / onion / exportはgroup合成を正本にする。
- ユーザーが見ている表示差が問題になった時だけ、選択中CAFのgroup preview overlayを強化する。

### 4. History / save-load / UIだけ先に開けると破綻する

データモデルはFolder clippingを保持できる形に近いが、表示・出力が未対応のままUIを開けると、保存はされるが見た目が一致しない状態になる。

Folder clippingを実装するPhaseでは、UI解放は最後にする。

## 棚上げする場合の整理

棚上げするなら、現状の残存は次の2点だけとして扱う。

- Folder clippingは未実装。UIでは引き続きFolderのclip toggleを出さない。
- Folder blendは「CAF preview / onion / exportと通常export寄りのgroup合成は進んだが、通常Layerライブ表示とCAF working Layer表示は完全なPhotoshop互換の保証対象外」とする。

現行HEADではCAF `clippingMode` 3状態契約と通常Folder属性保存に回帰が見つかったため、Phase 5v Slice 0で先に修復する。

次に別改修へ進む前の確認項目:

- 通常Folder opacity / V変形は動作済み。
- CAF Folder opacity / V変形は動作済み。
- CAF Folder表示/非表示、開閉、マウス操作は動作済み。
- Folder blend selectorは残してよいが、Folder clipping UIは開けない。
- Project保存復元でFolder `blendMode` が落ちないことは後続の通常保存回帰で見る。

## テストマトリクス

### 通常Layer

1. Raster A、Raster BをFolder Fへ入れる。Aはmultiply、Bはnormal、Fはadd。下に色Layerを置き、Fのaddがgroup全体へ効くこと。
2. F opacity 50%。AとBの重なり部分が「各子50%」ではなく「group合成後50%」になること。
3. Raster sourceの上にFolder targetを置き、Fをclipping。F配下全体がsource alphaだけに出ること。
4. Folder sourceの上にRaster targetを置き、targetをclipping。Folder配下alphaでtargetがclipされること。
5. Folder sourceの上にFolder targetを置き、targetをclipping。両方のgroup alphaで成立すること。
6. nested Folderで、clipping source探索が同一parentを越えないこと。
7. inverse clippingが通常 / Folder source / Folder targetで成立すること。
8. Folder open / close、visibility、opacity popup、blend popup、V変形、selection制限、Undo / Redo、Project保存復元が壊れないこと。

### CAF

1. CAF internal Folderで通常Layerと同じblend / opacity / clipping構成を作る。
2. Animation Table preview、Onion、GIF / APNG / PNG exportで同じ合成結果になること。
3. working Layerでpen / eraser / bucket / selection / V変形後、Folder group previewへ戻しても結果が飛ばないこと。
4. CAF internal Folder clippingのUndo / Redo、Frame切替、CAF copy / paste、Project保存復元が成立すること。

### 性能

1. 400x400 / Folder 3階層 / Raster 10枚で通常描画のFPS低下が許容内。
2. CAF 24F / 各Frame Folder 2個 / Raster 5枚でpreview切替が詰まらない。
3. group RenderTexture cacheがLayer削除、Project load、CAF切替、Animation Table closeで破棄されること。

## 実装着手の判断基準

Folder clippingはPhase 5vで着手する。Folder blend完全合成は棚上げでよい。

着手する価値が出る条件:

- Folder clippingを使いたい具体的な制作例が出た。
- Folder blendのライブ表示とexportの差が制作上問題になった。
- CAF preview / onion / exportのFolder合成をさらに強く使うPhaseが来た。
- Group RenderTexture cacheを触る別Phaseが来た。

着手しない方がよい条件:

- UIのclip buttonだけ先に開けたい。
- 通常Layerだけ、またはCAFだけの限定修正で済ませたい。
- WebGPUや別レンダラ移行と同時にやりたい。
