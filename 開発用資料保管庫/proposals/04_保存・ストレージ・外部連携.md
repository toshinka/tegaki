# 保存・ストレージ・外部連携計画

更新日: 2026-06-19

## 現在の到達点

- 通常AlbumはIndexedDBへ保存。
- 緊急復帰checkpointもIndexedDBを使用。
- 通常Layer/Folder、CAF、TimelineModelの保存・復元足場。
- Album HTML保存・読込と旧JSON互換の基盤。

## 保存方式の役割

- IndexedDB: 日常の高速保存、Album、緊急復帰。
- 外部ファイル: 長期保管、別PCへの移動、Git外バックアップ。
- 外部フォルダ同期: 将来の大容量連番画像・素材管理。

## 実装候補

### ストレージ使用量表示 `候補`

- `navigator.storage.estimate()` を利用する。
- 使用量、quota、割合をAlbum上部に表示する。
- 80%等の固定警告値はブラウザ差を考慮し、設定可能性を残す。

### 容量整理 `候補`

- 削除対象を一覧表示し、ユーザーが選択して削除する。
- Album作品、緊急復帰、キャッシュを同じ「古いデータ」として一括削除しない。
- Undo履歴はメモリ責務であり、IndexedDB整理と分ける。

### 外部フォルダ同期 `候補`

- File System Access APIでユーザー指定フォルダへ保存する。
- handle権限失効、Chrome依存、再接続UIを設計に含める。
- 初回はプロジェクト単位の明示保存から始め、自動差分同期は後段にする。

### ブラウザと外部の相互移動 `候補`

- IndexedDB作品を外部へexport。
- 外部projectを検証・正規化してAlbumへimport。
- version、canvas、Layer、CAF、Timeline、thumbnailをmanifestへ持たせる。

## 外部ツール連携

- PSD入出力は `01_描画・編集・出力.md` を参照。
- 連番画像・動画素材はアニメ計画とAI支援計画の共通入出力として設計する。
- 保存形式変更時は既存Albumデータを黙って書き換えず、migrationを明示する。
