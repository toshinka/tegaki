## 1. WebGPU / SDF / MSDF 分岐を Phase 1c では封印する

対象候補：

- `tegaki_work/config.js`
- `tegaki_work/system/drawing/fill-tool.js`
- 必要なら `tegaki_work/system/drawing/stroke-renderer.js`

方針：

1. `TEGAKI_CONFIG.webgpu.enabled` は Phase 1c では `false` にする。
2. `fill-tool.js` の `_checkWebGPUSupport()` は、Phase 1c では `this.webgpuAvailable = false` にして即 return でよい。
3. `_fillLayerWithSDF()` は残してもよいが、通常実行経路からは呼ばれないようにする。
4. `stroke-renderer.js` 側に WebGL2 / MSDF の初期化分岐があっても、今回の修正で触る必要がなければ触らない。もし触る場合は「1cでは無効化」だけにする。

受け入れ条件：

- バケツ操作時に SDF / WebGPU 側へ入らない。
- コンソールに WebGPU / SDF 初期化失敗ログが出ない、または通常操作を妨げない。

## 2. レイヤーサムネイルが描画を反映しない問題を直す

対象：

- `tegaki_work/system/drawing/thumbnail-system.js`
- `tegaki_work/ui/layer-panel-renderer.js`
- `tegaki_work/ui/timeline-thumbnail-utils.js`

重要な原因候補：

PixiJS v8.17.0 の `renderer.extract.pixels()` は、ローカル型定義では raw `Uint8Array` ではなく以下を返す。

```ts
{
  pixels: Uint8ClampedArray,
  width: number,
  height: number
}
```

現在の `thumbnail-system.js` は `renderer.extract.pixels(_tempSprite)` の戻り値を raw 配列のように扱っている可能性がある。この場合、`pixels.buffer` が存在せず、空配列相当になり、`nonTransparent: 0` のままになる。

必須対応：

1. `renderer.extract.pixels()` の戻り値を PixiJS v8.17.0 形式で扱う。
2. `result.pixels` / `result.width` / `result.height` を優先する。
3. 互換のため raw配列が返る場合も扱えるようにしてよい。
4. `sourceRT.width/height` と `result.width/height` が違う場合はログで分かるようにする。

実装例：

```js
const result = this.app.renderer.extract.pixels({
    target: sourceRT,
    clearColor: '#00000000'
});

const clampedPixels = result?.pixels
    ? result.pixels
    : (result instanceof Uint8ClampedArray ? result : new Uint8ClampedArray(result?.buffer || result));

const sourceWidth = Math.round(result?.width || sourceRT.width || canvasWidth);
const sourceHeight = Math.round(result?.height || sourceRT.height || canvasHeight);
```

`extract.canvas()` を使う方が簡潔で確実なら、そちらを採用してもよい。ただし透明 alpha が維持されることを必ず確認する。

必須ログ：

```js
console.log('[ThumbnailSystem] updated', JSON.stringify({
    layer: layer.layerData?.name,
    isBackground,
    source: `${sourceWidth}x${sourceHeight}`,
    thumb: `${thumbW}x${thumbH}`,
    nonTransparentPixels,
    maxAlpha,
    centerRGBA: Array.from(px)
}));
```

このログは Phase 1c 確認用。確認後は debug 制御へ移すか削除対象として `PROGRESS.md` に書く。

受け入れ条件：

- 空の通常レイヤーは `nonTransparentPixels: 0`、`maxAlpha: 0` になる。
- 描いた通常レイヤーは `nonTransparentPixels > 0`、`maxAlpha > 0` になる。
- レイヤーパネルのサムネイルに線や塗りが反映される。
- 未描画部分は黒くならない。
- 非正方形キャンバスでもサムネイルのアスペクト比が崩れない。

## 3. ThumbnailSystem の重複名を解消する

対象：

- `tegaki_work/system/drawing/thumbnail-system.js`
- `tegaki_work/ui/timeline-thumbnail-utils.js`
- `tegaki_work/ui/layer-panel-renderer.js`
- 必要なら初期化側

現状の問題：

- `system/drawing/thumbnail-system.js` が `window.ThumbnailSystem` / `window.thumbnailSystem` を登録している。
- `ui/timeline-thumbnail-utils.js` も末尾で `window.ThumbnailSystem = new ThumbnailSystem(...)` としており、同じグローバル名を別実体で上書きする可能性がある。

方針：

1. レイヤーパネル用のサムネイル生成は `system/drawing/thumbnail-system.js` を正とする。
2. `window.thumbnailSystem` はこの正本だけを指す。
3. `ui/timeline-thumbnail-utils.js` 側は、名前を変える。例：

```js
window.TimelineThumbnailSystem = new TimelineThumbnailSystem(...);
```

4. クラス名も可能なら `TimelineThumbnailSystem` へ変更する。
5. `rg "ThumbnailSystem"` で参照を全確認し、レイヤー用とタイムライン用を混同しない。

受け入れ条件：

- `window.ThumbnailSystem` が複数用途で上書きされない。
- `window.thumbnailSystem.getThumbnail(layerId)` がレイヤーパネル用キャッシュを返す。
- タイムライン側が壊れない。もし未使用なら、未使用であることを `PROGRESS.md` に書く。

## 4. バケツツールを Phase 1c 用に復旧する

対象：

- `tegaki_work/system/drawing/fill-tool.js`
- 必要なら `tegaki_work/system/layer-system.js`
- 必要なら `tegaki_work/system/history.js`

最小復旧方針：

1. Phase 1c では WebGPU/SDF flood fill を使わない。
2. まず「バケツで現在レイヤーに塗りがRenderTextureへ焼き込まれ、サムネイルにも反映される」ことを復旧する。
3. `_fillLayerLegacy()` の矩形サイズは `CONFIG.canvas.width/height` ではなく、原則 `layerData.renderTexture.width/height` を使う。
4. fill 後は必ず `thumbnail:layer-updated` を emit する。
5. `layerSprite` を削除・破棄しない。

最低限の実装例：

```js
const width = Math.round(layerData.renderTexture.width || CONFIG.canvas.width);
const height = Math.round(layerData.renderTexture.height || CONFIG.canvas.height);

const fillGraphics = new Graphics();
fillGraphics.rect(0, 0, width, height);
fillGraphics.fill({ color, alpha });

app.renderer.render({
    container: fillGraphics,
    target: layerData.renderTexture,
    clear: false
});
```

注意：

- これは厳密な「閉領域バケツ」ではなく、現時点では全体塗りの復旧でも可。
- クリック地点からの閉領域 flood fill は Phase 1d 以降で設計してよい。
- ただし、もし既存実装範囲でCPU flood fillを狭く実装できるなら提案してもよい。無理に入れない。

受け入れ条件：

- バケツツールを選び、通常レイヤーをクリックすると塗りが見える。
- 塗り後にレイヤーサムネイルも更新される。
- リサイズ後のキャンバス全体サイズに合わせて塗られる。
- `layerData.layerSprite` が消えない。
- 背景レイヤークリック時は現在通り背景色変更または禁止動作を維持する。

## 5. 過剰診断ログを整理する

対象候補：

- `tegaki_work/system/drawing/pointer-handler.js`
- `tegaki_work/system/drawing/drawing-engine.js`
- `tegaki_work/system/layer-system.js`
- `tegaki_work/system/drawing/thumbnail-system.js`

方針：

1. 液タブ調査用の高頻度ログは、今回の主課題ではないため削減する。
2. 残すログは今回必要なものだけ。
   - `ThumbnailSystem` の `nonTransparentPixels/maxAlpha`
   - バケツ実行時の `method/size/layerId`
   - エラー・警告
3. `PointerHandler raw pointerdown`、`DrawingEngine down gate` は、必要なら `window.TEGAKI_CONFIG.debug` 条件へ入れる。
4. `PROGRESS.md` に「残したログ」と「削除予定」を書く。

受け入れ条件：

- 通常のペン描画中に console が大量ログで埋まらない。
- サムネイルとバケツの確認に必要なログは残る。
- ビルドエラーがない。
