## 1. サムネイルの黒背景を透明にする

対象：

- `tegaki_work/system/drawing/thumbnail-system.js`
- `tegaki_work/ui/layer-panel-renderer.js`
- 必要なら `tegaki_work/system/checker-utils.js`

問題：

通常レイヤーのサムネイルで、透明なはずのキャンバス領域が黒く出ている。これは CSS の外枠だけでなく、`ThumbnailSystem.generateLayerThumbnail()` が生成する PNG 自体に黒が焼き込まれている可能性がある。

疑い：

- `RenderTexture.create({ width, height })` 後に `renderer.render({ clear:true, clearColor:0x000000, clearAlpha:0 })` しているが、PixiJS v8 の実際の clear/extract 経路で透明が保持されず黒く抽出されている可能性。
- `extract.canvas(renderTexture)` の結果が、透明ではなく黒背景の canvas になっている可能性。
- CSS 側の checker 背景は、上に乗る `<img>` 自体が黒不透明なら見えない。

必須調査：

1. `sourceCanvas` または `thumbCanvas` の代表ピクセルを `getImageData()` で確認する。
2. 透明領域の alpha が `0` なのか、黒 `rgba(0,0,0,255)` なのかをログで確認する。
3. alpha が 255 なら CSS では直らない。サムネイル生成側を修正する。

修正方針：

- 通常レイヤーは透明背景の thumbnail PNG を生成する。
- 背景レイヤーだけは実背景色で塗る。
- 通常レイヤーの余白/非描画部分は transparent のままにし、UI側の checker または透明背景が見えるようにする。

実装候補：

1. 通常レイヤーは、`layer` Container 全体を一時RenderTextureへ描くのではなく、`layer.layerData.renderTexture` または `layer.layerData.layerSprite.texture` から直接抽出できないか確認する。
2. 直接抽出で alpha が保持されるなら、その経路を採用する。
3. 直接抽出でも黒くなる場合は、`renderer.extract.pixels()` 等で alpha を検証し、透明ピクセルを保持する経路へ切り替える。
4. `thumbCanvas` は必ず `ctx.clearRect(0, 0, thumbW, thumbH)` から始め、背景色で塗らない。

注意：

- 黒い線を描いたレイヤーが将来あり得るため、「黒を全部透明にする」雑な後処理は禁止。
- 透明判定が必要な場合は alpha を基準にする。
- 背景レイヤーの色は現状かなり良いので壊さない。

受け入れ条件：

- 空の通常レイヤーのサムネイルは黒くならない。
- 通常レイヤーに赤線だけ描いた場合、透明部分は checker または透明背景として見え、線だけが表示される。
- 背景レイヤーは実キャンバス背景色に見える。
- キャンバスを非正方形にリサイズしても、透明部分が黒くならない。

## 2. サムネイルの灰色/薄色枠を消す

対象：

- `tegaki_work/ui/layer-panel-renderer.js`

問題：

最新SSでは、黒いキャンバス領域の左右に淡い灰色/薄色帯が見えている。これはサムネイル画像の `object-fit: contain` で余った領域か、`.layer-thumbnail` の背景が見えている可能性が高い。

方針：

- 通常レイヤーの `.layer-thumbnail` 外枠背景は、灰色ベタではなく透明または checker にする。
- 背景レイヤーだけは実背景色にする。
- 枠線 `border: 1px solid #cf9c97` はUIとして必要なら維持してよいが、内部余白が灰色ベタに見える状態は避ける。

確認箇所：

- `createThumbnail()`
- `_updateSingleThumbnail()` 後に `thumbnailContainer.innerHTML = ''` した際、背景設定が失われていないか。
- `createFolderThumbnail()` は今回の主対象外だが、同じ灰色背景が見えるなら合わせて修正。

受け入れ条件：

- 通常レイヤーのサムネイル余白が灰色ベタに見えない。
- リサイズ後の縦横比差による余白も透明/checkerとして見える。
- 背景レイヤーの表示色は現状より悪化しない。

## 3. Space + ドラッグ移動は同時押し中だけ有効にする

対象：

- `tegaki_work/system/camera-system.js`

問題：

Spaceを押しながらドラッグするとキャンバス移動になるが、ドラッグ中にSpaceを先に離すとキャンバス移動が残り、操作しにくい。

方針：

- Space起点のキャンバス移動は、Spaceが押されている間だけ継続する。
- Spaceを離した瞬間に `isDragging` / `isScaleRotateDragging` / `canvasMoveMode` を false にする。
- pointerup を待たずに移動モードが終わるようにする。

現在の疑い：

- `pointermove` 側が `this.isDragging` だけを見て移動しており、`this.spacePressed` が false になった後でも移動が継続する。
- `keyup` が確実に拾えないケースがあると、drag状態が残る。

修正候補：

- Space起点のdragか右クリック起点のdragかを `this.dragStartMode = 'space' | 'rightButton' | null` のように記録する。
- `pointermove` 冒頭で、`dragStartMode === 'space' && !this.spacePressed` なら即座にdrag終了処理を呼ぶ。
- `keyup Space` でも同じ終了処理を呼ぶ。
- keydownでSpaceを押しただけでは `canvasMoveMode = true` にしない。実際にSpace+pointerdownした時に true にする方が自然。

受け入れ条件：

- Space + ドラッグ中にSpaceを離すと、その瞬間にキャンバス移動が止まる。
- Spaceを押していない通常ドラッグでは描画できる。
- マウス右クリックドラッグのキャンバス移動が壊れない。
- 液タブペン描画が再び `canvasMoveMode` にブロックされない。

## 4. キャンバスリサイズ時のセンタリング

対象：

- `tegaki_work/system/camera-system.js`
- 必要なら resize popup / resize event 周辺

問題：

キャンバスリサイズ後、キャンバスが画面外または意図しない位置へ飛ぶ。リサイズ後はキャンバス中心が画面中央付近に来るようにしたい。

緊急度：

- 低。軽微に直せるなら今回一緒に対応。
- 大きく触る必要があるなら `PROGRESS.md` に残して後続。

方針：

- `resizeCanvas(newWidth, newHeight, ...)` 後に、キャンバス中心を画面中央へ合わせる。
- 現在の `worldContainer.position.x += offsetX` / `position.y += offsetY` は、scale/rotation/flip を考慮しないため飛びやすい。
- `screen.center - worldContainer.toGlobal({ x: canvasWidth/2, y: canvasHeight/2 })` の差分で `worldContainer.position` を補正する方が安全。

実装候補：

```js
centerCanvasOnScreen() {
    const screen = this.app.renderer?.screen || this.app.screen || { width: window.innerWidth, height: window.innerHeight };
    const canvasCenter = this.worldContainer.toGlobal({
        x: this.config.canvas.width / 2,
        y: this.config.canvas.height / 2
    });
    this.worldContainer.x += screen.width / 2 - canvasCenter.x;
    this.worldContainer.y += screen.height / 2 - canvasCenter.y;
}
```

`resizeCanvas()` の最後に呼ぶ。ただし、リサイズUIに「左/中央/右」「上/中央/下」の配置指定がある場合は、その設計と衝突しないよう確認する。

受け入れ条件：

- リサイズ後、キャンバス中心が画面中央付近にある。
- ズーム倍率が勝手に変わらない。
- ガイド線、マスク、サムネイル更新が壊れない。
