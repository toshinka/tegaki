# Phase 2h — 履歴 / 変形 / Redo 安定性の棚卸し

> Phase 2g で、複数選択レイヤーの一括削除まで確認済み。
> Phase 2h は新機能追加より先に、アクティブレイヤー変形と Undo / Redo の安定性を確認する。

---

## 背景

オーナーから、アクティブレイヤーで回転や縮小を試した後に Redo が安定しない、元に戻らないことがある、という報告がある。

履歴・変形は、描画、レイヤー、RenderTexture、サムネイル、保存復元へ波及しやすい。Phase 2h ではまず棚卸しと再現条件の整理を行い、原因が狭く見えた場合だけ Codex が局所修正する。

---

## 目的

- Undo / Redo の現在の責務と command 構造を確認する。
- Vキー変形、レイヤー移動、回転、拡縮、bake が履歴へどう積まれるか確認する。
- Redo が不安定になる条件を、再現手順として絞る。
- 必要なら、アクティブレイヤー編集パネルのリセットボタンや警告表示など、後続UI候補を整理する。

---

## Gemini に任せてもよい範囲

- 現状棚卸し。
- 関係ファイルと EventBus の検索。
- `PROGRESS.md` への再現手順・責務整理の追記。
- 50行以内のヘッダー同期やコメント整理。

Gemini は独自判断で履歴本体、変形 command、RenderTexture 書き戻し、保存形式を変更しない。

---

## Codex 担当候補

- Redo / Undo の具体的な不具合再現。
- command の `do` / `undo` の不整合修正。
- 変形確定時の履歴スナップショットや bake 順序の補修。
- 必要な debug 限定ログの追加と削除。
- アクティブレイヤー編集パネルのリセットボタンを後続に回すか判断する。

---

## 調査対象候補

- 履歴管理ファイル。
- `system/layer-system.js`
- Vキー変形パネル / 変形 UI。
- キーボードハンドラ。
- サムネイル更新、RenderTexture bake、保存復元に関わる経路。

---

## 完了条件

- Redo 不安定報告に対して、再現できた/できない、条件候補、関係ファイルが `PROGRESS.md` に整理されている。
- 原因が狭い場合は、Codex が局所修正し、`npm.cmd run build` が成功している。
- 原因が広い場合は、実装せず後続設計へ回す判断が記録されている。
- Phase 2g の一括削除や属性パネル操作を壊していない。

---

## Phase 2h ではやらないこと

- 一括結合の本実装。
- 一括クリッピングの本実装。
- 独自D&Dの本実装。
- クイックパネル全面刷新。
- トップバー本実装。
- 範囲選択、図形、スポイトなどの新ツール実装。
