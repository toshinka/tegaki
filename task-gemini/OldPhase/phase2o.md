# Phase 2o — UI コンポーネント標準化と CSS 集約

> Phase 2n でアルバム UI の洗練が完了。
> Phase 2o は、アルバムで成功した「透過デザイン」「共通ボタン」「目に優しいスライダー」などの UI パターンをアプリ全体に横展開し、スタイルの大部分を CSS へ集約するフェーズ。
> これにより、AI が JS 側で煩雑なスタイル指定をせずとも、クラス指定のみで一貫した UI を構築できるようにする。

---

## 背景・目的

- 現状、各ポップアップやパネルごとに JS 内でインラインスタイルが散見され、AI が修正するたびに微細なデザインのズレやハルシネーションが発生しやすい。
- デザインルール（共通クラス、共通部品）を強固に定義することで、今後の「クイックパネル刷新」などの大きな改修を AI が正確に遂行できる「足場」を作る。

---

## 実装内容（Gemini主体）

1. **CSS 変数と共通クラスの再定義**
   - `--futaba-maroon` 等の配色変数を徹底活用。
   - `ui-icon-button` (small/medium/large) の挙動を安定させ、全パネルでこれを使う。
   - アルバムで作成した「目に優しいスクロールバー」を `.ui-scrollbar` として共通化。

2. **ポップアップ構造のテンプレート化**
   - `Resize`, `Export`, `Settings` などの各パネルを巡回し、以下の構造に統一。
     - `popup-panel--translucent` クラスの適用。
     - `ui-close-button` の統一配置。
     - タイトルバーとコンテンツエリアの余白ルールの統一。

3. **JS 側のインラインスタイル排除**
   - `dom-builder.js` や各 UI クラス内で `style.backgroundColor = ...` のように指定している箇所を、可能な限り CSS クラス（`classList.add`）へ置き換える。

4. **スライダー（input type="range"）の刷新**
   - 設定パネル等のスライダーも、CSS でカスタマイズされた「ふたば系デザイン」へ統一。

---

## 完了条件

- 全てのポップアップパネルが、アルバムパネルと同等の質感（透過・配色・角丸）で統一されている。
- スクロールバーが全パネルで「目に優しい透過栗色」になっている。
- JS コードから大量の `style.xxx = ...` 指定が消え、クラス名による制御が主体になっている。
- `npm.cmd run build` が成功している。

---

## Codex 判断へ戻すもの

- ポップアップの配置ロジック（座標計算など）を根本的に変える必要がある場合。
- 既存の EventBus 通信やデータフローに影響が出る構造変更。
