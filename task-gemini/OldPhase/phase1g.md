# Phase 1g — 保存・出力・レイヤー編集基盤

> Phase 1f で、描画/消しゴム/バケツ/リサイズをまたいだアンドゥ・リドゥと座標整合は実機確認で良好。
> Phase 1g は制作物を外へ出す、後から戻す、レイヤーを整理するための基礎機能を復旧・安定化する。

---

## 作業対象

```text
tegaki_work/
```

`PastFiles/` と `tegaki_phase0/` は参照のみ。通常作業では編集しない。

---

## 前提

- 標準描画は PixiJS v8.17.0 + RenderTexture へのライブラスター焼き込み。
- WebGPU / SDF / MSDF / WebGL2 Mesh 経路は凍結継続。
- Canvas2D を本番描画へ混ぜない。ただしエクスポート時の最終抽出・保存形式変換に Canvas を使うのは可。
- DPR は 1 倍固定。内部 2 倍化や `resolution: 2` は採用しない。
- 出力サイズは作業キャンバスサイズと一致させる。400x400 で描いたものは 400x400 の出力を標準にする。
- Phase 1f で追加された RenderTexture スナップショット履歴を壊さない。
- ペン、消しゴム、バケツ、リサイズ、アンドゥ/リドゥ、ステータス表示を壊さない。
- `dist/` は成果物として必要ない限り差分に残さない。

---

## 目的

1. PNG エクスポートを現行 RenderTexture レイヤー構造で確実に動かす。
2. 保存/読み込みの最小形式を決め、復旧する。
3. レイヤー複製、下レイヤー結合、クリッピング、フォルダのうち、安全な順に実装する。
4. レイヤーパネルを、今後レイヤー数が増えても破綻しにくい状態へ近づける。
5. 範囲選択・切り取りは調査と設計までに留め、無理に混ぜない。

---

## 優先タスク

### 1. 事前棚卸し

対象候補：

- `system/export-manager.js`
- `system/exporters/png-exporter.js`
- `system/exporters/webp-exporter.js`
- `system/exporters/psd-exporter.js`
- `ui/export-popup.js`
- `ui/album-popup.js`
- `system/layer-system.js`
- `ui/layer-panel-renderer.js`
- `system/data-models.js`
- `system/history.js`

確認すること：

- PNG エクスポートが現在の `Container + Sprite(RenderTexture)` レイヤー構造を正しく合成しているか。
- 背景レイヤー、透明背景、非表示レイヤー、レイヤー不透明度が出力に反映されるか。
- エクスポート時にカメラ位置・ズーム・UI 表示の影響を受けていないか。
- 現行の album/save 相当機能があるか。ある場合、何を保存しているか。
- レイヤー複製/結合/フォルダ/クリッピングの既存実装や残骸があるか。
- レイヤー操作を history に積む既存 command があるか。

棚卸し結果を `tegaki_work/PROGRESS.md` に追記する。

### 2. PNG エクスポート復旧を最優先

受け入れ条件：

- 現在のキャンバスサイズと同じ PNG が出力される。
- 表示中レイヤーだけが出力される。
- 背景表示 ON なら背景込み、背景非表示なら透明背景になる。
- ペン、消しゴム、バケツ、リサイズ後の描画が出力に反映される。
- 倍率指定や高度な書き出し設定は削除または無効化してよい。

### 3. 保存/読み込みの最小復旧

まずはアプリ内保存かローカルダウンロードのどちらが安全かを調査する。

候補：

- JSON + PNG/bitmap dataURL の簡易保存。
- レイヤーごとの PNG dataURL とメタデータ保存。
- IndexedDB / localStorage は容量と復旧性を確認してから採用する。

注意：

- 完成形の独自ファイル形式を急がない。
- PSD 互換や高度な圧縮は後続。
- 保存形式を決める前に、読み込み復元でレイヤー順・可視/不透明度・キャンバスサイズが戻るか検証する。

### 4. レイヤー編集の安全な順序

推奨順：

1. レイヤー複製
2. 下レイヤー結合
3. クリッピング
4. レイヤーフォルダ

理由：

- 複製と結合は RenderTexture のコピー/合成で比較的範囲が明確。
- クリッピングは描画・表示・出力・サムネイルに影響するため次。
- フォルダは UI、階層、出力順、履歴への影響が大きいため最後。

各操作は可能な範囲で `History` に載せる。

### 5. レイヤーパネル

- レイヤー数が増えたときにスクロール・選択・名前表示が破綻しないか確認する。
- 常時表示をサムネイル中心へ寄せる案を検討する。
- 大きい UI 改修は Phase 2 へ送ってよい。

## 実装せず Codex 判断へ戻すもの

- 保存形式の大きな設計判断が必要。
- PSD/WEBP/GIF/APNG をまとめて復旧する必要が出た。
- レイヤー階層モデルの大幅変更。
- History command 形式の大幅変更。
- RenderTexture の保持方式変更。
- WebGPU / SDF / MSDF / WebGL2 Mesh の復活。
- 範囲選択、切り取り、自由変形の本格実装。

---

## 受け入れ条件

- `npm.cmd run build` が成功する。
- PNG エクスポートが現行描画内容を正しく出力する。
- 可能なら最小保存/読み込みが動く。難しい場合は保存形式案と未実装理由を `PROGRESS.md` に残す。
- レイヤー複製または下レイヤー結合のどちらかが安全に動く。
- ペン、消しゴム、バケツ、リサイズ、アンドゥ/リドゥが退行しない。
- Console に通常操作ログが過剰に増えない。
