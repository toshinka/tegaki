# Phase 2g — 複数選択レイヤーの一括操作 / レイヤー行判定の軽量化

> Phase 2f で active と selected を分離し、Ctrl/Cmd クリックの個別選択、Shift クリックの範囲選択、F2/属性パネルでの名前変更が入った。
> Phase 2g は、この選択状態を実際のレイヤー操作へ少しずつ接続する。

---

## 目的

- 複数選択したレイヤーに対して、まず低リスクな一括操作を実装する。
- レイヤー行へ新しい判定を増やしすぎず、右サイドバーや属性パネルへ操作を寄せる。
- フォルダD&Dの不安定さを再燃させず、選択状態と操作バーの整合を優先する。

---

## 実装優先順位

1. **一括削除**
   - 選択中レイヤーが2件以上なら、右サイドバーの削除ボタンは選択中レイヤーをまとめて削除する。
   - 背景レイヤーは対象外。
   - フォルダ削除時に子も消える現行仕様との整合を確認する。

2. **一括クリッピング切替**
   - 属性パネルの paperclip または右サイドバー操作から、選択中の通常レイヤーへまとめてクリッピング ON/OFF できる形を検討する。
   - 初回は UI を急がず、仕様確認と小実装に留めてもよい。

3. **一括結合の設計**
   - 選択された複数レイヤーを結合する場合、順序、フォルダ跨ぎ、背景禁止、履歴をどう扱うかが重い。
   - Phase 2g 初回では、実装より先に条件整理を優先する。

4. **まとめてフォルダへ入れる**
   - 現行 SortableJS D&D では複数ドラッグに向かないため、Phase 2g では本実装しない。
   - 後続の独自D&Dまたは「選択中をアクティブフォルダへ入れる」ボタン式を検討する。

---

## 今回の追加メモ・画像資料の影響

### Phase 2gに直接影響するもの

- レイヤー行へ透明度スライダーや名前変更などの判定を増やさない。
- レイヤー行は選択、D&D、目/クリップ/フォルダ開閉など、行でなければ成立しにくい操作を優先する。
- 補助操作は右サイドバーと属性パネルへ寄せる。

### Phase 2後半またはPhase 3候補

- レイヤーパネルと右サイドバーの距離詰めは、UI仕上げ時に扱う。
- フリック削除は、D&Dや複数選択と衝突しやすいため、Phase 2g の一括削除後に再検討する。
- フォルダ内まとめ移動は、独自D&Dまたはボタン式のどちらにするか決めてから扱う。

### Phase 3以降へ送るもの

- クイックパネルの全面スリム化、カラーサークル/パレット再配置。
- 選択ツール時にクイックパネル内容を切り替える案。
- トップバー、Undo/Redo/ホーム/反転ボタンの環境設定付き導入。
- アクティブフォルダのVキー変形、選択範囲変形、物理演算連携。
- アルバムのクリック即ロード廃止とロードボタン化。

## Gemini に任せてもよい範囲

- 現状棚卸し。
- `PROGRESS.md` への整理記録。
- CSS の選択色、ボタン色、余白など50行以内の小調整。
- 一括結合や複数フォルダ投入の条件表作成。

Gemini は独自判断で `LayerSystem` の削除/結合/フォルダ移動本体を変更しない。

---

## Codex 担当候補

- 選択中レイヤーIDから安全な削除順を作る。
- 一括削除を履歴へどう積むか決める。
- 選択中に右サイドバー操作が単体/複数のどちらへ効くかを統一する。
- 一括クリッピングの最小実装または仕様確定。

---

## Phase 2g 初回の完了条件

- 選択中レイヤーが2件以上ある状態で、最低1つの一括操作が安全に動く。
- 通常の単一 active 操作は壊れない。
- 背景レイヤーは一括操作の対象外。
- レイヤー行クリック、Ctrl/Cmd、Shift、F2、属性パネルが競合しない。
- `npm.cmd run build` が成功する。

---

## Phase 2g ではやらないこと

- SortableJS の複数ドラッグ対応。
- 独自D&Dの本実装。
- 一括結合の無条件実装。
- クイックパネル全面刷新。
- トップバー本実装。
