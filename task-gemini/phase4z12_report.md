# Phase 4z12 — ClipAsset / Lane Migration Plan (GEMINI報告)

## 1. 現状の依存関係（棚卸し結果）

現在の実装において、タイムラインの「Lane (Track)」と「通常レイヤー (LayerSystem)」は強力に結合されています。

### 1.1. データモデル (`animation-data-model.js`)
- **`TimelineModel.syncWithLayers()`**: 最も核心的な依存箇所。`LayerSystem` から全レイヤーを取得し、1:1 で `LaneModel` を生成・更新します。現在、通常レイヤーがない Lane は存在できません。
- **`LaneModel.sourceLayerId`**: 各レーンがどの通常レイヤーと紐づいているかを保持しています。
- **`ClipInstanceModel.sourceLayerId`**: クリップ（セル）自体も紐付け ID を保持しており、移動（`moveClip`）時に移動先レーンの ID を継承します。

### 1.2. UI・プレビュー (`animation-table-popup.js`)
- **レーン表示**: `render()` 内でレーン名を通常レイヤー名から取得して表示しています。
- **キャプチャ (`_captureSelectedClip`)**: `lane.sourceLayerId` を元に、キャンバス上の特定レイヤーからピクセル情報を抽出しています。
- **プレビュー合成 (`_applyVisibilityPreview`)**: タイムラインに存在する全通常レイヤーを一時的に非表示にし、プレビューコンテナに Snapshot を描画する際、元のレイヤーの不透明度や合成モードを参照しています。

---

## 2. 将来構造案の整理

将来的に、Lane は通常レイヤーから独立し、「時間軸上の配置行」としての役割に専念します。

| 概念 | 現状 (Phase 4z11) | 将来候補 (Target) | 実装リスク |
| :--- | :--- | :--- | :--- |
| **Lane** | 通常レイヤーと 1:1 (同期) | 独立した配置行。手動追加・削除可能 | **高**: 同期ロジックの抜本的改修が必要 |
| **ClipAsset** | 内部レイヤーを持つ素材 | 保管庫に保存され、複数の Lane で再利用可能 | **中**: 参照カウント管理の厳密化 |
| **ClipInstance** | 特定 Lane/Frame への配置 | どの Lane にも配置可能。配置ごとに個別設定 | **低**: すでに基盤あり |
| **Internal Layer**| アセット内の作画階層 | アセット内の作画階層（マルチレイヤー） | **低**: Phase 4z6〜z10 で基盤済み |
| **Source Layer** | レーンの固定参照先 | 編集時（EDITモード）の一時的な描画先 | **中**: キャプチャ経路の柔軟化が必要 |

---

## 3. 案A / 案B 比較

### 案A: 既存Laneを徐々に独立化
- **概要**: `sourceLayerId` を保持しつつ、UI 上の表示（名前）だけを独立させる。
- **利点**: 既存プロジェクトとの互換性が高く、壊れにくい。
- **懸念**: 内部的な「1:1 同期」が残るため、自由なレーン追加が難しい。

### 案B: 新しい独立Laneを追加
- **概要**: 通常レイヤー同期レーンとは別に、完全に独立した「Animation Lane」を追加する。
- **利点**: プロ向けツールに近い柔軟な構成。通常レイヤーパネルを汚さずにアニメーションを作れる。
- **懸念**: 描画ターゲットをどこにするか（Virtual Layer Panel）の設計が複雑になる。

## 4. 次の小Phase候補

1. **Phase 4z13 — Lane Display Label MVP**
   - **内容**: レーンの表示名を「Lane 1, Lane 2...」のようにインデックス形式に変更。
   - **リスク**: 低。表示ロジックのみ。
   - **完了条件**: UI上でレイヤー名ではなくレーン番号が表示され、かつプレビューが動作すること。

2. **Phase 4z13 — ClipAsset Assignment Shelf**
   - **内容**: Asset Library で選択したアセットを、タイムライン上の既存クリップに「上書き割り当て」する機能。
   - **リスク**: 中。アセット参照の書き換えが発生。
   - **完了条件**: 割り当て後、プレビューに即座に反映され、共有アイコン（鎖）が正しく表示されること。

3. **Phase 4z13 — Virtual Layer Panel Connection Inventory**
   - **内容**: 通常レイヤーパネルの表示を、選択中クリップの「内部レイヤー」に切り替えるための技術調査とフック作成。
   - **リスク**: 高。LayerSystem への介入。
   - **完了条件**: パネルを切り替える信号を EventBus で疎通させ、ダミーの内部レイヤー一覧を表示できること。

---

## 5. 危険箇所・禁止すべき一括変更

- **`syncWithLayers()` の即時廃止**: 現在のプレビュー・編集・キャプチャがすべて停止します。
- **`sourceLayerId` の一斉削除**: 過去のデータが EDIT モードで編集不能になります。
- **レイヤーパネルの全置換**: 内部レイヤーへの「描き込み」手段が確立される前にパネルを差し替えると、作画ができなくなります。

## 7. Codex確認追記

以下はオーナー確認済みの設計前提として扱う。

- 編集対象の正本は、最終的には通常レイヤーではなく ClipAssetFolder / ClipAsset / ClipAssetInternalLayer 側へ寄せる。
- そのフレーム上に存在する ClipAssetFolder またはそれに相当する上位単位は、将来的にレイヤーパネル側にも表示される。
- レイヤーパネル上では、通常フォルダのさらに上位概念として ClipAssetFolder 相当のフォルダが見える想定。
- 通常フォルダとClipAssetFolderを混同しないため、色を濃くする、別系統の色にする、英字ラベルを付けるなどの視覚的区別が必要になる可能性が高い。
- ただし、レイヤーパネル側への表示やVirtual Layer Panel化は一気に実装しない。まずは設計・表示骨格・小さな疎通確認へ分割する。

Gemini報告の「Codexに判断してほしい点」への現時点の返答:

1. レーン表示は当面 `Lane 1` などの通し番号を候補にしてよいが、これは最終UIではなく、通常Layer名から切り離すための暫定表示として扱う。
2. 複数Laneが1つの通常レイヤーを共有する設計は、移行期の作業用ワークエリアとしては許容し得る。ただし正本データとして通常レイヤーへ戻す方向にはしない。
3. 次Phaseは、Lane表示名の変更だけに進むより、レイヤーパネルにCAF相当の上位概念をどう見せるかの棚卸し、またはAssetを既存Clipへ割り当てる小機能のどちらを先にするかをCodex側で選ぶのがよい。
