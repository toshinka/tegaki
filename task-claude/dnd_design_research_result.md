# レイヤーパネル/アルバム独自D&D設計 調査報告 (GEMINI報告)

## 結論

現状の SortableJS ベースの実装は、同階層の並べ替えには適しているものの、**「フォルダへの入れ込み」と「Sortable本来の並べ替え動作」が競合**しており、複雑な判定（マウス位置の比率計算など）で無理やり制御しています。

独自 D&D (Pointer Events) への移行は、操作の安定性と将来的な「複数レイヤー一括移動」の実現において**極めて有効**です。ただし、ドラッグ中のプレビュー表示やスクロール制御、フォルダ開閉状態との整合性など、実装すべき項目が多いため、段階的な移行を推奨します。

---

## 1. 現状分析

### レイヤーパネル (`ui/layer-panel-renderer.js`)
- **使用ライブラリ**: SortableJS
- **現在の仕組み**:
    - `initializeSortable()` で `Sortable.create` を実行。
    - `onMove` ハンドラ内で `_findFolderDropTarget` を呼び出し、マウス座標 (`document.elementsFromPoint`) から「フォルダの口」にヒットしているかを判定。
    - フォルダ投入判定中は `return false` して Sortable 本体の並べ替え動作を停止させている。
- **課題**:
    - フォルダへの投入判定が「要素の高さの 22%〜78%」といったマジックナンバーに依存しており、ペン操作や高速なドラッグで判定が漏れやすい。
    - ドラッグ中に実際の DOM が動くため、フォルダ投入判定と並べ替えが激しく競合し、意図しない場所へレイヤーが移動する事故が起きる。

### アルバムポップアップ (`ui/album-popup.js`)
- **使用ライブラリ**: SortableJS
- **現在の仕組み**:
    - `_renderGallery()` 内でギャラリー全体を Sortable 化。
    - 同階層のグリッド内並べ替えのみを行っており、フォルダ概念がないため比較的安定している。
- **課題**:
    - 独自 D&D 化するメリットは、レイヤーパネルとのコード共通化と、アルバム特有の「選択モード」中の複数枚一括移動への対応にある。

## 2. 独自D&D設計案 (Pointer Eventsベース)

### 基本コンセプト
「ドラッグ中は一切のデータ変更を行わず、UIフィードバック（プレビューと挿入線）に徹する。ドロップ時に一括でデータを更新する」

### 視覚的フィードバック
1.  **ドラッグプレビュー**: 掴んでいるアイテムの半透明コピーをポインタに追従させる。
2.  **挿入線 (Insertion Marker)**: アイテムとアイテムの間に水平線を表示。フォルダ内への挿入時はインデントさせて表示。
3.  **フォルダハイライト**: フォルダの上に重なった際、投入可能であればフォルダ全体を枠線や背景色で強調。

### 操作ロジック
- **判定領域**:
    - アイテムの上部・下部 25%: 並べ替え（挿入）判定。
    - フォルダアイテムの中央 50%: フォルダ投入判定。
- **フォルダ対応**:
    - ドラッグ中のポインタが閉じているフォルダの上に一定時間滞留した場合、自動でフォルダを展開する「オートオープン」機能（オプション）。
    - フォルダ外への脱出は、フォルダ外のアイテム間にドロップすることで実現。

## 3. アルバムD&Dへの横展開

- **共通化の可否**: 可能。
- **共通化戦略**:
    - `DraggableListController` (仮称) といったベースクラスを作り、Pointer Events の管理、ドラッグ開始・移動・終了の基本サイクルを共通化する。
    - 挿入位置の計算ロジック（リスト形式 vs グリッド形式）のみをサブクラスで切り分ける。
- **順序**: まず構造が複雑なレイヤーパネルで実装を確立し、そのエッセンスをアルバムへ展開する方が、技術的な困難を先に潰せるため安全。

---

## 4. 段階実装計画

### Phase A: 同階層並べ替えの独自D&D化
- **内容**: `SortableJS` を廃止し、Pointer Events によるリスト内並べ替えのみを実装。フォルダ投入は一旦「現在の位置での並べ替え」に留める。
- **触るファイル**: `ui/layer-panel-renderer.js`, `styles/main.css`
- **検証**: 背景レイヤーが動かないこと、ドラッグ中にリストが自動スクロールすること。

### Phase B: フォルダ投入・脱出の実装
- **内容**: フォルダ判定ロジックを追加し、`LayerSystem.moveLayerIntoFolder` と連携。
- **触るファイル**: `ui/layer-panel-renderer.js`
- **検証**: フォルダ内から別のフォルダへの移動、フォルダ外（ルート）への移動。

### Phase C: 複数選択レイヤーの一括移動
- **内容**: 複数選択されているレイヤーをひとつの束としてドラッグし、一括で移動・投入する。
- **検証**: 連続しないレイヤーを選択して移動した際の、移動後の並び順の整合性。

### Phase D: アルバムD&Dへの横展開
- **内容**: 共通化したロジックを `AlbumPopup` に適用。
- **触るファイル**: `ui/album-popup.js`
- **検証**: グリッドレイアウトでの並べ替え、大量のアルバムデータがある際のパフォーマンス。

## 5. リスクと対策

- **リスク1: 履歴の不整合**: D&D 中に `LayerSystem` のデータを頻繁に更新すると、Undo 履歴が大量に積まれる。
    - **対策**: ドロップ完了時に一度だけ `reorderLayers` または `moveLayerIntoFolder` を呼び出し、一回のアクションとして記録する。
- **リスク2: クリッピングレイヤー**: 親が移動した際にクリッピングが外れる、またはクリッピング関係にある複数枚をどう扱うか。
    - **対策**: ドラッグ開始時にクリッピング関係をチェックし、警告を出すか、またはセットでの移動を推奨する（今回は単体移動を許容し、ドロップ後に再計算）。
- **リスク3: スクロール競合**: リストが長い場合、上端・下端での自動スクロールが必要。
    - **対策**: `pointermove` 内で境界チェックを行い、`scrollBy` でリストを動かす処理を組み込む。

## 6. 検証手順チェックリスト

- [ ] **基本**: マウス、ペン（筆圧対応ペン）、指（タッチ）でドラッグが開始できるか。
- [ ] **並べ替え**: リストの一番上、一番下、中間に正しく挿入できるか。
- [ ] **フォルダ**: フォルダの上に乗せた時、ハイライトが出るか。ドロップして中に入るか。
- [ ] **脱出**: フォルダ内のアイテムを、フォルダ外のアイテム間にドロップして出せるか。
- [ ] **ガード**: 背景レイヤーを動かそうとした際、ドラッグが開始されないか。
- [ ] **スクロール**: アイテムを掴んだままリストを上端・下端に持っていき、スクロールされるか。
- [ ] **キャンセル**: ドラッグ中に `Escape` キーを押す、または右クリック、あるいはリスト外で離した場合に元の位置に戻るか。
- [ ] **Undo/Redo**: 移動後に Undo して元の位置・フォルダ階層に完全に戻るか。
