【新しい実装ロードマップ案】

全体の目標:

アクティブレイヤーのコピー時の劣化をなくす（ImageDataベースでのコピー）。
描画時の線のジャギー化を解消し、滑らかな線を実現する（ハイブリッド描画）。
既存の主要機能（レイヤー操作、ツールの切り替え、Undo/Redoなど）を維持・動作確認する。



フェーズ1: ハイブリッドレイヤー構造の導入と表示の確立

目的: レイヤーがImageDataと描画用Canvasの両方を持つハイブリッド構造に切り替え、そのImageDataの内容が正しく画面に表示されるようにする。描画ツールはまだ動かなくても良い。
内容:
Layer クラスに imageData と drawingCanvas/drawingCtx を導入。
LayerManager のレイヤー生成処理を修正し、新しい構造に対応させる。
既存の imageData があれば、その内容を drawingCanvas に描画して初期状態を同期させる。
CanvasManager で、各レイヤーの imageData を結合してメインの displayCanvas に描画する renderAllLayers 関数（または類似の機能）を実装。
レイヤー切り替えや表示/非表示で画面が更新されるようにする。
確認ポイント:
ツール起動時にエラーが発生しないか。
新規レイヤー追加時、既存レイヤー読み込み時に、開発者ツールで Layer オブジェクトが imageData と drawingCanvas を正しく持っているか。
既存のレイヤーデータ（もしあれば）が正しく画面に表示されているか（ジャギーな線でもOK）。
レイヤーパネルが表示され、背景色も正しく表示されているか。
レイヤーの表示/非表示切り替えで、画面表示が連動するか。
この時点ではまだ描画はできないこと。



フェーズ2: ハイブリッド描画システムの導入と主要ツールの動作確認

目的: 描画ツールの処理を drawingCanvas を使った滑らかな描画に変更し、描画完了時に imageData へ内容を反映させる。ペン、消しゴム、塗りつぶしなどの主要ツールが動作することを確認する。
内容:
CanvasManager や各描画ツールの描画ロジックを修正。ctx.putImageData() やピクセル直接操作から layer.drawingCtx を使った描画（beginPath, lineTo, stroke など）に置き換える。
描画が行われるたびに、アクティブレイヤーの drawingCanvas の内容をメインの displayCanvas に描画する処理を呼び出す。
重要: 描画完了時（マウスアップ時など）に、drawingCanvas の内容を imageData に反映させる処理（例: layer.imageData = layer.drawingCtx.getImageData(0, 0, width, height);）を追加する。
塗りつぶしツールの実装も drawingCanvas ベースに変更し、描画完了時に imageData に反映させる。
確認ポイント:
ペンツールで線が描けるか。線はジャギーではなく、滑らかか。
消しゴムツールが正しく動作するか。
塗りつぶしツールが正しく動作するか。
レイヤーを切り替えても正しく描画できるか。
ツールを切り替えても正しく動作するか。
描画後にツールを終了（マウスアップなど）してから、ツールを再開しても以前の描画が残っているか（imageData に反映されているか）。



フェーズ3: レイヤー操作とUndo/Redoの対応

目的: レイヤーの複製、結合、削除、クリアなどの操作が、新しいハイブリッド構造に対応して正しく動作することを確認する。また、Undo/Redo機能も正しく動作するようにする。
内容:
LayerManager の duplicateLayer、mergeLayer、deleteLayer、clearCanvas（アクティブレイヤーのクリア）などのメソッドを、imageData と drawingCanvas の両方を考慮するように修正。
特に duplicateLayer は、imageData を直接コピーすることで劣化を防ぐ。drawingCanvas も新しいものを作成し、コピー元の imageData から初期描画する。
mergeLayer は、結合元レイヤーの imageData を結合先レイヤーの imageData に合成し、drawingCanvas も更新する。
Undo/Redoスタックに imageData の状態を保存するように変更。Undo/Redo時に imageData を復元し、同時に drawingCanvas もその imageData から再描画するようにする。
確認ポイント:
新規レイヤー追加が正しくできるか。
アクティブレイヤーの複製（Shift+C）で劣化せずにコピーされるか。複製されたレイヤーに描画できるか。
レイヤーの結合（Shift+B）が正しくできるか。結合後も描画が残っているか。
アクティブレイヤーの削除（Shift+D）が正しくできるか。
アクティブレイヤーのクリア（Delete）が正しくできるか。
Undo（Ctrl+Z）とRedo（Ctrl+Y）が、描画操作、レイヤー操作（追加、複製、削除、結合、クリア）でそれぞれ正しく動作するか。
ショートカットキーが意図通りに動作するか。（以前の問題点への確認）



フェーズ4: その他の機能と最終調整

目的: 残りの機能（PNG出力、スポイト、画面移動など）を新しい構造に対応させ、全体的な動作確認とパフォーマンス調整を行う。
内容:
PNG出力の処理を imageData をベースに高品質に行うように修正（☆ ImageData実装プラン2（Claude).txt の「8.2 PNG出力の高品質化」を参考に）。
スポイトツールなどのピクセル情報を取得するツールを imageData から取得するように修正。
画面移動（ドラッグ）や拡大縮小（ホイール）などの表示操作が正しく動作するかを確認し、必要に応じて調整。
カーソル表示がツールに応じて正しく更新されるか確認。
その他、既存の全ての機能が意図通りに動作するかを網羅的に確認。
確認ポイント:
PNG出力で劣化せずに画像が保存されるか。
スポイトツールが正しく色を取得できるか。
画面の移動や拡大縮小がスムーズにできるか。
すべてのショートカットキーが正しく機能するか。
アプリケーション全体として、パフォーマンスに問題がないか。
反転バグなど、以前発生した問題が再発していないか。
一括での差し替えについて

上記のフェーズを順に進め、それぞれのフェーズで確認ポイントをクリアしていけば、最終的に完成したコードを core.js、ui.js、ToshinkaTegakiTool.html としてご提示します。その際、変更が少ない ui.js や html は、差分ではなく全体を提示し、差し替えるだけで済むようにします。
