## Codex調査報告 — tegaki_work 不具合調査 2026-05-14

### 総合判断

現在の不具合は大きく4系統です。

1. 鋭角ペンの三角形塗り：PerfectFreeHand本体より、WebGL2 Mesh経路・三角形分割・Mesh生成が怪しい
2. 液タブペン操作：描画ブロック条件、Vキー/変形モード残留、UIのpointerdown依存、Sortable設定が怪しい
3. レイヤーサムネイル：イベントpayload形式の不一致がほぼ原因
4. レイヤー変形後のペンサイズ/描画範囲ズレ：レイヤーContainer自体をscaleしているため、描画系も一緒に変形されている

---

### 要確認・要修正

#### 1. 鋭角ストロークが三角形に塗られる

- `tegaki_work/system/drawing/stroke-renderer.js:155`
  - ペン確定時、`webgl2Enabled && glStrokeProcessor` の場合は `_renderWithPerfectFreehand()` に入る。
  - ここで `Graphics.poly()` ではなく `Mesh + Geometry` 経路になる。

- `tegaki_work/system/drawing/webgl2/gl-stroke-processor.js:99`
  - perfect-freehandの輪郭を `earcut` で三角形分割している。
  - 鋭角・自己交差気味の輪郭で triangulation が破綻すると、巨大三角形が出やすい。

- `tegaki_work/system/earcut-triangulator.js:47`
  - earcut失敗時に fallback fan triangulation がある。
  - これは凸多角形限定なので、perfect-freehand輪郭には危険。

推奨対応：
- Phase 1cでは WebGL2 Mesh確定描画を一旦無効化し、ペン/消しゴムとも `Graphics.poly()` → RenderTexture焼き込みに統一。
- `stroke-renderer.js` の `renderFinalStroke()` では `_renderWithPerfectFreehand()` を呼ばず、まず `_renderFinalStrokeGraphics()` に固定する。

Gemini向け例文：
```js
// Phase 1cではMesh経路を使わない。鋭角ストロークの三角形化を避けるため、Graphics.poly()に統一する。
return this._renderFinalStrokeGraphics(strokeData, settings, mode);
```

#### 2. レイヤーサムネイルが通常描画後に更新されない

- `tegaki_work/system/drawing/brush-core.js:329`
  - `thumbnail:layer-updated` を以下の形で発火している。

```js
{
  component: 'drawing',
  action: 'stroke-completed',
  data: {
    layerIndex,
    layerId,
    immediate: true
  }
}
```

- しかし `tegaki_work/system/drawing/thumbnail-system.js:33` と `tegaki_work/ui/layer-panel-renderer.js:68` は `data.layerId` / `data.layerIndex` を直接読んでいる。
- つまり `data.data.layerId` に入ってしまい、通常描画後のサムネイル更新が通らない。

推奨対応：
- `brush-core.js` のイベントpayloadをフラットに統一する。
- または受信側が `payload.data ?? payload` を読むように統一する。
- Geminiには片方だけ直させず、プロジェクト全体で `thumbnail:layer-updated` の形式を統一させること。

Gemini向け例文：
```js
this.eventBus.emit('thumbnail:layer-updated', {
  layerIndex,
  layerId: activeLayer.layerData?.id,
  immediate: true
});
```

#### 3. V変形モード時だけサムネイルが見える理由

- `layer-transform.js:859` も `thumbnail:layer-updated` を発火しているが、こちらも `data: { layerIndex, layerId }` のネスト形式。
- ただし `layer-system.requestThumbnailUpdate()` は `layerIndex/layerId` をフラットに出す。
- 現在、複数形式が混在しているため「一部操作では出る、一部では出ない」状態になっている。

推奨対応：
- `thumbnail:layer-updated` は全てフラット形式に統一。
- `component/action/data` 形式を使いたい場合でも、サムネイル専用イベントだけは簡潔にする。

---

#### 4. レイヤー変形でペンサイズ・描画範囲も一緒に変形される

- `tegaki_work/system/layer-transform.js:236`
  - 変形時に `activeLayer` Container 自体へ `pivot/position/rotation/scale` を適用している。

```js
layer.pivot.set(centerX, centerY);
layer.position.set(centerX + x, centerY + y);
layer.rotation = rotation;
layer.scale.set(scaleX, scaleY);
```

- `tegaki_work/system/drawing/brush-core.js:161`
  - 描画プレビューも `activeLayer.addChild(this.previewGraphics)` で同じContainer配下に入る。
- そのため、変形モード中または変形が残っている状態では、ブラシ表示・描画範囲・座標がレイヤーscaleに巻き込まれる。

推奨対応：
- 短期修正：`DrawingEngine` は `layerSystem.vKeyPressed` 中は描画しないので、Vモード解除時に必ず `confirmLayerTransform()` または transform reset が走るか確認。
- 中期修正：変形対象Containerと描画受付Containerを分離する。
- Raster移行後は、変形確定時にRenderTextureへ焼き込み、Container transformを必ず `position=(0,0), scale=(1,1), rotation=0, pivot=(0,0)` に戻す。

Gemini向け注意：
- `activeLayer.scale` が1以外の状態でペン描画を開始してはいけない。
- `BrushCore.startStroke()` 前に activeLayer transform が残っていたら、描画を拒否するか確定処理へ誘導する。

#### 5. 液タブペンで描画できない件

直接原因はまだ断定不可。ただし確認ポイントは明確。

- `drawing-engine.js:65`
  - `cameraSystem.isCanvasMoveMode()` がtrueなら描画しない。
- `drawing-engine.js:69`
  - `layerSystem.vKeyPressed` がtrueなら描画しない。
- V変形モードが残留していると、ペン描画は完全に止まる。

まずGeminiに確認させる値：
```js
console.log('pointer', window.lastPointerType);
console.log('vKey', window.layerManager?.vKeyPressed);
console.log('canvasMove', window.cameraSystem?.isCanvasMoveMode?.());
console.log('activeLayer', window.layerManager?.getActiveLayer?.()?.layerData);
```

推奨対応順：
1. 液タブペン接触時に `pointer-handler.js` の `onPointerDown` が呼ばれているか確認
2. `DrawingEngine._handlePointerDown()` の early return 位置をログで確認
3. `vKeyPressed` / `canvasMoveMode` が残留していたら、モード解除処理を修正
4. `e.button` がペンで `0` 以外になる環境があるなら、`pointerType === 'pen'` では右クリック判定を緩める

---

#### 6. ポップアップの×、レイヤードラッグが液タブペンで効かない件

- ツールバー選択やレイヤー+が効くなら、ペンイベントそのものはブラウザに届いている可能性が高い。
- `quick-access-popup.js:263` は `pointerdown` のみ。
- `layer-panel-renderer.js:788` の Sortable は `forceFallback: false`。
- 液タブペンではネイティブHTML5 dragが不安定な場合があるため、Sortableは fallback強制が安全。

推奨対応：
- ポップアップ閉じるボタンは `pointerdown` だけでなく `click` も併用。
- Sortableは `forceFallback: true`, `fallbackOnBody: true`, `fallbackTolerance: 3` を試す。

Gemini向け例文：
```js
this.sortable = sortableLib.create(this.container, {
  animation: 200,
  forceFallback: true,
  fallbackOnBody: true,
  fallbackTolerance: 3,
  // 既存設定は維持
});
```

### Claudeへの申し送り

現状は、ラスター焼き込み化の方向性自体は正しいです。ただし「既存のベクター/Container変形設計」と「Raster RenderTexture設計」が混在しており、そこから副作用が出ています。

特に優先すべき設計判断は次の2つです。

- Phase 1cでは WebGL2 Mesh描画を切り、Pixi Graphics → RenderTexture焼き込みへ一本化するか
- レイヤー変形を Container scale で持ち続けず、確定時に必ずRenderTextureへ焼き込む設計で固定するか

Codex判断では、今は機能追加より「描画パイプラインを一本化して、AIが迷う分岐を減らす」ことを優先すべきです。
