1. 基本理念と前提条件
----------------------------------------------------------
- Tegaki v8 は「星型アーキテクチャ」を採用する。
  - 主星(MainController)が中核となり、衛星(Service/Tool)を統括する。
  - 衛星は主星にのみ依存し、衛星同士は直接依存しない。
  - 機能拡張は新しい衛星を追加することで実現する。

前提条件
- 実行環境：Chrome 最新、ローカルで HTML 直開き可能（file://）を想定
- 技術スタック：PixiJS v8（CDN読み込み）、ES2023（モジュールは使用しない）、fetch API 可
- 禁止事項：TypeScript / Vite / ESM / bundler / Babel 不使用
- グローバル名前空間：window.MyApp を採用（全モジュールはここへ登録）

1. 主星（MainController）は「発火・仲介・確定と記録」のみに責務を限定する（肥大化は許容するが機能は仲介に限定）。
2. 衛星（各サービス・ツール）は「単一責務」を持つ。衛星間直接参照は禁止。主星を介してのみ通信。
3. 命名と API を厳格に定義し、ファイル横断で誤解が起きないようにする。
4. エラーはサイレントに隠蔽せず、ErrorService 経由で主星に上げ、UI に通知する。
5. コンソールログは最小限にし、デバッグフラグで詳細化可能にする。
6. 非破壊レイヤー移動やベクターペン／消しゴムはツール単位で分離して実装する。


主要モジュール（衛星）一覧と責務（推奨ファイル名） — v8_StarDivide_rev5 準拠パッチ
----------------------------------------------------------
※ 現行ファイル構成（StarDivide rev5）に合わせ、各衛星の正式名称と責務を明確化しました。
※ 将来的に TransformService を切り出す可能性はあるものの、現状は Transform 機能は TransformTool と LayerManager の協調で扱います。

- MainController (main.html)
  - 初期化順序の保証、window.MyApp.mainApi の生成・登録。
  - 衛星の register(mainApi) 呼び出し順序の管理（Engine → LayerManager → PositionManager → ToolManager → UIManager → HistoryService → ErrorService）。
  - 全イベントの中継窓口（notify(event)）および dispatch ロジック。
  - 確定処理（confirm）を仲介し、HistoryService への履歴登録を行う。
  - ErrorService の一元受け取りと UI 通知（致命・回復可能の分離）。
  - mainApi に getEngineBridge(), getLayerList(), getAppState(), requestConfirm() 等の公開 API を提供する。

- DrawingEngine (drawing-engine.js)
  - Pixi.Application の生成・破棄（app, renderer, stage 管理）。
  - world/UI コンテナの分離と管理（world はユーザ描画、ui は固定 UI）。
  - EngineBridge を公開（最小 API: drawTemporaryStroke, commitStroke, clearLayer, takeSnapshot, app (read-only)）。
  - pointer イベント受け取り→ mainApi.notify 経路を提供（パン／描画のための低レベル入力流）。
  - レイヤー用コンテナの生成（ensureLayerContainer）と graphics 管理（graphicsList）。

- LayerManager (layer-manager.js)
  - レイヤーメタ管理（id,name,visible,locked,order 等）の単一ソース。
  - レイヤー生成／削除／順序変更（reorderLayer）を提供。
  - レイヤーに関する検証ロジック（locked チェック等）と prepareReorder() などの事前検証API。
  - DrawingEngine の EngineBridge と連携して、レイヤーコンテナの割当/解放を仲介する。
  - Transform の適用（簡易な coordinate bookkeeping）は行うが、複雑な非破壊変形は TransformTool 側で計算した結果を受け入れる。

- PositionManager (position-manager.js)
  - カメラ（パン/ズーム）状態の保持と更新（camera.x/camera.y/zoom）。
  - screen<->world の座標変換ユーティリティ。
  - カメラのプログラム的操作 API（setCamera, translateCamera, zoomTo 等）を提供。
  - パンは DrawingEngine の world コンテナ移動と協調して動作。

- ToolManager (tool-manager.js)
  - 登録済みツールの管理（registerTool(toolInstance)）。
  - 現在選択中ツールの切替（setActiveTool/getActiveTool）。
  - mainApi からのポインタ/イベントを受け取り、active tool の start/move/end にルーティング（イベント→メソッドマッピングの一元化）。
  - Tool のライフサイクル（register(mainApi) 呼び出しタイミング）を管理し、EngineBridge の利用可否を保証する。

- BrushTool / EraserTool / TransformTool (tools/xxx-tool.js)
  - 共通責務（Tool インターフェースを実装）：id, register(mainApi), start(point, meta), move(point, meta), end(point, meta), cancel(meta), serialize()。
  - BrushTool（tools/brush-tool.js）
    - ストローク点の収集、drawTemporaryStroke を呼んでプレビュー、確定時に mainApi.notify({type:'tools.strokeCommit', payload:{layerId, strokeData}}) を送る。
    - strokeData は { points: [{x,y}], style: { tool:'brush', size, color, alpha } } 形式を遵守。
  - EraserTool（tools/eraser-tool.js）
    - 消しゴム用スタイルを管理し、EngineBridge に渡す際に style.tool='eraser' を付与。
    - DrawingEngine 側で PIXI.BLEND_MODES.ERASE を用いて透明化を行う。環境依存の機能は mainApi を介して判定し、代替手段は主星の指示で行う（例：白塗りではない）。
  - TransformTool（tools/transform-tool.js）
    - 非破壊的移動・回転・スケールの操作を計算し、LayerManager へ変更リクエスト（layers.reorderRequest や layers.transformRequest）を発行する。
    - 大きな変形計算はツール側で行い、LayerManager に apply させる方式を標準とする。

- HistoryService (history-service.js)
  - record(action) を受け取り Undo/Redo の管理を行う（stack と index）。
  - action 形式：{ type: STRING, payload: OBJECT, snapshot?: ANY, timestamp: NUMBER, meta: { requestId } }。
  - main が確定したアクションのみを記録する（衛星から直接 record を呼ばせない）。
  - undo()/redo() は main を介して再適用/巻き戻し処理を行うためのフック (例: mainApi.requestConfirm or engineBridge.applySnapshot)。

- UIManager (ui-manager.js)
  - ツールバー、レイヤーパネル、ステータス表示等の DOM を生成・更新。
  - ユーザー操作（ボタンクリック、スライダー変更等）を mainApi.notify 経由で MainController に送信。
  - refreshLayers() 等の公開メソッドを持ち、MainController が初期化後に UI 同期を要求できるようにする。
  - ポップアップやツール設定UIは UIManager 内でサブコンポーネント化（例: ToolbarUI, LayerPanelUI）して肥大化を防ぐ。

- ErrorService (error-service.js)
  - 衛星からの error.recoverable / error.fatal イベントを受け取り、MainController へ通知または UIManager 経由で表示。
  - エラーメッセージの整形ルールを保持（ユーザ向け短文 / 開発向け詳細 stack を分離）。
  - ErrorService.receiveError(event) を全衛星で用いるルールに統一する（直接 console を吐くのは最小化）。

補足（命名・公開ポリシー）
- ファイル名：kebab-case（例: drawing-engine.js / layer-manager.js）。
- クラス名：PascalCase（例: DrawingEngine / LayerManager）。
- グローバル登録：
  - window.MyApp 直下には mainApi（MainController）と「外部連携が必要な少数のインスタンス」のみ登録。
  - 内部的に完結する衛星（LayerManager, PositionManager, ErrorService, HistoryService 等）は mainApi 経由でアクセスすること（直接グローバル登録は原則禁止）。
- メソッド名：camelCase、動詞始まり（createLayer, reorderLayer, startStroke）。
- イベント名：domain.action をドットで区切る（トップドメインは固定セット：[ui, tools, layers, history, engine, system]）。
- Tool インターフェース：start, move, end, cancel, serialize を必ず実装する。

運用上の注意
- TransformService が未導入のため、Transform 機能は現状 TransformTool と LayerManager の協調で運用する。将来の分離は容易にできるよう、TransformTool は変形データ（before/after）を明確に生成すること。
- Eraser の「真の消去（透明化）」は PIXI.BLEND_MODES.ERASE に依存するため、動作検証を行い、サポートされない環境では明示的にフォールバックポリシーを main に問い合わせる。
- UIManager の肥大化を避けるため、UI 内の責務が増える場合は UIManager 配下のサブモジュール（toolbar-ui.js / layer-panel-ui.js 等）として分割する。
（このパッチは既存 rev5 ファイルにそのまま適用可能な形で、ファイル名と責務を現状に合わせて整合しました。必要であれば、各ファイル冒頭に @module ヘッダー雛形を追加したバージョンも出力します。）



命名規約（必須）
----------------------------------------------------------
- ファイル名：kebab-case（例: layer-service.js）
- クラス名：PascalCase（例: LayerService）
- グローバル登録：window.MyApp.LayerService = LayerService
- メソッド名：camelCase、動詞始まり（createLayer, reorderLayer, startStroke）
- イベント名：domain.action をドットで区切る（例: tools.brushStart / layers.reorderRequest / history.record）
- Tool インターフェース関数名は統一（start, move, end, cancel, serialize）
- グローバル登録：
  - window.MyApp 直下には MainController と、外部連携に必要な公開APIのみを登録する。
  - 内部的に完結する衛星（例: PositionManager, ErrorService）は登録せず、MainController 経由でのみアクセス。
- イベント名：
  - トップレベルの domain は固定セットとする。
    [ui, tools, layers, history, engine, system]
  - 例: tools.brushStart / layers.reorder / ui.toggleSidebar / system.error


イベント仕様（JSON スキーマ簡易）
----------------------------------------------------------
- 基本イベント形式：
  { type: "domain.action", payload: {...}, meta: { requestId: "uuid", origin: "Tool|UI" } }
- 例：
  - tools.brushStart -> payload:{x,y,toolId,settings}
  - layers.reorderRequest -> payload:{fromIndex,toIndex,layerId}
  - layers.reorderConfirm -> payload:{fromIndex,toIndex,layerId}
  - tools.strokeCommit -> payload:{layerId,strokeData}
- 全イベントは MainController.notify(event) を経由して dispatch する


エラー方針（厳格）
----------------------------------------------------------
- 衛星は内部で try/catch して recoverable なエラーは main.notify({type:'error.recoverable', payload}) で報告
- unrecoverable は main.notify({type:'error.fatal', payload}) を呼び例外を main に委ねる
- main は ErrorService に喚起し、UIService 経由でユーザに簡潔に表示する（詳細ログは開発モードのみ）
- フォールバックや多重化は基本回避。代替動作が必要な場合は明示的に main が制御する


ログ方針
----------------------------------------------------------
- window.MyApp.debug = false/true フラグで制御
- main.log(level, tag, payload) を通じた一元ロギング。console.log の直接使用を禁止（コード規約）
- 本番（debug=false）：error と warning のみ出力


設計ルール（必読）
----------------------------------------------------------
1. 衛星間直接参照禁止（例外：Immutable 値の読み取りは可だが参照は不可）
2. 衛星は register(mainApi) を持ち、mainApi からのみ呼ばれるメソッドを限定提供する
3. Tool は DrawingEngine に直接描画要求を出さず、MainController へイベントを送って確定を行う（render のパフォーマンスを考え、必要時は部分的に最適化）
4. transform 操作は TransformService が計算し、LayerService に変換結果を返す（LayerService は変換の適用と履歴登録のためのデータを返す）
5. HistoryService は main の確定（confirm）を受けて record すること（衛星が直接 record してはならない）


ローリングバック・安全策
----------------------------------------------------------
- 改修は小さな段階（LayerService 切出し → Tool 分離 → TransformService 切出し）で行う
- 各段階で既存の rev タグを作成して、失敗時にすぐ戻せるようにする
- 破壊的変更は必ず feature branch で行う（GitHub 上のワークフロー）


備考
----------------------------------------------------------
- 命名やメソッドの意図が曖昧な箇所はすべて「役割コメント」を付ける（// ROLE: ...）
- ErrorService の導入で「隠蔽されたエラー」を排除する。フォールバックは main の指示で行う
- console.log は main.log を通して管理し、デバッグモード切替で制御すること

（続き：簡潔な API カタログ例 / イベント一覧）

---- MainController API (exposed to satellites via register(mainApi)) ----
mainApi = {
  notify(event)                // 衛星から main へイベント通知（必須）
  requestConfirm(request)      // 衛星が main に「確定」を要求する（例：reorder）
  getEngineBridge()            // EngineBridge への参照（描画用最小API） - readonly
  getLayerList()               // 層情報の読み取り用（immutable copy）
  debugMode: boolean           // デバッグフラグ
}

---- EngineBridge (DrawingEngine が main に登録する最小 API) ----
engineBridge = {
  app: PIXI.Application,                         // readonly
  drawTemporaryStroke(layerId, strokePoints)    // 描画バッファへの一時描画（ツールが呼ぶ）
  commitStroke(layerId, strokeData)              // 描画の確定（main が呼ぶ）
  clearLayer(layerId)                            // レイヤークリア
  takeSnapshot(layerId)                          // レイヤーのスナップショットを返す（undo用）
}

---- Tool インターフェース（各ツールが実装すべき） ----
Tool = {
  id: "brush" | "eraser" | "transform" | ...,
  start(point, meta),
  move(point, meta),
  end(point, meta),
  cancel(meta),
  serialize() -> object
}

---- イベント一覧（抜粋） ----
- tools.brushStart  payload:{toolId, x,y, settings}
- tools.brushMove   payload:{points:[{x,y}], toolId}
- tools.brushEnd    payload:{strokeData, layerId}
- layers.reorderRequest payload:{fromIndex,toIndex,layerId}
- layers.reorderConfirm payload:{fromIndex,toIndex,layerId}
- history.record payload:{action:{type,payload}}
- error.recoverable payload:{code,msg,context}
- error.fatal payload:{code,msg,stack}

---- 命名禁止リスト（例） ----
- 汎用的すぎる名前：Manager, Handler（単独で使う禁止。必ず Service/Tool の suffix を付ける）
- ambiguous verbs：moveLayer（→ reorderLayer or translateLayer に置き換える）
- 同一ファイル横断で同名メソッドを定義しない。やむを得ず同名にする場合は namespace prefix を必須（例：layer.reorder, transform.reorder）

---- 最短の実装チェックリスト ----
1. main.html に MainController で mainApi を生成し window.MyApp.mainApi として登録
2. engine-bridge.js（DrawingEngine）を作成し mainApi.registerEngine(engineBridge)
3. layer-service.js を作成し mainApi へ register
4. tool-manager.js と tools/brush-tool.js を作成し mainApi へ register
5. history-service.js を main 内で初期化し、各確定時に record を呼ぶ
6. UIService を作成し、ツールの UI 操作が main.notify(event) を呼ぶことを確認


☆ 主星の書き方ガイド
----------------------------------------------------------
- 主星は衛星を直接操作しない。すべてイベントや契約(bridge)を介す。
- 主星の責務は以下に限定する：
  - ユーザー入力と衛星呼び出しの仲介
  - 衛星の状態管理とライフサイクル管理
  - エラーハンドリングの集約
- 推奨構造
  - notify(event): 衛星からの通知を受け取る
  - dispatch(action): 衛星に対してイベントを送る
- 主星は依存方向の終点であり、他のモジュールに依存しない。

★ 衛星の書き方ガイド
----------------------------------------------------------
- 衛星は必ず以下のヘッダーを持つこと：
  @module  モジュール名
  @role    担当する役割
  @depends 依存先（基本は MainController のみ）
  @provides 公開するメソッドやAPI
  @notes   補足情報
- 衛星は主星にのみ依存する。他の衛星や外部UIに依存してはならない。
- 衛星は提供するAPIを明示し、内部実装の詳細は公開しない。
- エラー発生時は主星.notify() を通じて報告する。
- Consoleログは必要最低限に抑える。


※@notes に設計ルールや注意点を残す ことで将来の改修が容易になる
