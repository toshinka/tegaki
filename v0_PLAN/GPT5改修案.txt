想定されるエラー原因

今回の StarPlan_rev5 では 星型分離の徹底 を進めていますが、その過程でよく起こるのは以下です：

イベントの未定義 / type 不一致

main.html に 📢 Event from engine: engine と出ているが、event.type が undefined のまま渡されている。

つまり dispatchEvent 側で { type: "xxx", payload: ... } が正しく構築されていない。

改修ルールブックで「全イベントに type 必須」と定義したが、まだ徹底されていない。

モジュール依存の循環 or export/import 不一致

例：position-manager.js が drawing-engine.js の API を呼ぶが、main.html 側で先にロードされていない。

今の構成は <script> 直読みなので、ロード順序の依存が解決されていない可能性あり。

キャンバスサイズ・座標変換の初期化漏れ

Console に 0,0 のまま出ている場合、PositionManager が resize() のイベントを受けていない。

main → position-manager への初期呼び出し不足。

レイヤー追加ボタンが無効

ui-manager.js が layer-manager.js の公開 API を呼べていない。

ヘッダーに @provides createLayer とあるが、実装が export function createLayer ではなく、mainApi 経由に書き換えられていない可能性。

対策案
改訂版 対策案（ルールブック準拠）
A. イベント契約の厳格化（payload 軽量化込み）

各マネージャ（衛星）は Pixi の生オブジェクトを直接投げてはいけない。

主星が受け取るイベントは必ず シリアライズ可能な軽量データ に限定する。

例（旧・NG例）：

this.dispatchEvent({
  type: "layer-created",
  payload: { layer: newLayer } // ❌ Pixiオブジェクトをそのまま入れている
});


例（新・OK例）：

this.dispatchEvent({
  type: "layer-created",
  payload: { id: newLayer.id, name: newLayer.name }
});


✅ これにより Converting circular structure to JSON を防止し、主星と衛星の責務を分離できる。

B. ロード順序の固定化

依存関係を守りつつ、主星（main.html）の最後に制御を集約する。
推奨順序：

error-service.js

drawing-engine.js

layer-manager.js

position-manager.js

tool-manager.js

ui-manager.js

main.js（エントリーポイント）

✅ 依存解決を明示的にして、undefined 呼び出し を防ぐ。

C. 初期化フローの責務分離

主星が「誰をいつ起動するか」を決定する。

衛星は「自分自身の init」以外を呼ばない。

例（main.html 側の責務）：

await DrawingEngine.init(canvas);
await LayerManager.init(DrawingEngine);
await PositionManager.init(DrawingEngine);
await UIManager.init(LayerManager, PositionManager);


✅ 「主星が順序を管理、衛星は内部だけ初期化」に徹底できる。

D. デバッグログは payload 中心に

Pixi オブジェクトを直接ログに出さず、id / name / 座標など軽量情報のみ を出力。

例：

console.log("[LayerManager] createLayer", { id: newLayer.id, name: newLayer.name });
console.log("[PositionManager] worldToScreen", { input, output });


✅ 循環参照ログを避けつつ、バグ調査を可能にする。

今すぐ修正すべき箇所

dispatchEvent の payload

全ての dispatchEvent 呼び出しを確認し、Pixi オブジェクトを排除する。

id, name, 座標などの軽量化されたデータだけを渡すように修正。

ui-manager.js のイベント接続

レイヤーボタン → LayerManager.createLayer() へ正しく繋がっているか確認。

undefined を呼んでいる場合はイベント接続を修正。


👉 修正を進めるとしたら

まず dispatchEvent 全検索して payload を軽量化

そのあと main.html の初期化順序と責務分離



※お手本：イベント発火コード例

// LayerManager.js 内
createLayer(name = "新規レイヤー") {
  const newLayer = new PIXI.Container();
  newLayer.name = name;
  newLayer.id = `layer-${Date.now()}`;

  this.layers.push(newLayer);
  this.activeLayer = newLayer;

  // ✅ Pixiオブジェクトは payload に含めない
  this.dispatchEvent({
    type: "layer-created",
    payload: {
      id: newLayer.id,
      name: newLayer.name,
      index: this.layers.length - 1
    }
  });

  return newLayer;
}

// PositionManager.js 内
worldToScreen(worldX, worldY) {
  const screenX = worldX - this.camera.x;
  const screenY = worldY - this.camera.y;

  // ✅ 軽量化した情報のみ通知
  this.dispatchEvent({
    type: "position-updated",
    payload: {
      world: { x: worldX, y: worldY },
      screen: { x: screenX, y: screenY }
    }
  });

  return { x: screenX, y: screenY };
}

// DrawingEngine.js 内
startStroke(x, y, color, size) {
  this.currentStroke = { points: [{ x, y }], color, size };

  this.dispatchEvent({
    type: "stroke-started",
    payload: {
      color,
      size,
      start: { x, y }
    }
  });
}

// UIManager.js 内
registerLayerButton(button) {
  button.addEventListener("click", () => {
    const layer = this.layerManager.createLayer("UIから作成");

    // ✅ UIからの通知もシンプルに
    this.dispatchEvent({
      type: "ui-layer-button-clicked",
      payload: {
        id: layer.id,
        name: layer.name
      }
    });
  });
}

ポイント整理

Pixi の生オブジェクトは payload に含めない

Container や Sprite は JSON.stringify 不可 → NG

必要なら id, name, 数値情報だけ取り出す → OK

全イベントは { type, payload } 形式に統一

type が無ければ「Unhandled event type: undefined」が出る

payload は常にオブジェクト形式

デバッグログも軽量化

console.log("[Event]", type, payload);

Pixi オブジェクトをそのまま出力しない
