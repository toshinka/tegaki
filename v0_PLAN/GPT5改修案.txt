SatellitePlan_RuleBook_v7.txtの改修案

※これを元にしていてのエラーが多かったため、元の構造は活かしつつ改修すること。

🔍 点検結果と改善提案
1. 座標系統一ルールが弱い

v7では「Pixiを用いる」ことは明記されていますが、DOM座標・World座標・Layer座標の変換ルールが曖昧です。

結果として clientX/Y や getBoundingClientRect を直接参照する旧実装が残っていてズレが発生しています。

👉 改修案

「座標は必ず app.renderer.events.pointer.normalizeGlobal(e) で正規化し、toLocal() を通すこと」と明文化。

禁止事項として offsetX/offsetY・clientX/clientY の直接利用を追加。

2. DPR（DevicePixelRatio）・レンダラ解像度の扱いが未規定

v7では「キャンバスは400x400」とはあるが、CSSピクセルと物理ピクセルの扱いや renderer.resolution の使い方は未定義。

そのため古い「手計算で devicePixelRatio を掛ける」処理が残り、ズレの温床に。

👉 改修案

規則追加：「DPR補正はPixiの renderer.resolution に任せる。手計算で devicePixelRatio を掛けない。」

サイズ指定ルール：「CSS上のサイズとレンダラの論理サイズを一致させ、renderer.resize() は必ず専用のラッパ関数を通す」。

3. pointerup / cancel の取り扱いルールがない

現行のルールは「描画ツールの開始・終了イベントを実装する」程度。

しかし実際には pointerup が抜けて「描画が終わらない」問題が出ています。

👉 改修案

ルール追加：「pointerup/cancel は必ず window または setPointerCapture でグローバルに捕捉すること」。

「各ツールごとに isDrawing 状態を分離」する設計ルールも明記。

4. サムネイル生成基準が未定義

v7では「サムネイルを作成する」とだけあり、「キャンバス基準」か「描画内容基準」か未決定。

現状は描画内容基準だが、UI一貫性の観点から悩みどころ。

👉 改修案

ルールに選択肢を明記：

デフォルトは「キャンバス基準（統一スケール）」

オプションで「描画内容基準」を切替可能

実装ルール：「renderer.render(container, renderTexture) を用い、generateTexture の直接利用は禁止」

5. カメラ枠とキャンバスの関係が未定義

v7では「カメラ機能を追加予定」とあるが、座標変換や canvasContainer との階層関係は書かれていない。

今回の「カメラ枠が固定」問題はその不在が原因。

👉 改修案

ルール追加：「カメラ枠は canvasContainer の子要素とする」

もしくは「外に置く場合は canvasContainer.transform に追従する処理を必ず書く」

6. 旧実装の残骸排除ルールが不足

v7では「段階的に分割改修する」とあるが、旧処理を見つけた場合にどうするかのガイドラインが無い。

結果として ctx. 系の Canvas2D API や offsetX 参照が残りやすい。

👉 改修案

ルール追加：「Pixi導入後は CanvasRenderingContext2D API の直接呼び出しを禁止」

定期検査項目：「clientX/offsetX/getBoundingClientRect/generateTexture などが残っていないか grep でチェック」

📝 改修後ルールブックの例（追記部分のみ）
## 座標・座標変換ルール
- 入力座標は必ず `app.renderer.events.pointer.normalizeGlobal(e)` で正規化し、`toLocal()` を経由してレイヤー座標に変換すること。
- DOMイベントの `clientX/clientY`, `offsetX/offsetY`, `getBoundingClientRect()` を直接利用することを禁止する。

## 解像度・DPR
- DPR補正は Pixi の `renderer.resolution` に一元化し、devicePixelRatio を手計算で掛けることを禁止する。
- レンダラのリサイズは専用関数 `resizeRendererToCSS(app, width, height)` を通じて行い、CSSサイズと論理解像度を一致させる。

## ポインタイベント
- pointerup/pointercancel は必ず window または setPointerCapture を用いて捕捉する。
- ツールごとに isDrawing 状態を持ち、他ツールと共有しない。

## サムネイル
- デフォルトは「キャンバス基準」で生成し、オプションで「描画内容基準」を許容する。
- サムネイル生成は `renderer.render(container, renderTexture)` を用いる。`generateTexture()` の直接利用は禁止。

## カメラ
- カメラ枠は canvasContainer の子要素とする。
- 外部に置く場合は canvasContainer の transform に追従させる処理を必ず実装する。

## 旧実装の禁止
- CanvasRenderingContext2D API (`ctx.`) の呼び出しは禁止。
- 以下のコードパターンが残っていないか定期的に grep すること：
  - `clientX|clientY|getBoundingClientRect|offsetX|offsetY|generateTexture|ctx\.`


✅ まとめると：
ルールブック v7 には「Pixi で描画する」「段階的に改修」などの大枠はありますが、座標変換／DPR／pointerup 捕捉／サムネ生成基準／カメラ枠階層／旧API禁止 が未定義でした。
今回の不具合はまさにその隙間から発生しているため、これらを v8 ルールとして追記するのが望ましいです。

2) ポインタ（ペン）入力の統一実装（コピペ可）

以下を pen/interaction モジュール（例：draw/pen-engine.js）に入れて、既存のイベント処理を差し替えてください。ポイントは：

app.renderer.events.pointer.normalizeGlobal(evt) で正規化された global を得ること（ルールブック準拠）

canvasContainer.toLocal(globalPoint) → さらに layer.toLocal() で最終ローカル座標へ

pointerup/ cancel を window / renderer.view で捕まえ、pointer capture も可能なら使う（release漏れ防止）

DPR や renderer.resolution は手計算せず Pixi に任せる

// --- copy/paste: unified pointer handling (pen-engine.js) ---
function installPenHandlers(app, canvasContainer, getActiveLayer, onStrokeStart, onStrokeMove, onStrokeEnd) {
  // getActiveLayer: ()=> PIXI.Container (layer) for drawing
  let drawing = false;
  let pointerId = null;

  function normalizedGlobalFromEvent(e) {
    // Pixi v8 推奨経路：renderer.events.pointer.normalizeGlobal(e)
    // e は DOM PointerEvent または Pixi interaction event
    return app.renderer.events.pointer.normalizeGlobal(e);
  }

  function getLayerLocalPoint(globalPt, layer) {
    // stage -> canvasContainer -> layer と変換を統一
    // まず canvas space
    const canvasPoint = canvasContainer.toLocal(globalPt, app.stage);
    // 次に layer local
    return layer.toLocal(canvasPoint, canvasContainer);
  }

  function onPointerDown(e) {
    const globalPt = normalizedGlobalFromEvent(e);
    const layer = getActiveLayer();
    if (!layer) return;
    pointerId = e.pointerId ?? null;
    drawing = true;

    // pointer capture if available on the view (helps with lost-up issues)
    try { e.target && e.target.setPointerCapture && e.target.setPointerCapture(e.pointerId); } catch (err) {}

    const pt = getLayerLocalPoint(globalPt, layer);
    onStrokeStart(layer, pt, e); // user-supplied handler
  }

  function onPointerMove(e) {
    if (!drawing) return;
    const globalPt = normalizedGlobalFromEvent(e);
    const layer = getActiveLayer();
    if (!layer) return;
    const pt = getLayerLocalPoint(globalPt, layer);
    onStrokeMove(layer, pt, e);
  }

  function finishStroke(e) {
    if (!drawing) return;
    const globalPt = normalizedGlobalFromEvent(e);
    const layer = getActiveLayer();
    if (layer) {
      const pt = getLayerLocalPoint(globalPt, layer);
      onStrokeEnd(layer, pt, e);
    } else {
      onStrokeEnd(null, null, e);
    }
    drawing = false;
    pointerId = null;
    try { e.target && e.target.releasePointerCapture && e.target.releasePointerCapture(e.pointerId); } catch (err) {}
  }

  // Attach - use renderer.view (canvas element) for down/move, and window for up/outside
  const view = app.renderer.view;
  view.addEventListener('pointerdown', onPointerDown);
  view.addEventListener('pointermove', onPointerMove);
  // Catch up/out/cancel globally so we do not miss pointerup outside canvas
  window.addEventListener('pointerup', finishStroke);
  window.addEventListener('pointercancel', finishStroke);
  view.addEventListener('pointerleave', (e) => {
    // don't finish here; some apps use pointerupoutside. Choose finish or not based on UX.
  });

  return function uninstall() {
    view.removeEventListener('pointerdown', onPointerDown);
    view.removeEventListener('pointermove', onPointerMove);
    window.removeEventListener('pointerup', finishStroke);
    window.removeEventListener('pointercancel', finishStroke);
  }
}


期待される効果

右下でズレる現象の多くは、DOM→Canvas の変換か DPR の二重適用が原因。上実装で Pixi の正規化と toLocal を確実に使うためズレが激減します。

pointerup が抜けるケースは window で捕まえることで安定します。

3) DPR（Device Pixel Ratio）／キャンバスサイズ一貫化パッチ

DPR（window.devicePixelRatio）を自前で掛けていると二重掛けなどでズレます。Pixi の app.renderer.resolution と view の CSS サイズを一致させ、描画座標は Pixi に任せます。

// --- copy/paste: ensure renderer resolution / view size correct ---
function resizeRendererToCSS(app, cssWidth, cssHeight) {
  const view = app.renderer.view;
  // set CSS size
  view.style.width = cssWidth + 'px';
  view.style.height = cssHeight + 'px';
  // let Pixi decide backing size via resolution
  const resolution = window.devicePixelRatio || 1;
  app.renderer.resolution = resolution;
  app.renderer.resize(Math.round(cssWidth * resolution), Math.round(cssHeight * resolution));
  // Important: if using manual draw positions anywhere, do NOT multiply by devicePixelRatio again.
}


使い方：Canvas: 400x400px を表示したいなら resizeRendererToCSS(app, 400, 400) を呼ぶ。既存コード中に * devicePixelRatio や width = view.clientWidth * devicePixelRatio といったものがあれば削除／置換してください。

4) 「ペンを離しても描画が終わらない」原因対策

pointerup を拾えていない（上のグローバル捕捉で対処）

pointercancel / pointercapture の扱い不足 → 上ハンドラで setPointerCapture / releasePointerCapture を試す

内部 state が競合（複数ツールの isDrawing を共有）→ 各ツールごとに isActive を分離

追加チェックログ（デバッグ一時）：

// Quick debug help: dump pointer lifecycle

5) サムネイル（thumbnail）改修案 — 「基準を決める」 & 高速化

要件が2通りに分かれるため選択肢を用意します。あなたのツール上で UX を決めてください。

A. キャンバス基準（全サムネが同じ拡大率） — レイヤーはキャンバスサイズに合わせて縮小表示
B. 内容基準（描かれた範囲に合わせて拡縮） — 現状の「内容に合わせて拡縮」

一般的にツールとして 同じ拡大率（A） が分かりやすい（同じスケールで比較できる）。ただし、細部が多いと一部が潰れるため「内容に合わせる」オプションを付けると良い。

高速で安定した生成（RenderTexture） の実装例（キャンバス基準）：

// --- copy/paste: thumbnail generation (canvas-based uniform scale) ---
function generateThumbnail(app, container, thumbW = 64, thumbH = 64) {
  // create renderTexture of fixed size
  const rt = PIXI.RenderTexture.create({ width: thumbW, height: thumbH });
  // Use a temporary transform so we render the canvasContainer scaled to thumb size
  // We compute scale from canvas logical size -> thumb size
  const canvasW = app.renderer.width;    // backing pixels (resolution-applied)
  const canvasH = app.renderer.height;
  const scaleX = thumbW / canvasW;
  const scaleY = thumbH / canvasH;
  const scale = Math.min(scaleX, scaleY);

  // Save original transform
  const originalScale = container.scale.clone();
  const originalPos = { x: container.x, y: container.y };

  // center and scale for render into thumbnail
  container.scale.set(originalScale.x * scale, originalScale.y * scale);
  container.position.set(0, 0);

  // render
  app.renderer.render(container, { renderTexture: rt, clear: true });

  // restore
  container.scale.copyFrom(originalScale);
  container.position.set(originalPos.x, originalPos.y);

  // return a base64 or texture depending on needs
  const sprite = new PIXI.Sprite(rt);
  return { texture: rt, sprite };
}


注意：上の canvasW/H は Pixi の renderer 背景ピクセル数（app.renderer.width）なので、DPR の扱いも一致します。generateTexture() より renderer.render(..., renderTexture) のほうが高速かつ制御しやすい（ルールブック推奨）。

6) カメラ枠（camera frame）が固定の問題

原因は カメラ枠が canvasContainer の外にあり transform を受けていない か、または canvasContainer の transform を参照して更新していないことです。解決は二択：

カメラ枠（視界の四角）は canvasContainer の子にして、container の transform で一緒に動くようにする
または

カメラ枠はステージ上に置くが canvasContainer.transform に追従させる（変換を toGlobal / toLocal で計算して位置合わせ）

推奨：カメラ枠を canvasContainer の子にする（一番単純で安全）。

// --- copy/paste: attach camera frame to canvasContainer ---
function ensureCameraFrameAttached(canvasContainer, cameraFrameGraphics) {
  if (cameraFrameGraphics.parent !== canvasContainer) {
    // convert current global pos into container local to preserve screen position
    const globalPos = cameraFrameGraphics.getGlobalPosition();
    const local = canvasContainer.toLocal(globalPos, cameraFrameGraphics.parent);
    cameraFrameGraphics.position.set(local.x, local.y);
    canvasContainer.addChild(cameraFrameGraphics);
  }
}


もし意図的に分けたいなら、canvasContainer の transform に追従する実装（app.ticker で cameraFrame.position = canvasContainer.position 等）を置くこと。

10) まとめ（短く）

問題群の根本は 座標系のバラつき（DOM→Pixi 変換が混在）と DPR の二重処理/未処理、および pointerup の取りこぼし です。

対処は（A）Pixi の normalizeGlobal() と toLocal() を徹底、（B）renderer.resolution による DPR 統一、（C）pointerup を window で捕る、（D）サムネ処理を RenderTexture に統一してスケール基準を決める、の4本柱で十分です。

上に貼ったコードはそのまま差し替え可能な小パッチです。必要なら既存ファイルの中で**差し替え箇所を直接作ってパッチ形式（diff）**で出します。出力はファイル単位の差分が良ければ今すぐ作ります（どのファイルを差分で欲しいか指示してください）。
