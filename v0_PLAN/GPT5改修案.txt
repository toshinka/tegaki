要点（結論）

画面が“800×800に見える/倍化する”
→ ほとんどの場合 devicePixelRatio（DPR） と canvas の CSS サイズ／実ピクセルサイズの不一致、あるいは二重でスケーリングする処理（旧実装の CSS scale や manual scale の掛け合わせ）が原因です。Pixiの renderer.resize と view.style.width/height を正しく分離して下さい。

カメラ枠（白）が左上（0,0）基準で、キャンバスは中央基準
→ canvasContainer の pivot／position の扱いが混在しています。どちらかに統一（トップレフト基準 or 中央基準）し、ルールブックにある canvasContainer を座標の「単一真実（single source of truth）」にしてください。個人的には トップレフト基準（0,0 が左上） を推奨します（DOM座標と整合しやすい）。

リサイズがカメラ枠に影響しない
→ リサイズ処理が canvasContainer や cameraFrame に伝播していない。app.renderer.resize() 後に cameraFrame の描画／mask サイズ、canvasContainer の pivot/position を更新する必要があります。

キャンバス移動してもカメラ枠が連動しない（表示が消える）
→ cameraFrame が別の座標空間（例：stage直下）に置かれているか、mask や renderTexture の座標が固定されたまま更新されていません。cameraFrame を同一 container に置くか、toLocal/toGlobal で同期してください（ルールブック準拠）。

レイヤーサムネイルが画面中央のごく一部しか描かない
→ サムネイル生成時に getLocalBounds() を無視している、または renderTexture にレイヤー全体を正しいオフセットで描画していない可能性。layer.getLocalBounds() を使ってバウンディングに合わせて RenderTexture を作り、レイヤーをオフセットしてから renderer.render(layer, {renderTexture}) する必要があります。generateTexture() は単純で使いやすいが、トランスフォーム無視やスケール問題が起きやすいので RenderTexture + renderer.render を推奨（ルールブックに記載の方針と一致）。

改修方針（短い計画）

座標ユーティリティを作成（utils/coordinates.js） — 全ての pointer→canvas→layer の変換をここ経由にする。toLocal/toGlobal を必ず使う。

Pixi 初期化を一本化（core/pixi-engine.js） — DPR 対応、renderer.resize、app.view の CSS 幅設定をここだけで行う。

canvasContainer を“トップレフト基準”に統一 — pivot を 0,0、position を 0,0。キャンバス内でのカメラ移動は container.position で行う。

cameraFrame を canvasContainer の子にする（あるいは overlay コンテナで常に canvasContainer の位置に追随させる）。

サムネイル生成を renderTexture ベースに書き換え（utils/thumbnail.js）。

旧実装検索と除去 — CSS transform: scale() / JS 側の manual scale 倍加処理 / devicePixelRatio を直接乗算している箇所を grep で洗い出して取り除く。

すぐ使えるコード例（最小改修セット）

以下は pixi 初期化 → canvasContainer基準統一 → DPR対応 → カメラ枠を同一 container にする → サムネイル生成 をまとめたサンプルコードです。fixed_drawing_app.html の既存コードに切り替える箇所だけ置き換えてください。

注意：実ファイルが複数に分かれている想定なので、app や canvasWidth/Height 変数名は既存に合わせてください。下のコードは概念的に安全に入れ替えられるようにしてあります。

// core/pixi-engine.js（あるいは <script> 内にまとめて追加）
// 目的：DPR処理、canvasサイズの実ピクセルとCSSサイズを分離、canvasContainer基準を統一

const CANVAS_W = 400;
const CANVAS_H = 400;

function createApp(containerEl) {
  const DPR = window.devicePixelRatio || 1;

  // Pixi アプリ生成（解像度は 1 にしておき、内部で resize により実ピクセルを設定）
  const app = new PIXI.Application({
    width: CANVAS_W,
    height: CANVAS_H,
    backgroundAlpha: 0,
    resolution: 1, // we'll handle DPR on resize
    autoDensity: false, // we'll manage style + renderer size explicitly
    resizeTo: null,
    antialias: true
  });

  // mount canvas to DOM
  containerEl.innerHTML = '';
  containerEl.appendChild(app.view);

  // DO NOT set canvas width/height by CSS scale. Use style to set CSS size,
  // and use renderer.resize(width * DPR, height * DPR) to set backing pixels.
  function setCanvasSize(cssW, cssH) {
    // backing pixel size
    const backingW = Math.round(cssW * DPR);
    const backingH = Math.round(cssH * DPR);

    app.renderer.resize(backingW, backingH);
    // Set CSS size so the element is shown at logical pixels
    app.view.style.width = `${cssW}px`;
    app.view.style.height = `${cssH}px`;

    // Important: set stage scale to normalize coordinate space back to CSS logical pixels
    // so we can operate in logical coordinate space (0..cssW, 0..cssH)
    const scaleX = backingW / cssW;
    const scaleY = backingH / cssH;
    // Usually both equal DPR; but set stage.scale to 1/DPR so coordinates remain consistent?
    // Simpler: add a dedicated container with inverse scale so drawing coordinates == logical px
    if (!app.logicalRoot) {
      app.logicalRoot = new PIXI.Container();
      // This container's "display" is scaled down so children use logical pixels
      app.stage.addChild(app.logicalRoot);
      // Move current child containers into logicalRoot in existing code. We'll create canvasContainer below.
    }
    app.logicalRoot.scale.set(1 / DPR, 1 / DPR);
  }

  // central canvas container (where all canvas content lives)
  const canvasContainer = new PIXI.Container();
  canvasContainer.name = 'canvasContainer';
  // Use TOP-LEFT origin: pivot = (0,0), position = (0,0)
  canvasContainer.pivot.set(0, 0);
  canvasContainer.position.set(0, 0);

  // create overlay / ui containers as siblings under logicalRoot
  const uiContainer = new PIXI.Container();
  uiContainer.name = 'uiContainer';
  uiContainer.sortableChildren = true;

  // add to logicalRoot (so all are normalized to logical pixels)
  app.logicalRoot.addChild(canvasContainer);
  app.logicalRoot.addChild(uiContainer);

  // camera frame (white rectangle representing camera area) as child of uiContainer
  const cameraFrame = new PIXI.Graphics();
  cameraFrame.name = 'cameraFrame';
  uiContainer.addChild(cameraFrame);

  function drawCameraFrame(w, h) {
    cameraFrame.clear();
    cameraFrame.lineStyle(2, 0xffffff, 1);
    cameraFrame.beginFill(0x000000, 0); // transparent fill
    cameraFrame.drawRect(0, 0, w, h);
    cameraFrame.endFill();
    // ensure camera frame sits at same position as canvas (top-left)
    cameraFrame.position.set(canvasContainer.position.x, canvasContainer.position.y);
    cameraFrame.zIndex = 1000;
  }

  // initial sizing
  setCanvasSize(CANVAS_W, CANVAS_H);
  drawCameraFrame(CANVAS_W, CANVAS_H);

  // expose useful handles
  return {
    app,
    canvasContainer,
    uiContainer,
    cameraFrame,
    setCanvasSize,
    drawCameraFrame,
    DPR
  };
}


解説ポイント

renderer.resize に渡すのは「バックエンドの実ピクセル数（CSSサイズ × DPR）」です。app.view.style.width/height は CSS 表示サイズ（論理ピクセル）を与えます。これを混同すると「見た目2倍」などが起きます。

logicalRoot を作る手法は、DPR が高い環境でもアプリの座標系を「論理ピクセル（400x400）」に保つための一つの方法です。既存コードに autoDensity を使っている場合はそちらに合わせてくださいが、混在は避けるべきです。

カメラとキャンバスの連動（移動/ズーム）

キャンバス（canvasContainer）をパン/ズームしたら、cameraFrame は uiContainer に置くか canvasContainer の子にして、必ず同じ基準で位置更新を行います。簡易例：

// カメラを移動（例: マウスドラッグ）
function panCanvas(dx, dy) {
  // canvasContainer の position を更新
  engine.canvasContainer.position.x += dx;
  engine.canvasContainer.position.y += dy;

  // cameraFrame を同じ座標に移動させる（if uiContainer used）
  engine.cameraFrame.position.set(engine.canvasContainer.position.x, engine.canvasContainer.position.y);
}


もしくは cameraFrame を canvasContainer の子にしておけば自動で追随します（ただし cameraFrame の scale や rotation の影響を受ける点に注意）。

サムネイル生成（問題を起こしにくい実装 — RenderTexture 使用）

サムネイルが中央の小さな部分しか映さない根本原因は「描画対象のバウンディングを取らず、レイヤーのローカルオフセットを無視して render している」ことが多いです。以下は安全な関数：

// utils/thumbnail.js
function generateThumbnail(app, layerContainer, thumbW = 64, thumbH = 64) {
  // 1) get local bounds of layer
  const bounds = layerContainer.getLocalBounds();

  // handle empty layer
  if (bounds.width === 0 || bounds.height === 0) {
    // return blank sprite or placeholder
    const empty = new PIXI.Graphics();
    empty.beginFill(0x888888);
    empty.drawRect(0, 0, thumbW, thumbH);
    empty.endFill();
    return app.renderer.generateTexture(empty); // cheap placeholder
  }

  // 2) create render target sized to bounds (or scaled version)
  const rt = PIXI.RenderTexture.create({
    width: Math.ceil(bounds.width),
    height: Math.ceil(bounds.height)
  });

  // 3) temporarily offset layer so bounds.x/bounds.y -> 0,0 in RT
  const originalPos = new PIXI.Point(layerContainer.position.x, layerContainer.position.y);
  const originalParent = layerContainer.parent;
  // We'll render a clone or temporarily adjust position.
  layerContainer.position.x -= bounds.x;
  layerContainer.position.y -= bounds.y;

  // 4) render into RT
  app.renderer.render(layerContainer, { renderTexture: rt });

  // 5) restore position
  layerContainer.position.x = originalPos.x;
  layerContainer.position.y = originalPos.y;

  // 6) create thumbnail sprite, scale down to thumbW/thumbH
  const sprite = new PIXI.Sprite(rt);
  const scale = Math.min(thumbW / bounds.width, thumbH / bounds.height);
  sprite.width = Math.round(bounds.width * scale);
  sprite.height = Math.round(bounds.height * scale);
  return sprite;
}


注意点

generateTexture では layer の transform（pivot/scale）や親 container の transform を無視する場合があります。renderer.render(layer, {renderTexture}) の方が確実です。

getLocalBounds() を必ず使い、bounds.x/bounds.y を考慮してオフセットを掛けること。これをやらないとサムネイルはレイヤーの一部しか写しません。

ショートカット周りの注意（実装バグになりやすい箇所）

keydown を window に登録しているが、入力フィールドにフォーカスがあるときに誤動作する → event.target をチェックしてショートカットを抑止する。

重複登録（モジュールのホットリロードやファイル分割で同じハンドラを複数登録） → 初期化時に既存ハンドラを削除する仕組みを作る。例：window.removeEventListener('keydown', onKeyDown) の呼び出しを入れる。

既存の旧実装（scale *= 2 や CSS transform: scale(2)）が残っていると DPR と掛け合わせて2倍・4倍になる。

簡単なショートカット集中管理例：

// shortcuts.js
const ignoreInputs = ['INPUT', 'TEXTAREA'];

function onKeyDown(e) {
  if (ignoreInputs.includes((e.target && e.target.tagName) || '')) return;
  if (e.ctrlKey && e.key === 'z') {
    // undo
    e.preventDefault();
    undo();
    return;
  }
  // ... 他ショートカット
}

// init（初期化時に一度だけ）
function initShortcuts() {
  window.removeEventListener('keydown', onKeyDown);
  window.addEventListener('keydown', onKeyDown);
}
