❌ 問題になりやすい点

Pixi の Container 入れ替えによるレンダリング不整合

container.removeChild() → addChildAt() のような処理で階層移動している場合、RenderTexture にバインドされているサムネイル更新処理が切れることがある。

これが「サムネイルが真っ黒になる」原因になりやすい。

サムネイル生成時に描画が同期していない

renderer.render(layer, renderTexture) でサムネイル更新する際、描画フレームと競合して真っ黒のテクスチャが書き込まれることがある。

描画遅延や「ペンの線がワンテンポ遅れて見える」原因にもなる。

ドラッグ中の要素の参照を壊している

Pixi の DisplayObject は「同じ親に属するのは一度だけ」なので、DOM的に差し替え→再構築していると参照が切れてイベントが遅延する。

✅ 改善アプローチ
1. データモデルと表示の分離

クリスタや Photoshop 風の「レイヤーパネル」は UI側のリスト並べ替えと、実際のPixi Container階層は別にする 方式が堅牢です。

つまり：

内部的には layers[] の配列で Z-order を管理する

Pixi Container は必要な時だけ addChildAt(layerContainer, newIndex) で再配置

レイヤーパネルの UI（HTML側）は drag-and-drop ライブラリ（例えば SortableJS
）で操作し、並び順が変わったら内部の layers 配列を書き換える。

2. サムネイル生成の非同期化

requestAnimationFrame の中でサムネイル更新を行い、描画と競合しないようにする。

function updateThumbnail(layer, renderTexture, renderer) {
    requestAnimationFrame(() => {
        renderer.render(layer, { renderTexture, clear: true });
    });
}


こうすると「黒塗りサムネイル」問題が大幅に減ります。

3. 遅延描画対策

Pixi v8 の autoRender を使わず、明示的に renderer.render(stage) を呼ぶ。

特にペンツールのストローク中は「現在のレイヤーだけに描画」→「フルステージ再描画」に分けた方がレスポンスが良い。

4. よりAI実装向きな方法

「レイヤーパネル = DOM要素」＋「Pixi = 実描画」という分離方式。

Pixi の DisplayObject 自体をドラッグ移動対象にせず、HTML側の <ul><li> の順番をドラッグ&ドロップで入れ替え、その結果だけ Pixi Container の zIndex を更新。

container.sortChildren() が使えるので、Pixi 側は child.zIndex = newIndex を付けるだけで済む。

🔧 実装イメージ
// レイヤーの管理配列
let layers = [];

// パネルUIはHTML側のSortableJSでドラッグ入れ替え
Sortable.create(layerPanel, {
  onEnd: function(evt) {
    const moved = layers.splice(evt.oldIndex, 1)[0];
    layers.splice(evt.newIndex, 0, moved);

    // Pixi上のZ順更新
    layers.forEach((layer, i) => {
      layer.zIndex = i;
    });
    stage.sortChildren();
  }
});

// サムネイル更新
function refreshThumbnails() {
  layers.forEach(layer => {
    updateThumbnail(layer, layer.thumbnailRT, renderer);
  });
}

まとめ

原因: Pixi コンテナの直接入れ替え・描画競合でサムネイルが黒くなり、ストロークが遅延して見える。

改善策:

レイヤーパネル操作は HTML 側で行い、Pixi には「結果だけ」を反映。

サムネイルは requestAnimationFrame で非同期更新。

Pixi の zIndex＋sortChildren() を使って階層移動。


💡 起きていること

レイヤー選択イベント

パネルクリック → 「アクティブレイヤー切り替え処理」が走る

そのタイミングで renderer.render(layer, renderTexture) が走る or thumbnailSprite.texture が差し替えられる

Pixi 内部で一度クリア

clear: true を付けている場合、一度真っさらにしてから描画するので「白or黒背景の▣」が一瞬映る

次のフレームで正しい描画

その後 requestAnimationFrame でステージ全体がレンダリングされるので、結果的に「点滅」に見える

✅ 対策案
1. サムネイル更新を遅延させる

クリック直後に即更新するのではなく、次のフレームで更新。

function safeUpdateThumbnail(layer, renderTexture, renderer) {
  requestAnimationFrame(() => {
    renderer.render(layer, { renderTexture, clear: true });
  });
}


これで「クリック直後のフラッシュ」が消える可能性が高いです。

2. clear: false で前フレームを保持

サムネイル描画のときに clear: true を外すと、「一瞬クリアされる」挙動が無くなる。

renderer.render(layer, { renderTexture, clear: false });


ただしこれだと「古い描画が残る」場合があるので、まず renderTexture を一度専用の透明背景で初期化しておく と安心です。

3. 選択インジケータを Pixi に描かせる

もし「▣ 太枠」っぽいのが Pixi の Graphics で描かれているなら、それが選択中レイヤーのインジケータとして毎回出てる可能性あり。
この場合は：

選択中のレイヤーに outline を描く処理を削るか

CSS 側のパネルUIで「選択枠」を出すように変更する方が自然

4. ダブルバッファ的に一時テクスチャを使う

tmpRenderTexture を作り → そこにまず描画

それをサムネイルスプライトに貼る
こうすると「更新途中の黒画面」がユーザーに見えない。

📝 まとめ

原因 → 選択時にサムネイル用 renderTexture が一度クリアされ、描画タイミングがずれて一瞬黒（または枠だけ）になる

対策 →

サムネイル更新を requestAnimationFrame で遅延

clear: false を使う（初期化は別途行う）

選択枠の描画は Pixi ではなく HTML 側のUIに任せる
