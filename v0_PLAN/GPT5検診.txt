問題の本質

前回見た PenTool 実装では onPointerDown に moveTo(x, y) を呼んでいますが、その後の onPointerMove が **「現在の座標に lineTo するだけ」**になっているのが問題です。

PixiJS の Graphics は状態を保持しているため、もし最初の moveTo がイベント処理順序やタイミングで効いていないと、デフォルト位置 (0,0) から最初の lineTo(x,y) が走ってしまうのです。
これが「中央から描いたつもりが左上から斜め線になる」原因。

🔧 修正ポイント（PenTool 側）
修正案 1: onPointerMove で必ず「前の点から moveTo」
onPointerMove(x, y, event) {
    if (!this.isDrawing || !this.currentPath) return;

    const prev = this.points[this.points.length - 1];
    this.points.push({x, y});

    if (prev) {
        this.currentPath.moveTo(prev.x, prev.y); // ← 直前点を明示
    }

    if (this.smoothing && this.points.length > 2) {
        this.drawSmoothLine();
    } else {
        this.currentPath.lineTo(x, y);
    }

    this.currentPath.beginFill(this.color, this.opacity);
    this.currentPath.drawCircle(x, y, this.lineWidth / 2);
    this.currentPath.endFill();
}

修正案 2: drawSmoothLine() 内でも moveTo を補正
drawSmoothLine() {
    if (this.points.length < 3) return;

    const len = this.points.length;
    const p1 = this.points[len - 3];
    const p2 = this.points[len - 2];
    const p3 = this.points[len - 1];

    this.currentPath.moveTo(p2.x, p2.y); // ← ここで始点を補正

    const cp1x = p1.x + (p2.x - p1.x) * 0.5;
    const cp1y = p1.y + (p2.y - p1.y) * 0.5;
    const cp2x = p2.x + (p3.x - p2.x) * 0.5;
    const cp2y = p2.y + (p3.y - p2.y) * 0.5;

    this.currentPath.bezierCurveTo(cp1x, cp1y, cp2x, cp2y, p3.x, p3.y);
}

✅ まとめ

原因: PenTool が pointermove 時に始点を保証していないため (0,0) から線が引かれる。

対策: onPointerMove / drawSmoothLine 内で必ず moveTo を呼び、前回の座標を明示する。


*************************

消しゴム (EraserTool) は ペンと違って線分を引かずに「円スタンプを連続で置く」方式を採用しているため、先ほどのペンの (0,0) 直線問題は発生しにくいです。

✅ 良い点

drawEraseCircle(x, y) で必ず「座標を中心とした円」を描くので、moveTo / lineTo の扱いに依存しない。

onPointerMove でスムーズ補間処理が入り、座標が飛んでも穴があかない。

blendMode = PIXI.BLEND_MODES.ERASE を後から適用しているのも正しい（beginFill の前にやると描画結果がおかしくなることがある）。

背景レイヤー (layer0) への消去を禁止しているのも 👍

🔍 気になる点・改善提案

blendMode の適用タイミング

現在は onPointerDown のあとに一度だけ this.currentErasePath.blendMode = ERASE を設定しています。

問題はありませんが、より安全のために drawEraseCircle 内でも blendMode を再保証しておくと、不意に Graphics の状態が変わっても確実に消去動作します。

drawEraseCircle(x, y) {
    if (!this.currentErasePath) return;
    this.currentErasePath.blendMode = PIXI.BLEND_MODES.ERASE; // 再保証
    this.currentErasePath.beginFill(0xffffff, this.eraserOpacity);
    this.currentErasePath.drawCircle(x, y, this.eraserSize / 2);
    this.currentErasePath.endFill();
}


大きい消しゴムサイズでの補間ステップ数

今は steps = Math.ceil(distance / (this.eraserSize / 4)) ですが、大きなサイズを設定すると「補間が粗くて消し跡が途切れる」可能性があります。

距離に応じたスムーズさを保ちたいなら、/ 2 くらいにするとより自然に。

パフォーマンス最適化

今のやり方だと PIXI.Graphics に円を都度追加していくため、長い消去経路ではオブジェクトが膨れ上がります。

後で最適化するなら PIXI.SimpleRope + テクスチャ筆方式（ブラシテクスチャを並べる）にすると軽量化できます。

座標管理が整理されているかを気にしているんですね 👍

現状コード（pen-tool.js・eraser-tool.js・canvas-manager.js をざっと見た範囲）だと、以下の状態です：

✅ 良い点

イベント座標 (event.data.global) を直接使っている
→ PixiJS の InteractionManager は global をキャンバス全体の座標系に変換してくれるので、ズレにくい。

レイヤー（layer0, currentLayer）ごとに描画している
→ 将来レイヤーを増やす際に分離しやすい。

ペンと消しゴムが同じ座標管理フロー
→ 一貫性があるので不具合が混在しにくい。

⚠️ 問題になりそうな点

キャンバス移動（パン・ズーム）を入れると座標変換が破綻する可能性

今は event.data.global をそのまま使っていますが、
canvasContainer に scale や position をかけたとき、ツール側がズーム・移動を考慮できない。

本来は toLocal を通して「レイヤー座標」に変換してから描画すべき。

const pos = this.canvasManager.currentLayer.toLocal(event.data.global);


こうすれば、キャンバスを動かしてもツール側は常に「レイヤーのローカル座標」で扱える。

座標管理がツール側に散らばっている

PenTool と EraserTool それぞれで event.data.global を参照していて、
将来「タッチ補正」「座標変換」「グリッドスナップ」などを入れると重複改修が必要になる。

💡 改善提案

CoordinateManager（仮）を作る
→ すべてのツールは「ローカル座標変換済みの値」だけを受け取る。
例：

// utils/coordinate-manager.js
class CoordinateManager {
    constructor(canvasManager) {
        this.canvasManager = canvasManager;
    }
    getLocalPosition(event) {
        return this.canvasManager.currentLayer.toLocal(event.data.global);
    }
}
window.Tegaki.CoordinateManager = CoordinateManager;


これを PenTool / EraserTool で使えば、将来ズームやパンを入れてもツール修正が不要になります。

描画ロジックと座標ロジックを分離

ツールは「点を受け取って描く」だけに集中。

座標系は CoordinateManager か CanvasManager が責任を持つ。

ToolManager 側は問題なし。そのままでOK。
