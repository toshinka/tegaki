【要点（結論）】
1) 最初の致命的原因：managers/tool-manager.js における構文エラー（Uncaught SyntaxError: Unexpected token '.'）が原因でそのファイルの実行が中断され、結果として window.Tegaki.ToolManager が定義されないため「is not a constructor」エラーが発生しています。
   → 構文エラーでスクリプト全体が評価されないと、その後で new しようとしても未定義扱いになります（bootstrap/log に出ている現象）。:contentReference[oaicite:0]{index=0}

2) 二次的な問題：スクリプト読み込み順が不適切である可能性。Manager（ToolManager, CanvasManager 等）定義前に AppCore 側が参照していると未定義エラーになります。bootstrap の起動フローと index.html の <script> 順を必ず整列させる必要があります。:contentReference[oaicite:1]{index=1}

3) 余談だが Pixi v8 の初期化警告（powerPreference 警告）は致命的ではないが、WebGPU/adapter の差分に注意して初期化フローを分離しておく方が堅牢です。:contentReference[oaicite:2]{index=2}

-------------------------------------------------------------------------
【即効でやること（優先順位順）】
A. tool-manager.js の 479 行付近の構文エラーを確実に潰す（optional chaining '?.' やクラス外の余分な '.' が原因であれば明示的な if-check に書き換える）。
   - 理由：このファイルがシンタックスエラーならそれ以降が実行されず「ToolManager がコンストラクタでない」になる。:contentReference[oaicite:3]{index=3}

B. 各 manager スクリプトは必ず `window.Tegaki` 名前空間に「クラス定義（またはファクトリ）→ 登録」の順で公開する。
   - 例：`class ToolManager { ... } window.Tegaki.ToolManager = ToolManager;`（必須）
   - 理由：グローバル参照で new している既存コードとの互換性を担保するため。

C. index.html の script 読み込み順を厳格にする（下記「起動フロー」参照）。特に managers と utils は AppCore / bootstrap より先に来ること。

D. AppCore 側は manager を new する前に existence check と try/catch を追加し、失敗時はデバッグ情報を出す（init を中断しても原因が特定しやすいようにする）。:contentReference[oaicite:4]{index=4}

E. optional chaining 等のモダン構文は避ける（ESM/Bundler/トランスパイル禁止条件のため）。代わりに古典的なガード (if (obj && obj.prop) ...) を使う。

【起動フロー（簡潔で正しい順）】
（必要なスクリプト順。HTML 内の <script> はこの順に並べる）
1. minimal-deps.js（ポリフィルや軽いユーティリティ。名前空間初期化）
2. utils/* （error-manager.js, config-manager.js, event-bus.js, coordinate-manager.js, navigation-manager.js, record-manager.js, shortcut-manager.js）
3. managers/* （canvas-manager.js, tool-manager.js, 他の manager）
4. assets/icons.js（アイコン置換ロジック等）
5. tools/* （abstract-tool.js, pen-tool.js, eraser-tool.js 等）
6. debug/phase15-integration-test.js （デバッグ用、任意）
7. js/bootstrap.js（DOM ready → TegakiApplication.initialize() 呼び出し）
8. js/main.js（もしあるなら UI 初期化等）

理由：utils が提供する isReady / EventBus / ErrorManager を managers が利用し、managers がツールや canvas を提供して初めて AppCore が安全に起動できるため。bootstrap は最後に実行。

-------------------------------------------------------------------------
【設計ルール（守ること）】
- すべての公開クラス/関数は window または window.Tegaki 名前空間に露出する。
- 各ファイルは単一責務（single responsibility）に厳密に従う。
- ES2023 の構文を用いるが、ブラウザネイティブで互換性がない機能（optional chaining 等）は使わない。
- 例外発生箇所は try/catch で拾い、ErrorManager に渡す。
- Manager の API 契約（少なくとも isReady(), init(), dispose() を持たせる）。未実装ならデフォルト実装を注入する。

【最小限の修正サンプル： tool-manager.js（完全置き換え用の安全版）】
※実運用の全実装ではなく、安全に読み込めてエラーを出しやすい「置き換え用テンプレート」です。これにより SyntaxError を潰し、window.Tegaki.ToolManager が確実に存在する状態を作れます。

--- コピーして tool-manager.js を丸ごと置き換えてください ---
/* FILE: managers/tool-manager.js
   RESPONSIBILITY: 安全な ToolManager 定義（ES2023 準拠だが optional-chaining 等は排除）
*/
(function(){
  // 名前空間初期化
  if (!window.Tegaki) window.Tegaki = {};

  // シンプルなデフォルトエラー処理（ErrorManager がある場合はそちらに投げる）
  function reportError(e){
    try {
      if (window.Tegaki.ErrorManagerInstance && typeof window.Tegaki.ErrorManagerInstance.handle === 'function') {
        window.Tegaki.ErrorManagerInstance.handle(e);
        return;
      }
    } catch(_) {}
    console.error('[ToolManager] ', e);
  }

  // ToolManager クラス（最低限の責務）
  function ToolManagerConstructor(options){
    this.options = options || {};
    this.tools = {};
    this.activeTool = null;
    this._isReady = false;
  }

  ToolManagerConstructor.prototype.init = function(){
    try {
      // 必要ならここで依存 manager の存在チェックを行う
      // 例: if (!window.Tegaki.CanvasManager) throw new Error('CanvasManager required');
      this._isReady = true;
      return true;
    } catch (e) {
      reportError(e);
      this._isReady = false;
      return false;
    }
  };

  ToolManagerConstructor.prototype.isReady = function(){
    return !!this._isReady;
  };

  ToolManagerConstructor.prototype.registerTool = function(key, toolInstance){
    if (!key) throw new Error('registerTool: key required');
    this.tools[key] = toolInstance;
  };

  ToolManagerConstructor.prototype.getTool = function(key){
    return this.tools[key] || null;
  };

  ToolManagerConstructor.prototype.setActiveTool = function(key){
    var t = this.getTool(key);
    if (t) { this.activeTool = t; return true; }
    return false;
  };

  ToolManagerConstructor.prototype.dispose = function(){
    // 簡易 dispose
    this.tools = {};
    this.activeTool = null;
    this._isReady = false;
  };

  // 公開（必ず constructor を直接公開する）
  window.Tegaki.ToolManager = ToolManagerConstructor;
})();
--- end of file ---

解説：
- 上のテンプレートは「class」構文を避けて function/prototype にしてあります。これにより古い JS エンジンでも構文エラーを出しにくくなります（optional-chaining・private fields 等を避けるため）。この修正で "Unexpected token '.'" の発生源を排除できるはずです。:contentReference[oaicite:5]{index=5}

【AppCore 側の安全対策（app-core.js に入れるべき簡易ガード）】
- manager を new するコードは次のように保護する：
  try {
    if (typeof window.Tegaki.ToolManager === 'function') {
      this.toolManager = new window.Tegaki.ToolManager({/* options */});
      if (!this.toolManager.init()) throw new Error('ToolManager.init() failed');
    } else {
      throw new Error('ToolManager not available or not a constructor');
    }
  } catch (e) {
    // ここで ErrorManager に渡しつつ初期化を中断
    console.error(e);
    window.Tegaki.ErrorManagerInstance && window.Tegaki.ErrorManagerInstance.handle(e);
    throw e; // もしくは graceful fallback
  }
- これにより「is not a constructor」のエラーメッセージ発生箇所を捕まえ、原因が「未定義」なのか「初期化失敗」なのかを区別できます。:contentReference[oaicite:6]{index=6}

【coordinate-manager の警告対策】
- "未初期化状態での変換要求" は coordinate-manager の init チェック不足による呼出し順問題。
- 対策：coordinate-manager 側に `screenToCanvas()` など外部公開 API の先頭で `if (!this.isReady()) { throw new Error('CoordinateManager not ready'); }` を入れる。呼び出し側（UI ハンドラ等）は try/catch でガードする。

【補足（なぜ optional chaining を避けるか）】
- ユーザー条件で Vite/TypeScript/ESM 等を禁止しているためトランスパイラ無しでブラウザネイティブのみで動かす必要があります。optional chaining 等が原因で古い実行環境や不可解なツールパイプラインで SyntaxError を誘発するケースがあるため、手作業で安全なガードコードに置き換えるのが最も堅実です。:contentReference[oaicite:7]{index=7}
