🔍 原因

CoordinateManager のロード順問題

index.html の <script> 読み込み順で coordinate-manager.js が遅れている/読み込まれていない可能性。

CoordinateManager 内部の初期化が未完

ログ上では「CoordinateManager完全修正版 Loaded」と出ているが、実際に TegakiApplication.initializePhase15Managers() 呼び出し時に window.CoordinateManager が undefined 扱いされている。

つまり ファイルはロードされているが export/名前空間登録がされていない。

コードのスタブ残り

coordinate-manager.js:109 ❌ CoordinateManager未実装 - Phase1.5で必須
→ 開発中のスタブが残っており、あえて throw するように書かれている可能性。

✅ 対策案
① 一時回避（開発中でも動作確認したい場合）

coordinate-manager.js の 109 行目付近の throw をコメントアウト/try-catch化して「未実装でも起動だけできる」状態にする。

// coordinate-manager.js 内 initializePhase15Managers()
if (!this.CoordinateManager) {
    console.warn("⚠️ CoordinateManager 未実装ですがスキップします");
    return; // throwを止める
}

② 根本解決（正式実装）

coordinate-manager.js に 最低限のダミークラス を実装して、Phase1.5 以降の仕様に沿って存在確認を通す。

例:
// coordinate-manager.js
class CoordinateManager {
  constructor() {
    console.log("🧪 Dummy CoordinateManager initialized");
  }
  // 必須APIだけ仮実装
  toCanvasCoords(x, y) { return { x, y }; }
  toScreenCoords(x, y) { return { x, y }; }
}
window.CoordinateManager = CoordinateManager;

③ 読み込み順の保証

index.html の <script> タグで coordinate-manager.js を bootstrap.js より前 に読み込む。

現在 bootstrap.js で依存関係チェックをしているため、順序不備で undefined になると必ず失敗する。

🎯 提案フロー

短期: throw を外すかダミーを置いてテスト環境で統合確認を進める。

中期: coordinate-manager.js を正式実装。少なくとも toCanvasCoords, toScreenCoords が必須。

長期: NavigationManager・RecordManager・ShortcutManager と完全連携して Phase1.5 仕様を満たす。
