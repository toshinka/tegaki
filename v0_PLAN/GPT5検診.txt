// ============================================================================
// 1) 根本原因 (推定)
//    - 問題は「DOM clientX/Y -> Canvas バッキングピクセル -> world の変換」が不正確だったため。
//    - 具体的には CSS 表示サイズと実際の backing canvas のサイズ (devicePixelRatio による拡大) を混同していた。
//    - さらに、ツール側が直接 clientX/clientY を stage 座標として扱っていたため、ステージの transform（パン/ズーム）が反映されずズレが増幅した。
//    - Chromium の "powerPreference ignored" は警告であり描画座標ズレの直接原因ではない（注記は残す）。
//
// 2) 主要対応 (このコミットで適用済み)
//    - CanvasManager.clientToCanvasPixel() を導入して CSS -> backing pixel を正確に計算。
//    - CanvasManager.clientToWorld() で drawContainer.toLocal(...) を使用し transform を尊重して world 座標を求める。
//    - PenTool は常に canvasManager 経由で world 座標を受け取り、描画は drawContainer 内で行うよう一本化。
//    - ToolManager が managers を注入することでツール側の direct DOM 操作をやめさせた。
//    - RecordManager が無くても動作するようガードを追加（ログと警告出力）
//
// 3) 作業手順（リリース/デプロイ時）
//    - 1. 本ファイル群に置き換え（既存の各ファイルをバックアップ）
//    - 2. index.html が PIXI v8 を読み込んでいることを確認（node_modules 経由 or CDN）
//    - 3. ページを開き、コンソールに CanvasManager 初期化ログと DPI 表示が出ることを確認
//    - 4. マウスで (0,0) 〜 (right,bottom) を描き、コンソール上の PenTool 描画ログで world 座標が妥当であることを検証
//    - 5. RecordManager が存在する場合は stroke が記録されるかを確認
//
// 4) テストケース（必ず実行）
//    - A: 画面左上 (client 0,0) -> world ≒ 0,0（ほぼ一致）
//    - B: 画面中央 (client width/2,height/2) を描画して、ペンの軌跡がキャンバスの中央に描かれることを確認
//    - C: ブラウザズーム（Ctrl+）や devicePixelRatio を変えた環境で CSS ディメンションと backing size に差が出ても動作すること
//    - D: ペンで右/下方向へ長距離に描いてもズレが累積しないこと
//
// 5) 将来の注意点 / 拡張
//    - Container 階層の複雑化 (複数のスケーリング/回転) を考慮し、必要なら canvasManager.clientToWorld に対するユニットテストを作成すること
//    - WebGPU レンダラ利用時も PIXI の renderer.resolution / autoDensity の挙動は同様なので、同変換ロジックで対応可能
//    - v7/v8 の二重管理は厳禁。v7 API を利用するコードは削除/移行してください。
