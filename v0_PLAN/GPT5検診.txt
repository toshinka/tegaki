🔎 原因分析

初期化順序の問題
AppCore.initializeV8Managers 内で

const canvasManager = new CanvasManager(...);
const toolManager = new ToolManager(canvasManager, ...);


みたいに渡す設計のはずが、実際には

canvasManager が未生成

あるいは null/undefined のまま
で ToolManager に渡してしまっている。

依存性の循環

ToolManager が CanvasManager を要求

CanvasManager 側で ToolManager を参照している
→ 生成順がぐちゃぐちゃになって、どちらかが undefined のまま渡る。

ヘッダー対策は効いているが局所的
「このファイル単体の責務」は明示できているけど、全体の初期化シーケンスまでは保証されないので、改修時にAIが片方しか直さずループしやすい。

✅ 改修案
1. 初期化シーケンスを 1か所で集中管理

AppCore を「マネージャー生成オーケストレーター」として一本化し、
CanvasManager → ToolManager の順番を保証する。

// app-core.js
async initializeV8Managers() {
  this.canvasManager = new CanvasManager({ app: this.app });
  this.toolManager = new ToolManager({
    canvasManager: this.canvasManager,
    eventBus: this.eventBus
  });
  // 他のマネージャーも順次
}

2. ToolManager 側は DI前提を徹底
// tool-manager.js
constructor({ canvasManager, eventBus }) {
  if (!canvasManager) {
    throw new Error("ToolManager: 有効なCanvasManagerが必要です");
  }
  this.canvasManager = canvasManager;
  this.eventBus = eventBus;
}


これで責務が明確化され「CanvasManager が必須」だと明示できる。

3. 循環参照を避ける

もし CanvasManager 側で ToolManager を参照しているなら、それは設計的に危険。
→ 双方向参照をやめ、EventBus経由でやりとり させるのが正解。
例:

CanvasManager が「描画完了」を EventBus で publish

ToolManager がそれを subscribe

🧭 AIコーディングの「ループ脱出」方法

今のように

AppCore で CanvasManager / ToolManager の初期化順を直す

ToolManager 側で null チェックを強化する

という2点を両方触る必要があるのに、AIが片方ずつしか直さない → 往復ループになってるわけです。

👉 脱出法は

初期化フローをテキスト化して指示に含める
例:

// 初期化順序は必ず以下を守ること:
// 1. CanvasManager
// 2. ToolManager
// 3. 他のManager群


責務と依存関係をDIコンストラクタで統一

ToolManager には { canvasManager } が必須

逆に CanvasManager から ToolManager を直接参照しない

EventBus を中継点に固定
→ 双方向参照を回避することで「AIが片方だけ直して矛盾する」状況を防げる。

🔎 改修計画（v8全体整合性）
1. 初期化フロー全体の統一

bootstrap.js

PixiJS v8 Application (new PIXI.Application() → await app.init(options)) を生成し、
CanvasManager.initializeV8Application(app) に渡す形に修正。

v7風の PIXI.Application(options) は全面禁止。

Manager登録キー（window.Tegaki.CanvasManagerInstance など）を統一。

app-core.js

各 Manager の生成順序を固定化（CanvasManager → ToolManager → 他Manager群）。

initializeManagers() 内で CanvasManager の v8初期化完了を待ってから ToolManager に渡す 流れに修正。

main.js

v7依存の Tegaki.init() や直接 PIXI.Application を触る処理を禁止。

bootstrap.js に集約された Pixi 初期化結果を利用する。

2. Manager群の依存関係整理

ToolManager

CanvasManager.getV8DrawingContainer() を必須依存にする。

getMainLayer() や v7風 getLayer('main') には移行時注意。

イベント通知は EventBus 経由のみ許可。

ErrorManager / ConfigManager / RecordManager / ShortcutManager

CanvasManager や ToolManager に直接アクセス禁止。

EventBus → Manager 呼び出しに一本化。

NavigationManager / CoordinateManager

Canvas要素取得は CanvasManager.getCanvasElement() を利用。

旧来の pixiApp.view や app.renderer.view は廃止。

3. ツール群の対応

abstract-tool.js

描画先は必ず CanvasManager.getV8DrawingContainer() 経由。

PIXI.Graphics の生成は v8記法に統一（rect+fill）。

pen-tool.js / eraser-tool.js

ToolManager から Container を受け取り、直接 CanvasManager を叩かない。

v7の beginFill → v8の graphics.fill({color}) に修正済みか確認。

4. テスト・デバッグ関連

phase15-integration-test.js

CanvasManager.isV8Ready() を利用して「初期化完了」を確認。

v7の app.renderer.plugins.interaction などは削除。

デバッグ出力は CanvasManager.getV8DebugInfo() に統一。

5. 共通ルール（全ファイルにヘッダー追記必須）

他ファイル呼び出しメソッド一覧

初期化順序フロー（開始→処理→終了）

依存禁止事項（双方向禁止・フォールバック禁止）
