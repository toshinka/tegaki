了解。ログを手掛かりに、描画が始まらない原因とフローの健全性を点検しました。結論からいきます。

結論（最重要）

根本原因は「DOMイベントの座標が undefined のまま PenTool に渡っている」こと。
onPointerDown() は呼ばれているのに、直後に clientX/Y=undefined 警告 → 座標変換失敗で早期中断、という流れです（pen-tool.js:199 → app-core.js:262 の「無効なクライアント座標: x=undefined, y=undefined」→ pen-tool.js:217「座標変換失敗」）。この連鎖はイベントハンドラで原始イベント e を渡していない、または独自イベントに変換する過程で clientX/Y が落ちている時に典型的に起こります。

何が起きているか（ログからの因果）

初期化と Manager 注入は正常に進み、「ペン描画利用可能」まで到達しています（Bootstrap の完了ログと Phase1.5 統合テストの PASS）。

ところが、最初のペン操作で座標が undefined：
PenTool: onPointerDown() 開始 → 無効なクライアント座標: x=undefined, y=undefined → PenTool: 座標変換失敗 - 描画中断。その後 onPointerUp() が「isDrawing: false」のまま終了を繰り返すため線が一切引かれない。

直接の修正ポイント（最短で直す）
1) イベント配線：原始イベントを必ず渡す

addEventListener('pointerdown', handler) の handler が e を受け取り、そのまま tool.onPointerDown(e) に渡るようにしてください。
ありがちなバグは：

() => tool.onPointerDown() のように 引数なしで呼んでしまう

EventBus で 独自イベントに載せ替えた際に clientX/Y を失う（detail にコピーしてない 等）

例（バインド例）：

// 悪い例
canvasEl.addEventListener('pointerdown', () => tool.onPointerDown());

// 良い例（原始イベントを透過）
canvasEl.addEventListener('pointerdown', (e) => tool.onPointerDown(e), { passive: false });
canvasEl.addEventListener('pointermove', (e) => tool.onPointerMove(e), { passive: false });
canvasEl.addEventListener('pointerup',   (e) => tool.onPointerUp(e),   { passive: false });

2) イベントの発火元を Canvas(View) に限定

document や親コンテナに付けると、ターゲット移送時に offsetX/offsetY が不安定になったり、Transform とのズレが出やすいです。

app.view（Pixi の <canvas>）や描画レイヤ直下の DOMに紐づけてください。

3) convertEventToCoords の耐性強化（Pointer/Mouse/Touch 全吸収）

取得順にフォールバック：

function getClientXY(e) {
  if (!e) return null;
  // PointerEvent / MouseEvent
  if (typeof e.clientX === 'number' && typeof e.clientY === 'number') {
    return { x: e.clientX, y: e.clientY };
  }
  // TouchEvent
  const t = e.touches?.[0] || e.changedTouches?.[0];
  if (t && typeof t.clientX === 'number' && typeof t.clientY === 'number') {
    return { x: t.clientX, y: t.clientY };
  }
  // どうしても無い場合は offset 系を最後の手段で
  if (typeof e.offsetX === 'number' && typeof e.offsetY === 'number') {
    // 後段で rect 起点に変換できるなら offset でも可
    return { x: e.offsetX, y: e.offsetY, isOffset: true };
  }
  return null;
}


null の場合は早期 return＋一度だけ警告（スパム防止）。

4) DPR・Rect・コンテナ変形の一貫ロジック

画面→キャンバス座標は 「getBoundingClientRect → 物理ピクセル(DPR適用) → Pixi座標系」の一方向パイプで統一。

例（ざっくり）：

const rect = app.view.getBoundingClientRect();
const dpr = window.devicePixelRatio || 1;
// 画面座標 (client) をキャンバス内の正規化座標へ
const nx = (clientX - rect.left) * dpr;
const ny = (clientY - rect.top)  * dpr;
// Container にスケール/回転があれば CoordinateManager 経由で変換
const { x, y } = coordinateManager.toLocalFromCanvas({ x: nx, y: ny });


Container 変形を使用している前提なので、最後は必ず CoordinateManager の入口を通す。

フロー健全性レビュー（Manager/メソッド配置は概ね良好）

Manager の初期化順・注入ログは健全で、ToolManager→各 Tool への Map 注入も PASS。設計は適正に見えます。

異常は座標イベント変換レイヤのみ。ここが直れば、他の管理層はそのまま活きます。

再発防止のための具体策
A) イベント契約（Contract）をヘッダーに明文化

各ファイル先頭に、次の契約をコメント追加：

Event Contract（全 Tool 共通）

onPointerDown/Move/Up(e: PointerEvent|MouseEvent|TouchEvent) を必ず原始イベントで呼ぶ

clientX/Y を主要ソース。Touch は touches[0] を clientX/Y に正規化してから渡す

passive:false を要求（スクロール干渉防止／preventDefault 可能化）

発火元は app.view 直下の DOM に限定

Coordinate Contract

画面→キャンバス→ローカルは CoordinateManager を正規ルートとする

DPR／Container 変形対応は CoordinateManager に集約し、Tool 側は触らない

DPR Policy

取得は 1 箇所のみ（重複取得・重複適用禁止）

物理座標化は「rect 基準 → *dpr」の一回のみ

Error Policy

clientX/Y 欠落時は描画中断しつつ一度だけ警告

連続発生時は Event 配線の誤りを指摘するガイダンスを出す

B) ユニット&統合テスト

runPhase15Test() に 擬似 PointerEvent を投げる統合テストを追加し、最終点（Graphics に path が積まれるか）まで検証。

ケース：

PointerEvent（通常）

TouchEvent（モバイル）

EventBus 経由（カスタムイベント）：detail.clientX/Y に写経できているか

C) 型とガード

onPointerDown(e?: Event) の e を 必須にし、!e なら即 return（型 or ランタイムで強制）。

convertEventToCoords は 戻り値が不正なら Tool 状態を一切進めない。

D) CSS/DOM 側の地雷排除

<canvas> に pointer-events: auto; touch-action: none; を付与（ブラウザ既定動作の抑止）。

キャンバスの親に transform がある場合は Rect 計算に影響するので、getBoundingClientRect 起点で必ず補正。
