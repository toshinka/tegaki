/* ============================
   原因推定（簡潔）
   - 表示されている警告:
     "The powerPreference option is currently ignored when calling requestAdapter() on Windows. See https://crbug.com/369219127"
     -> Chromium 側の WebGPU 実装に関する警告。単体では致命的ではないが、環境によっては app の WebGPU 初期化コードが失敗（あるいは未対応のAPI呼び出しを行う）し、そのままレンダラ生成を止めてしまう実装になっている。
   - 追加でキャンバスが出ない/座標が拾えない理由:
     1) WebGPU 初期化失敗時にフォールバック処理（WebGL）を適切に行っていない。
     2) Canvas DOM の挿入箇所や append/replace ロジックが期待と異なる（既存 canvas を上書きしてしまう、または CSS で非表示にしている）。
     3) ポインタイベントの登録が canvas ではなく別要素に付与されている（サイドバーアイコンのみ反応している）。
     4) Manager の初期化順序が分散しており、Canvasより先にToolがイベント購読してしまう等のレースがある。
   ============================

/* ============================
   推奨包括改修方針（要点）
   1) 初期化を AppCore に集中化（上記コード）。AppCore は renderer 選定方針を決め、CanvasManager に一度だけ生成命令を渡す。
   2) 双方向依存は全て EventBus に置換。各 Manager/Tool は EventBus を使って通信する。
   3) WebGPU は明示的に無効化（現時点では WebGL を使う）。WebGPU が必要なら opt-in フラグと十分なフォールバック実装を追加する。
   4) Canvas は CanvasManager が sole owner。Canvas の DOM 追加/削除/スタイルは CanvasManager のみが行う。
   5) ポインタ座標取得は CanvasManager が正しい boundingClientRect を使って正規化して EventBus 経由で配信する。
   6) Tool は Graphics レイヤを作り、そのレイヤに描画。Tool は CanvasManager.stage を直接参照せず、app を介してステージに追加する（責務分離）。
   7) Console の冗長出力は AppCore の debug フラグを用いて制御。
   8) v7/v8 両対応コードは即刻排除。v8 前提でリファクタする。
   9) ファイルヘッダー（@provides, @uses, @initflow, @forbids 等）を必ず付与し、手作業での二重管理を避ける（CI チェック推奨）。
   10) 最小統合テストとして debug/phase15-integration-test.js を用意し、Pen/Eraser 切替、描画、座標取得を自動または手動で確認する。
   ============================
