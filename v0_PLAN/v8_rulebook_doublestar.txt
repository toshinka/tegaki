# Futaba Drawing Tool v8 ルールブック - ダブル・スター方式版

## 序文

このルールブックは、Futaba Drawing Tool v8（PixiJS v8ベース）の開発における設計原則とコーディング方針をまとめたものである。
PixiJS v8を基盤として、キャンバス描画、レイヤー管理、ツール操作、イベントハンドリングのフローを効率的かつ**AI可読性高く**実装することを目的とする。

本改訂では、従来の**列車型分割案を削除**し、**ダブル・スター方式**を中心とする構造に全面移行した。

---

## 🌟 ダブル・スター方式とは

### 基本理念
**中心となる2つの主星ファイル（A: 発火・イベントハブ、B: ツール処理）を軸に、肥大化が予想される部分のみを衛星ファイルとして分割する設計手法。**

```
         衛星A1(履歴)    衛星B1(アニメ)
              ↖             ↗
主星A(発火ハブ) ←→ 主星B(ツール処理)
              ↙             ↘
         衛星A2(状態)    衛星B2(書き出し)
```

### 🅰️ 主星A（発火・イベントハブ）
- **役割**: イベント受け取り、状態更新呼び出し、ツール呼び出しのハブ
- **比喩**: 発火する恒星のように、全体のフローを統括する
- **実装**: `futaba_main.html`
- **含む要素**:
  - AppController（統合制御）
  - DrawingEngine（描画エンジン）
  - DomPositionManager（カメラ制御）
  - SystemMonitor（パフォーマンス監視）

### 🅱️ 主星B（ツール処理中心）
- **役割**: ツール・描画処理の中心、レイヤー操作やブラシ・消しゴムなどを担当
- **比喩**: 工場の中心設備として、具体的な加工・処理を実行
- **実装**: `futaba_tools.js`
- **含む要素**:
  - LayerManager（レイヤー管理）
  - DrawingTools（描画ツール）
  - InterfaceManager（UI制御）

### 🛰️ 衛星（A1, A2, B1, B2...）
- **役割**: 状態管理や履歴、特定ツール処理など、主星が肥大化する可能性のある部分のみ切り出す
- **原則**: **主星を追えばAIは衛星を意識せずとも理解可能**
- **分離条件**: 
  - 肥大化が予想される機能（1000行以上）
  - 改修頻度が高い部分
  - 独立性の高い機能（アニメーション、書き出し等）

## 🎯 なぜダブル・スターなのか

### 1. **AIがフローを追いやすい構造**
- 「中心となる2つのファイルに重要処理を集約」することで、状態や描画の追跡が単純化
- Claude等のAIコーディング支援時に、主星ABを読むだけで全体像を把握可能

### 2. **バイ・コーディング対応**
- 人間（非プログラマー）が覚える必要があるファイルは主星ABのみ
- 衛星は必要時にのみ参照すれば良い

### 3. **将来拡張への柔軟性**
- 将来的な衛星ファイル分割も柔軟に行える
- アニメーション機能（SPINE/Live2D）や物理演算も衛星として追加可能
- 改修や機能追加時のリスクが低い

### 4. **現在のコード構造との親和性**
- 既存のfutaba_main.html + futaba_tools.js構造にほぼ一致
- 大幅な構造変更なしに移行可能

## 📋 設計原則

### 1. **主星中心設計**
- 全てのイベント・状態・描画フローは主星ABを中心にルーティング
- AIが一方向のフローで追跡可能
- 衛星は主星からのみアクセス、衛星間の直接通信禁止

### 2. **衛星分割の最小化**
- 肥大化が予想され、改修頻度が高い部分のみ衛星化
- 無駄な衛星は作らず、主星ABだけで可能な限り完結させる
- **衛星分割の判断基準**:
  - ファイルサイズ1000行以上
  - 週1回以上の改修頻度
  - 独立したテスト可能性

### 3. **命名規則の統一**
- Managerクラスや状態変数は役割が直感的に分かる名称を使用
- 重複・架空生成を防ぎ、AIの理解を助ける
- **命名パターン**:
  - 主星A: xxxController, xxxEngine, xxxManager
  - 主星B: xxxTools, xxxInterface, xxxLayer
  - 衛星: A1_xxx, B1_xxx（明確な親子関係）

### 4. **状態管理の明確化**
- レイヤー状態、ツール状態、履歴などは衛星ファイルに分離可能
- 状態更新関数はA1, A2などの衛星から呼び出されるが、フローは常に主星を経由
- **状態フロー**: 衛星→主星A→主星B→描画

### 5. **描画と状態の分離**
- 描画はB（ツール処理）が担当、状態はA（イベントハブ）が管理
- 主星を見ればイベント→状態→描画フローが明確になる

## 🛠️ 実装上のポイント

### 主星ABの責務分担

| 主星A（futaba_main.html） | 主星B（futaba_tools.js） |
|---------------------------|--------------------------|
| イベント受信・配信 | 具体的な描画処理 |
| 状態変更の制御 | ツールロジック |
| システム監視 | UI操作の実装 |
| 座標・カメラ管理 | レイヤー管理 |
| ライブラリ初期化 | インタラクション処理 |

### 衛星分離のタイミング
1. **Phase 1**: 主星ABのみで開発・動作確認
2. **Phase 2**: 肥大化した部分を衛星として分離
3. **Phase 3**: アニメーション等の大型機能を衛星追加

### AIがフローを追いやすい命名と構造の遵守
- **フロー明示**: startXxx(), processXxx(), finishXxx()
- **依存明示**: xxxManager.connect(otherManager)
- **状態明示**: isXxx, hasXxx, canXxx

### 将来的拡張への対応
- 新規ツール: B系衛星として追加
- アニメーション: B1_animation.js
- 物理演算: B2_physics.js
- 書き出し機能: B3_export.js
- 履歴管理: A1_history.js

## 🎮 アニメーション対応設計

### SPINE/Live2D統合戦略
1. **B1_animation.js**として衛星化
2. 主星Bから呼び出される構造
3. 既存レイヤーシステムとの統合
4. 物理演算はB2_physics.jsとして分離

### 実装フロー
```
主星A(イベント) → 主星B(レイヤー管理) → B1(アニメ処理) → B2(物理演算)
     ↑                    ↓
   A1(履歴) ←── A2(状態管理)
```

## 🚫 禁止事項

### 構造面
- 衛星間の直接通信
- 主星を経由しない状態変更
- 循環依存の作成
- 不必要な衛星分離

### 実装面
- フォールバック・フェイルセーフでの誤魔化し
- 過度なconsoleログ
- v7/v8混在コード
- 外部ビルドツール使用

- 既存のfutaba_main.html（主星A）の責務明確化

- クラス名・関数名の統一
- フロー追跡用ログの追加
- コメント整理

### Step 3: 衛星分離準備
- 肥大化部分の特定
- 分離可能性の評価
- 依存関係の整理

### Step 4: 段階的衛星分離
- 優先度順に衛星分離
- 動作確認
- ドキュメント更新
