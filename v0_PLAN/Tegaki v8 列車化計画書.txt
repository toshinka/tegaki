──────────────────────────────
          Tegaki v8 列車化計画書
──────────────────────────────

目的:
1. v7ベースのツールをPixi v8に移行する
2. AIによる改修を効率化するため列車型に構造化する
3. 依存関係・初期化順序を明確化し、肥大化・循環依存を防ぐ
4. 将来的な機能追加をB車両化で独立改修可能にする

──────────────────────────────
1. 列車型構造定義

- 列車 = 複数の車両で構成されるモジュール群
- 車両は前の車両にのみ依存する（単方向依存）
- 列車内部が一定量を超えた場合に新規車両を追加
- 後続車両は前車両のみ参照し、横断参照は禁止
- 循環依存禁止

──────────────────────────────
2. 列車・車両例（初期構想）

A列車（コア初期化＋共通データ）
  ├─ A-1: ヘッダー・ルールブック・ChangeLog
  ├─ A-2: シンボル辞典・共通定数・汎用データ
  └─ A-3: コア初期化処理 (Pixi.Application / Canvas / Loader)

※ A列車内部はモジュール単位でハイブリッド型に分割可能
   - Module1: CanvasManager
   - Module2: EventBus / NavigationManager
   - Module3: CoordinateManager
   - Module4: ToolManager

B車両（機能別独立改修ユニット）
  ├─ B-1: 描画列車（レイヤー管理、スプライト、GPU処理）
  ├─ B-2: アニメ列車（タイムライン、フレーム管理）
  └─ B-3: UI列車（Sidebar / Icons / EventBus連携）

※ B車両は独立改修可能。A列車側はノータッチで済む

──────────────────────────────
3. 列車化ルール

1. 列車内部の車両は前方の車両にのみ依存する
2. 車両間の横断依存は禁止
3. 車両内部はモジュール化し、AIは局所改修できる
4. 列車内部が肥大化したら車両を追加する
5. B車両は必要に応じて独立改修・追加可能

──────────────────────────────
4. V8化方針

- A列車とB車両をそれぞれV8対応化
  - Pixi v8ライブラリ統合
  - Manager / Tool / Canvas 連携修正
- ヘッダー（ルールブック・シンボル辞典）を全車両先頭に集約
- A列車のv8化完了後にB車両を独立改修可能

──────────────────────────────
5. AI改修フロー

1. A-1（ヘッダー）を読み込み、ルールと定数を参照
2. A-2/A-3のモジュール単位で改修・テスト
3. A列車全体が安定したらV8化統合
4. B車両はA列車に依存するだけで独立改修
5. 必要に応じて新B車両を追加して機能拡張

──────────────────────────────
6. 改修効率化ポイント

- ヘッダー化によりAIは依存関係の全体把握が容易
- 列車型構造で局所改修＋単体テストが可能
- 車両ごとのファイルサイズ制御によりAIのコンテキスト負荷を軽減
- A列車は「主要初期化＋共通データ」、B車両は「機能別改修単位」で責務分離

──────────────────────────────
7. 注意点

- 双方向依存、循環参照は禁止
- 車両間の横断依存禁止
- ヘッダー・ルールブックは必ず最新化しAIに参照させる
- v7/v8両対応の二重管理は禁止
- 改修時は単体テストと統合テストを必須

──────────────────────────────
8. 今後の拡張方針

- B車両を機能別列車として増設可能
- 列車が安定したら、A列車内部のモジュールを独立車両化することも検討
- 将来的にはAIがB車両単位で逐次改修し、A列車は触らずに済む運用を目指す
