# ⚠️ 総合AIコーディング規約 v2.2 - 改修版

このドキュメントは、本プロジェクトにおけるコード実装の品質、一貫性、保守性を確保するための統一されたルールです。AIがコードを生成する際は、必ずこの規約を厳守してください。

🎯 基本方針
────────────────────
- モダンブラウザ前提（古い仕様は無視）
- ES6 Modules + Viteベースで開発
- "再発明"の回避と優良ライブラリ活用
- メンテナンス性と構造美重視
- コア処理とUI/UXの責務を明確分離
- 外部AI連携（Claude/Gemini）を前提とした命名と構成

## 1. 📂 ファイル構造と命名規則の基本原則

### 「関心事の分離」に基づく設計思想

現在のプロジェクトでは、**「何をするか」でファイルを分ける「関心事の分離」**という原則に基づいた構造を採用しています。これにより以下の問題を解決します：

#### 従来の問題点
1. **「Manager」という曖昧な命名**: 何（データ）を、どのように（ロジック）管理するのかがファイル名から読み取れない
2. **core と ui の境界が曖昧**: UIとコアロジックの依存関係が複雑に絡み合ってしまう
3. **アプリケーションの「状態」が分散**: 状態が各マネージャーに散らばり、全体像の把握が困難

### 新しい命名規則のポイント

#### Manager の廃止 → 具体的な役割名へ
曖昧な Manager は使わず、以下のような役割が明確な単語を使用：
- **Renderer** (描画するもの)
- **Viewport** (表示領域)
- **Controller** (制御するもの)
- **Store** (状態を保存するもの)
- **Actions** (操作を実行するもの)
- **Service** (外部サービスとの連携)

#### Store (ストア) による状態一元管理
- アプリケーションの状態（どのツールが選ばれているか、レイヤーのリストなど）は、すべて Store という名前のファイルに集約
- データの流れを「UI → Actions → Store → UIが状態を購読して再描画」という一方向にする
- これにより見通しが良く、予測可能な状態管理を実現

#### features ディレクトリによる機能分離
- 「レイヤー機能」「ツール機能」のように、アプリケーションの主要な機能ごとにディレクトリを分割
- 関連するファイル（Store, Actions, 実装クラス）が近くにまとまるため、開発効率が向上

### この設計による具体的メリット

1. **予測可能性の向上**: 「レイヤーのリストを操作したいなら features/layers/ を見ればいい」というように、ファイルやディレクトリの役割が明確になり、コードを探す時間が激減

2. **テストの容易化**: UIとロジック、状態が綺麗に分離されるため、それぞれの機能を単体でテストすることが容易

3. **AIとの連携強化**: 「ToolStore.js にブラシサイズの状態を追加して」のように、AIに対して非常に具体的で誤解のない指示を出せるようになり、生成されるコードの精度が格段に向上

### 基本アーキテクチャ: Store / Actions / UI

* **Store** (LayerStore.js, ToolStore.js): アプリケーションの状態（データ）を一元管理します。状態を直接変更できるのは対応するActionsのみです。
* **Actions** (LayerActions.js, ToolActions.js): ユーザー操作などの「ユースケース」を定義します。UIからのイベントを受け取り、Storeの状態更新を依頼します。
* **UI** (LayerPanel.js, Toolbar.js): Storeの状態を購読して画面を描画し、ユーザーイベントが発生したら対応するActionsを呼び出します。

## 2. 💎 設計原則 (SOLID)

すべてのコードは、保守性と拡張性を高めるためにSOLID原則に準拠する必要があります。

### S: 単一責任の原則 (SRP)
一つのクラスは、一つの、そして一つの理由によってのみ変更されるべきである。

**❌ 悪い例:** UIコンポーネントであるLayerPanel.jsが、UIの表示、レイヤーの状態管理、キャンバスへの直接描画ロジックまで担当してしまう。

**✅ 良い例:**
- LayerStore.js: レイヤーの状態管理に専念する。
- DrawingEngine.js: キャンバスへの描画処理に専念する。
- LayerPanel.js: LayerStoreを購読してUIを表示し、ユーザー操作をLayerActionsに通知することに専念する。

### O: 開放/閉鎖の原則 (OCP)
ソフトウェアの実体は、拡張に対して開かれており、修正に対して閉ざされていなければならない。

**❌ 悪い例:** 新しいツール（例: BucketTool.js）を追加するたびに、入力処理を担当するCanvasInteraction.js内のif-elseやswitch文を修正する必要がある。

**✅ 良い例:** CanvasInteraction.jsは現在アクティブなIToolインターフェース（PenToolやEraserToolが実装）を受け取り、そのtool.onPointerDown()のようなメソッドを呼び出すだけにする。新しいツールはIToolを実装したクラスを追加するだけで済み、CanvasInteraction.jsの修正は不要になる。

### L: リスコフの置換原則 (LSP)
派生型は、その基本型と置換可能でなければならない。

**❌ 悪い例:** IToolインターフェースを実装したPenTool.jsの描画メソッドはLayerオブジェクトを返すのに、EraserTool.jsの同名メソッドはvoidを返す。これでは呼び出し元で両者を同じように扱うことができない。

**✅ 良い例:** IToolインターフェースの各メソッドの戻り値の型（例: boolean）を明確に定め、すべての実装クラスがその規約を遵守する。

### I: インターフェース分離の原則 (ISP)
クライアントに、不要なインターフェースの実装を強制すべきではない。

**❌ 悪い例:** あらゆる機能を持つIEngineのような巨大インターフェースを作り、WebGLRendererに不要なundo()やredo()の実装まで強制する。

**✅ 良い例:** 機能をIRenderer, IViewport, IHistoryのように目的別に分離し、クラスは必要なインターフェースだけを実装する。

### D: 依存性逆転の原則 (DIP)
上位モジュールは下位モジュールに依存すべきではない。両者とも抽象に依存すべきである。

**❌ 悪い例:** LayerStore.jsが、具体的な実装であるStorageService.jsクラスを内部で直接newして利用する。

**✅ 良い例:** アプリケーションの起点であるAppController.jsがStorageServiceのインスタンスを生成し、それをLayerStoreのコンストラクタを通じて注入（DI）する。LayerStoreは具体的なクラスではなく、抽象的なストレージの口約束（インターフェース）にのみ依存する。

## 3. 🔥 重要度順：AI実装における懸念リスト

### 🔥 **最重要：責務とインターフェース設計**

#### ❌ 懸念A：関数名や責務の被り・曖昧性 (SRP違反)
**AIの失敗パターン:**
- DrawingEngine.jsとTransformUtils.jsの両方に、微妙に異なる座標変換ロジックが実装されてしまう
- `init()` のようなあいまいな関数名で何でも初期化しようとする
- 同じような役割の関数を複数箇所で作りがち（drawLine, renderLine, strokeLine など）

**✅ 対策:**
- 責務は単一のファイルに集約する。例えば、座標変換はすべてCoordinateSystem.jsに任せる
- 関数名は具体的で明確にする（`init()` → `initializeCanvas()`, `initializeUI()`）
- 同じ処理は1つのファイルに統一する

#### ❌ 懸念B：責務の分散・混在 (SRP違反)
**AIの失敗パターン:**
- 本来ToolStore.jsが管理すべき「現在のブラシサイズ」という状態を、UIコンポーネントであるPenSettingsPanel.jsがプロパティとして保持してしまう
- canvas-manager.js に描画とUIと設定が混ざる

**✅ 対策:**
- 状態は必ずStoreに一元化する。UIはStoreを参照するだけで、自身の状態として保持しない
- 1つのファイルは1つの責務のみを持つ

#### ❌ 懸念C：中間層の肥大化と無名化 (ISP違反)
**AIの失敗パターン:**
- AppController.jsが、単に各モジュール間の処理を受け渡すだけの「伝言ゲーム」の役割に終始し、何十ものパススルー関数を持つ巨大なクラスになる
- bridge.js や manager.js のような中間層に「とりあえず全部詰める」

**✅ 対策:**
- データフローはUI → Actions → Store → UIのサイクルに従う
- AppControllerは起動時の依存性注入と全体の統括に専念する
- 中間層は必要最小限に留める

### 🔧 **高重要：実装の一貫性**

#### ❌ 懸念D：毎回「微妙に違うやり方」での実装 (OCP違反)
**AIの失敗パターン:**
- ある場所ではPromise + .then()を使い、別の場所ではasync/awaitを使うなど、非同期処理の書き方が一貫しない
- 同じ処理なのに微妙に違う構文やフローで書く（例：配列のループ、イベント登録）

**✅ 対策:**
- プロジェクト内でコーディングスタイルを統一する
- 新機能追加時は既存の実装パターンを踏襲する
- 拡張に開放、修正に閉鎖の原則を守る

#### ❌ 懸念E：置換可能性の欠如 (LSP違反)
**AIの失敗パターン:**
- 同じインターフェースを実装しているはずなのに、戻り値の型が異なる
- エラーハンドリングの仕方が実装によって異なる

**✅ 対策:**
- インターフェースの契約を明確に定義する
- 同じインターフェースを実装するクラスは、完全に置換可能にする
- 戻り値の型とエラーハンドリングを統一する

### 🧹 **中重要：保守性と品質**

#### ❌ 懸念F：動作確認・デバッグ用コードの消し忘れ
**AIの失敗パターン:**
- console.log()やwindow.store = this.layerStoreのようなデバッグ用コードが本番に残ってしまう
- debugDrawLayer() みたいな関数を本番コードに残す

**✅ 対策:**
- 実装完了時に、デバッグに関連するコードはすべて削除する
- 必要に応じて適切なログレベルシステムを使用する

#### ❌ 懸念G：データ永続化処理の前提誤認
**AIの失敗パターン:**
- StorageService.jsの実装でlocalStorageを使う際、容量制限（約5MB）や、ブラウザのデータ消去で作業内容がすべて失われるリスクを考慮しない
- 保存フォーマットやキー競合、消失条件をあまり気にしない

**✅ 対策:**
- データの重要性を理解し、堅牢なIndexedDBをベースに実装する
- エラーハンドリングやデータ移行についても考慮する
- データ競合や消失のリスクを最小化する設計にする

#### ❌ 懸念H：処理順序や非同期性への見落とし
**AIの失敗パターン:**
- 外部から画像を読み込んでレイヤーを作成する際、画像の読み込み完了（onload）を待たずにWebGLRenderer.jsでテクスチャを更新しようとし、描画に失敗する
- async/await を適当に書く（fetch→await→render の順序がバラバラ）

**✅ 対策:**
- Promiseやasync/awaitを正しく使用し、非同期処理の完了を保証してから次の処理へ進む
- 状態の整合性を保つために、処理順序を明確にする

### 🧪 **低重要：コード品質**

#### ❌ 懸念I：冗長コード (DRY違反)
**AIの失敗パターン:**
- 同じ処理を複数箇所にコピペ
- 微妙に違う実装で同じ機能を再実装

**✅ 対策:**
- utils/配下に共通処理をまとめる
- 同じ処理は1つの関数に統一する

#### ❌ 懸念J：過度な複雑化 (KISS違反)
**AIの失敗パターン:**
- 必要以上に複雑な構造作成
- 「将来に備えて」過度な抽象化

**✅ 対策:**
- シンプルで理解しやすい実装を心がける
- 必要になったときに拡張する（YAGNI原則）

## 4. 🛡️ セキュリティ・パフォーマンス指針

### セキュリティ考慮事項
- ユーザー入力の検証とサニタイズ
- XSS対策（HTMLエスケープ）
- ファイル読み込み時の検証
- 適切なエラーハンドリング

### パフォーマンス考慮事項
- WebGLリソースの適切な管理
- メモリリークの防止
- 不要な再描画の抑制
- イベントリスナーの適切な削除

## 5. 🚀 AIへの指示テンプレートとチェックリスト

### 指示テンプレート

```
「[機能名]を実装してください。」

【実装方針】
1. **UIの変更**: `ui/components/[コンポーネント名].js` に[UI変更内容]を追加してください。
2. **アクションの追加**: `features/[機能名]/[機能名]Actions.js` に[アクション名]メソッドを追加してください。
3. **ロジックの実装**: `features/[機能名]/[機能名]Store.js` に[ロジック内容]を実装してください。
4. **履歴の記録**: 操作は`HistoryStore`に記録し、Undo/Redoできるようにしてください。

【SOLID原則遵守】
- 単一責任: [このクラスの責務を1文で明記]
- 開放閉鎖: [拡張方法を説明]
- 置換可能: [戻り値の型を明記]
- インターフェース分離: [必要な機能のみ実装]
- 依存性逆転: [抽象に依存させる]

【遵守事項】
- 上記のファイル構造とSOLID原則、懸念リストA-Jの内容を厳守してください。
- `console.log`などのデバッグコードは最終的に削除してください。
- 非同期処理は必ずPromise/async-awaitで適切に処理してください。
- エラーハンドリングを必ず実装してください。
```

### 完了確認チェックリスト

実装完了時に、以下の項目を必ずセルフチェックしてください。

#### SOLID原則チェック
- [ ] **単一責任**: 1つのクラス/ファイルが1つの責務だけを担っているか？
- [ ] **開放/閉鎖**: 新機能追加時に、既存コードの「修正」ではなく「追加」で対応できたか？
- [ ] **置換可能性**: 同じインターフェースを持つクラスは、問題なく入れ替え可能か？
- [ ] **インターフェース分離**: クラスが不要なメソッドの実装を強制されていないか？
- [ ] **依存性逆転**: 具体的な実装ではなく、抽象（インターフェース）に依存しているか？

#### 品質チェック
- [ ] **DRY原則**: 同じコードを複数箇所に書いていないか？（ユーティリティ関数などにまとめたか？）
- [ ] **デバッグコード**: console.logなどを消し忘れていないか？
- [ ] **アーキテクチャ**: UI → Actions → Store の一貫したデータフローを守っているか？
- [ ] **非同期処理**: Promise/async-awaitで適切に処理されているか？
- [ ] **エラーハンドリング**: 適切なエラーハンドリングが実装されているか？

#### セキュリティ・パフォーマンスチェック
- [ ] **入力検証**: ユーザー入力の検証とサニタイズが適切に行われているか？
- [ ] **リソース管理**: WebGLリソースやイベントリスナーが適切に管理されているか？
- [ ] **メモリリーク**: 不要なオブジェクトが残っていないか？

## 6. 📋 実装優先順位

### フェーズ1：基盤
1. engine/ - 描画エンジン
2. features/layers/ - レイヤー機能
3. features/tools/ - 基本ツール

### フェーズ2：機能拡張
1. features/history/ - 履歴機能
2. ui/components/ - UI実装
3. services/ - データ永続化

### フェーズ3：最適化
1. events/ - イベント処理
2. utils/ - 共通処理
3. パフォーマンス改善

この規約に従うことで、保守性が高く、拡張しやすい、品質の高いコードを一貫して生成できます。
