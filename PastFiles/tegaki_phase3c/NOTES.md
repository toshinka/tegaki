## バケツツール：囲い塗り

### 要望

線で囲った領域の中をバケツでクリックしたとき、内側だけが塗られるようにしたい。

例：

```text
○ を描く
内側でバケツを使う
● になる
```

### 後続実装時に決めること

- 参照対象を現在レイヤーのみにするか、表示中レイヤー合成にするか。
- 半透明線やアンチエイリアスの薄い縁を境界として扱うしきい値。
- 小さな隙間を閉じる gap close の有無と強さ。
- 塗りを同一レイヤーへ入れるか、線画の下の別レイヤーへ入れるか。

## Vキー変形パネルの回転スライダー改善案 (Phase 1j 以降)

- **現状の問題**: 
  - -360°〜360° の固定範囲であり、回転を繰り返すと端に到達して止まってしまう。
  - 数値入力は可能だが、スライダーでの連続的な回転操作がしにくい。
- **改善案**:
  - **無限ループスライダー**: 1周（360°）を超えても累積角度としてカウントし、スライダーのつまみ位置は周期的に戻るが内部数値は増え続ける仕組み。
  - **累積角度の保持**: 720° や -1080° などの値を許容し、アニメーション等への応用を容易にする。
  - **リセットボタン**: クリック一つで 0° に戻せるボタンをスライダー横に配置。

## Phase 2 レイヤーパネル再配置案（PC液タブ優先）

### 参照画像

- `資料_説明用画像/ToonSquid_スクショ.png`
- `資料_説明用画像/新レイヤーパネルイメージ.png`
- `資料_説明用画像/透明度スライダーポップアップ案.png`

### 方針

- このツールは PC / Windows / 液タブ優先。スマホ的な上下ツールバーより、上下方向のキャンバス面積を残す設計を優先する。
- レイヤーパネル内の編集アイコンは右サイドバーへ寄せ、レイヤー行は薄く、縦に多く並べられる形にする。
- 右サイドバーにはレイヤー追加、フォルダ追加、複製、下結合、削除に加え、後続でクイックパネル、Undo、Redo も置ける余地を作る。
- レイヤー行は ToonSquid 風に、左から `▼/▶`、サムネイル、フォルダ/レイヤー種別アイコン、名前、クリッピング状態、目アイコンを基本形にする。
- サムネイルは右ではなく左寄せが本命。右サイドバー側に目アイコンやクリップ状態を寄せることで、選択・表示切替・操作バーの移動距離を短くする。

### 透明度ポップアップ案

- レイヤー行の `100%` 表示、または右サイドバーの透明度ボタンから開く。
- 上段に透明度プリセット（例: 0 / 25 / 50 / 75 / 100 相当の丸）。
- 中段にスライダー。
- 下段に合成モード（通常 / 乗算 / 加算 / オーバーレイ等）とクリッピング切替を置く。
- クリッピング ON の場合、レイヤー行のクリッピング状態欄にアイコンを表示する。
- クリッピング本体は、下レイヤーで描画済みの不透明部分だけに描ける簡易マスキングとして扱う。
- 逆クリッピングも候補。ただし通常クリッピング本体が安定してから、UIスペースと実装コストを見て判断する。

### 実装順の候補

1. 右サイドバー化と操作アイコン縦配置。既存の追加・複製・下結合・削除 API を再利用するだけなので低リスク。
2. レイヤー行の DOM 再配置。サムネイル左、名前中央、クリップ/目アイコン右の薄型行へ変更する。保存形式には触れない。
3. 透明度ポップアップの UI だけ先行実装。既存の `setLayerOpacity()` に接続し、合成モードとクリッピングは表示だけまたは無効状態で置く。
4. 合成モードを `layerData.blendMode` と Pixi の文字列 `blendMode` に接続する。
5. クリッピング本体を実装する。保存/復元/履歴/サムネイルへの影響があるため Codex 担当。

### 注意点

- クリッピングは見た目だけでなく描画・保存・復元・サムネイル生成へ影響するため、Gemini の小修正範囲にしない。
- 合成モードは `data-models.js` と `project-manager.js` に既存フィールドがあるが、実表示への接続確認が必要。
- フォルダ行と通常レイヤー行で同じ横並びルールに寄せるが、背景レイヤーは特殊扱いを維持する。
- ToonSquid 風の見た目は参考にするが、トップバーを増やさず、キャンバス上下方向の面積を削らない。

## レイヤー/アルバムのドラッグ感触：独自D&D候補

### 現状

- レイヤーパネルのフォルダ収納は、SortableJS に口開き判定と青ハイライトを足して概念実証済み。
- ただし SortableJS は一次元リスト並び替え向けで、フォルダ収納、階層移動、閉じたフォルダ、同一フォルダ内並び替え、外へ出す操作が混ざると不安定になりやすい。
- アルバム並び替えでも、Windows/OS 的な「半透明の対象が滑らかに追従し、挿入先が自然に空く」感触には SortableJS 調整だけでは届きにくかった。
- `PastFiles/OldFiles/v8.13_anime14` 周辺に、レイヤーパネルのスムーズな移動ができていた時期の実装がある可能性がある。独自D&Dやレイヤー移動感触を再設計するフェーズに入ったら、まずこの過去実装を参照する。
- Claude 提案書 `DnD_設計提案書_Claude_to_Codex.md` も同じ方向。HTML5 Drag and Drop はペン/タッチ非対応のため採用不可、`@dnd-kit` はReact前提寄りで導入コストあり、長期的には Pointer Events 自前実装が本命という評価。

### 方針

- Phase 2c 時点では「恐る恐る使えば成立する暫定実装」として凍結する。
- 今後どうしても操作感を詰める場合は、SortableJS への継ぎ足しではなく、レイヤーツリー/アルバム共通の薄い独自D&Dを検討する。
- 独自D&Dでは、ドラッグ中に実DOMや `parentId` を動かさず、半透明プレビュー、挿入位置マーカー、フォルダ吸い込み候補だけを表示し、ドロップ時に一回だけデータを更新する。

### 後続で決めること

- レイヤーパネルとアルバムで共通化するか、先にレイヤーパネル専用にするか。
- フォルダ吸い込み判定を、中央ヒット、一定時間ホバー、修飾キーのどれに寄せるか。
- タブレット/ペン入力でのドラッグ開始距離、キャンセル操作、スクロールとの競合。
- 履歴へ積む粒度と、保存/復元時の親子順序の正規化。

## ペン実装：Claude提案書の現行照合メモ

参照: `ペン実装_現状評価と修正提案書_Claude_to_Codex.md`

### 照合結果

- 提案書の大枠「PointerEvent -> perfect-freehand -> Pixi Graphics -> RenderTexture」は、現行方針と矛盾しない。
- ただし提案書は Phase 1c 付近の古い問題認識を含む。現行コードでは以下がすでに対応済み。
  - `drawing-engine.js` は `info.button === 2 && info.pointerType !== 'pen'` で右クリックを除外しており、ペンの `button:2` は通す。
  - `pointer-handler.js` も `e.button === 2 && e.pointerType !== 'pen'` で同様に分岐している。
  - `stroke-renderer.js` は `smoothing: 0.08`、`streamline: 0.0` で補正をかなり弱めている。
  - `stroke-renderer.js` には異常輪郭検知とフォールバック描画がある。
  - 筆圧は `calculateWidth()` / getStroke 入力で 0.02 以上にクランプされている。
  - 消しゴムは `brush-core.js` のリアルタイムセグメント焼き込み経路で `renderEraserSegment()` を使い、RenderTexture へ直接反映する構造になっている。

### 方針

- Phase 2d の透明度/合成モード/クリッピング下準備は変更しない。
- ペン提案書は「即修正指示」ではなく、後続の描画品質監査チェックリストとして扱う。
- 必要なら Phase 3 以降、描画補助やブラシ品質を扱うタイミングで、実機症状ベースで再確認する。

### 後続で再確認する項目

- 消しゴムリアルタイム消去が、現在の実機操作でも安定しているか。
- `settings-manager.js` / 設定UIの smoothing 値が、実際に `stroke-renderer.js` のオプションへ接続されているか。
- WebGL2 系旧経路に残る `smoothing: 0.4` / `streamline: 0.3` は凍結コードとして完全に無害か。
- PixiJS v8.17.0 の `reparentChild()` は、将来の独自D&Dやフォルダ移動で使う価値があるか。ただし今すぐ差し替えない。

## アルバム管理拡張案

- **ドラッグ並べ替え**: アルバムカードの順番をドラッグで変更し、保存順を維持する。
- **複数選択**: まずは `Ctrl+クリック` で選択トグル。通常クリック復元の既存操作を壊さない。
- **一括操作**: 一括削除、一括エクスポート、移動は選択状態の土台が安定してから扱う。
- **選択操作の受け入れ方針**: `Ctrl/Cmd+クリック` は個別トグル、`Shift+クリック` は前回選択から範囲選択。選択中カードのゴミ箱は複数削除に使えると自然。
- **まとめて移動の扱い**: ランダムな複数選択カードをドラッグ移動すると、挿入位置、相対順序、欠けた位置の詰め方が複雑化する。現時点では複数選択は一括削除/一括エクスポート用に限定し、並べ替えは単体カード中心にする。
- **上部ツールバー案**: `現在の状態を保存` の横に、選択モード、一括削除、アルバムHTML保存、アルバム読み込みを置く。カード内の個別ダウンロード/削除は撤去候補。
- **ドラッグ感触**: Windowsのファイル移動に近い、ドラッグ対象が半透明で追従し、挿入先が自然に空く動きを目標にする。レイヤーパネルにも同じ考え方を適用したい。
- **ドラッグ感触の限界**: SortableJS 調整だけでは、Windowsのファイル移動のような「薄い絵がホバリングして等速追従する」感触には届きにくい。アルバム並べ替えは現状で完了扱いにし、どうしても必要なら後続UI刷新で独自ドラッグUIとして扱う。
- **HTML保存/読込**: thumbnail だけでは復元できないため、`projectData`、canvas情報、レイヤー情報、順番、作成日時、アプリバージョンを含める必要がある。Phase 1m ではサムネイル表示用HTMLに `tegaki-album-data` JSONを埋め込み、旧JSON読み込み互換を残す方針。
- **保存正本**: 現行 `AlbumPopup` は localStorage、`VirtualAlbum` は IndexedDB の別系統。どちらを正本にするかは設計判断が必要。

## ログ整理・保守方針

- 恒久ログ: 起動時の重要設定、重大エラー、復旧不能な状態。

## アルバム管理基盤拡張 (Phase 1l 設計メモ)

### HTML保存/読込の設計

- **コンセプト**: プロジェクト全体（全スナップショット + メタデータ）を一つのポータブルな HTML ファイルとして保存する。
- **HTMLに含めるべき情報**:
  - `projectData`: 全レイヤーの構造、画像、設定を保持する JSON。
  - `thumbnails`: ギャラリー表示用の軽量画像データ。
  - アプリケーション本体（ビューワーモード）を含めることで、ファイル単体での閲覧を可能にする案もあり。
- **なぜ両方が必要か**:
  - プレビュー（サムネイル）のみでは再編集が不可能。
  - プロジェクトデータのみでは、エクスプローラー等での一覧性が損なわれる。
- **ファイルサイズ問題**:
  - レイヤー画像が多いと数十MB〜数百MBに達する可能性がある。
  - 必要に応じて画像の圧縮（JPEG/WEBP）や、履歴の除外を検討する。
- **保存データの役割分担**:
  - **localStorage**: 編集中の自動保存、数件の最新スナップショット（即時アクセス用）。
  - **IndexedDB**: 数十〜数百件のローカルアーカイブ。大容量データの永続化。
  - **HTMLファイル**: 外部へのバックアップ、ポータブルな受け渡し用。
- **互換性**:
  - 旧形式（画像のみのスナップショット）も破損させずに「閲覧専用」として維持する。
