ブラウザ向けお絵かきツール『**TEGAKI**』 の将来的なアニメーション機能（物理演算シミュレーション・AI連携含む）の実装に向け、ご提示いただいた **Procreate Dreams** と **ToonSquid** の2大iPadアニメーションアプリを徹底分析・比較したUI選定提案書を作成しました。

『TEGAKI』が目指す「デスクトップ液タブ主対象＋スマホ・タブレット最小限対応」 というマルチデバイス環境と、「コーディング初心者な赤ちゃん（AI主導開発）」 という開発背景を踏まえ、実装の現実性とユーザー体験（UX）のバランスをとった最適なタイムラインUIを提案します。

# ブラウザお絵かきツール『TEGAKI』アニメーションタイムラインUI 選定・実装提案書

## 1. 比較対象製品のUI/UX特徴分析

### ① Procreate Dreams

タッチ操作に特化した現代的な「トラック型」ハイブリッド・タイムライン。

* **タイムライン構造**:
* 動画編集ソフト（Premiere/LumaFusion等）に近い、アセットを「トラック」に並べる構造。1つのトラック内に複数の描画（セル）を連続して配置可能。
* キーフレームはアセットの下部に隠されており、タップして展開する。


* **最大の強み**:
* **Perform（実演記録）機能**: 再生中にアセットを指やペンで直接ドラッグして動かすと、その軌跡がリアルタイムで自動キーフレーム化される。
* 直感的なジェスチャー操作（3本指で時間軸・縦幅の伸縮など）。


* **弱み・欠点**:
* 従来のセルアニメ（パラパラマンガ）的な「1フレーム＝1枚の紙」を厳密に管理する用途では、トラック構造が逆に扱いづらい（Flipbookモードという簡易切り替えはあるが往復が必要）。
* 高機能なベクトル変形や複雑なキーフレームシーケンスを緻密に数値管理する画面が不足している。



### ② ToonSquid

「ベクターアニメーション」と「伝統的セルアニメ」の完全な融合を目指した、高機能レイヤー・タイムライン。

* **タイムライン構造**:
* 一般的なお絵かきソフトの「レイヤーパネル」がそのまま右側に伸びてタイムラインになった形（After EffectsやAnimate/Flashに近い正統派）。
* 1レイヤーに対して「キーフレーム（変形・不透明度等）」と「セル（描画アセットの切り替え）」が同一レイヤー上で美しく共存する。


* **最大の強み**:
* **圧倒的な機能網羅性**: ベクターパスの変形、ボーン（リギング）、パスに沿ったアニメーション、カメラワーク、シンボル（ネスト構造）など、2Dアニメーションに必要な機能が完璧に揃っている。
* キーフレームカーブ（イージンググラフ）の編集が容易で、タイミングの微調整がプロレベルで行える。


* **弱み・欠点**:
* デスクトップアプリに近い緻密さがあるため、UIの情報密度が非常に高い。画面の小さいデバイスではボタンや階層の操作が煩雑になりがち。

## 2. 『TEGAKI』視点での機能・デザイン評点

『TEGAKI』の開発方針（WebGL2によるRenderTexture描画、将来的な物理演算、AI連携、およびAIがコードを全実装するシンプルかつ堅牢な設計） に基づき、両製品のアプローチを評価します。

| 評価項目 | Procreate Dreams | ToonSquid | 『TEGAKI』への適合度・寸評 |
| --- | --- | --- | --- |
| **画面の潔さ・UIの美しさ**<br>

<br>（はっちゃんマインドの継承） | **5.0** / 5.0 | 4.0 / 5.0 | **Dreamsの勝ち**: 描画中はタイムラインを極限まで隠せるクリーンさが秀逸。 |
| **伝統的セルアニメの描きやすさ** | 3.5 / 5.0 | **5.0** / 5.0 | **ToonSquidの勝ち**: 1枚ずつパラパラ描く作業はレイヤー直結型が圧倒的に直感的。 |
| **物理演算・変形制御との親和性**<br>

<br>（Live2D / Spine的表現） | 4.0 / 5.0 | **5.0** / 5.0 | **ToonSquidの勝ち**: ボーンやパス変形、イージングの可視化が物理演算と相性抜群。 |

## 3. 『TEGAKI』向けハイブリッドUI提案

双方の「良いとこ取り」をしつつ、現在の『TEGAKI』のシンプルなアーキテクチャ（`layer-system.js` や `RenderTexture` によるスナップショット履歴） の上に無理なく載せられる「レイヤー直結型・実演記録対応タイムライン」を提案します。

### UIデザイン案（レイアウト構成）

```text
+-----------------------------------------------------------------------+
|  [メニュー] [ツールバー: ペン/バケツ]                      [アルバム]  |
+-----------------------------------------------------------------------+
|                                                                       |
|                         【 メインキャンバス 】                        |
|                     (WebGL2 / RenderTexture)                          |
|                                                                       |
+-----------------------------------------------------------------------+
| ［▶ 再生］［● 実演(Perform)］  [現在のフレーム: 12/24]   [オニオンスキン] |
+-----------------------------------------------------------------------+
| タイムライン領域（トグル開閉式）                                        |
| [レイヤー3] ▣ [ セル1 ][ セル2 ]...                                    |
|   └─ ◆位置   [──キーフレーム軌跡──]                                   |
| [レイヤー2] ▣ [─────── セル1 ───────]                                 |
|   └─ ◆物理   [重力: ON / 揺れものパラメータ]                           |
| [レイヤー1] ▣ [─────── 背景画像 ──────]                                 |
+-----------------------------------------------------------------------+

```

### コア採用機能と設計方針

#### ① ベース構造は「ToonSquid型（レイヤー直結）」を採用

* **理由**: 現在の `TEGAKI` はすでに安定したレイヤーシステム（サムネイル表示、不透明度、複製・結合など）を構築しています。新しく動画用の「トラック」概念を新設すると、データモデル（`data-models.js`）やEventBusが二重実装により肥大化・破綻するリスク（最優先ルールに抵触）が高まります。
* **実装**: 既存の `Layer` オブジェクトに、フレームごとの画像データを保持する「タイムシート配列」と「変形（Transform）キーフレーム配列」をオプショナルで持たせる拡張を行います。

#### ② 演出・操作感は「Procreate Dreams型」をリスペクト

* **実演記録（Perform）モードの限定実装**:
『TEGAKI』の `Vキー変形モード`（`layer-transform.js`）を応用します。アニメーション再生中に、ユーザーがキャンバス上のレイヤーをドラッグして動かした位置（座標・回転）を、そのままフレームごとのキーフレームとして保存する機能を実装します。
* これを行うことで、Live2Dのような複雑なキーフレーム打ちをしなくても、手動のドラッグ（実演）だけで「髪の毛の揺れ」や「小物の移動」のベースが作れるようになります。


* **ジェスチャーライクな操作**:
HTMLのRuler（時間軸）部分をマウスや指で左右ドラッグすることで、Dreamsのような「なめらかなスクラブ（再生ヘッドの移動）」を実現します。

#### ③ 物理演算アニメへの布石（揺れもの・自動インビトウィーン）

* **物理キーフレーム**:
レイヤーごとに「物理演算ノード（例えば振り子、バネ、重力）」の適用トグルを配置します。
ユーザーが「実演記録（Perform）」でキャラクターの体を大きく動かした際、その親の動きに追従して、髪の毛レイヤーやスカートレイヤーに「遅れ（慣性）」や「戻り（弾性）」をWebGL（GPU）側、あるいはシンプルな頂点変形シェーダーで自動計算して付与する仕組み（Spineの物理演算に近いアプローチ）のUIを用意します。

## 4. 開発ロードマップ（段階的実装プラン）

AI（Gemini/Claude/Codex）が1ステップずつ確実に、コードの破綻なく実装を進めるためのロードマップです。

### 短期フェーズ：Phase 2a（アニメーション基盤・セル拡張）

* **目的**: 現行の単一キャンバスから、複数フレーム（パラパラマンガ）が動く最小構成を作ること。
* **UI要件**: 画面下部にシンプルな「フレームバー」を表示。
* **データ構造**: `Layer.frames = [RenderTexture1, RenderTexture2, ...]` のように、フレームごとの画像テクスチャを管理。
* **受け入れ条件**: 再生ボタンでパラパラマンガがループ再生され、オニオンスキン（前後のフレームが薄く透ける機能）が表示されること。

### 中期フェーズ：Phase 2b（タイムラインUIとキーフレーム・変形）

* **目的**: 描画（セル）の長さ調整と、レイヤー全体の移動キーフレームの実装。
* **UI要件**: ToonSquidを模した、レイヤー名と時間軸が横に並ぶタイムラインUI。
* **機能**: `Vキー変形` のマトリクスをフレームごとに保持し、フレーム間で自動的に直線補間（インビトウィーン）を行う。

### 長期フェーズ：Phase 2c（実演記録と簡易物理演算・AI連携）

* **目的**: 物理演算トグルと、AI動画（静止画→動画変換）用メタデータの書き出し。
* **UI要件**: 「● Perform」ボタンの追加、レイヤープロパティ内への「物理パラメータ（揺れ・重力）」スライダーの配置。
* **機能**: 物理演算による自動揺れもののシミュレーションと、プロジェクト全体のレイヤー別シーケンス（JSON + 透過連番PNG）の書き出し（FramePack eichi等のAIツールにそのまま投入できる形式）。

## 5. AI作業向け規律（本フェーズにおける注意点）

今後この設計をベースにAIへ実装指示を出す際は、`TEGAKI.md` の精神に基づき、以下の制約を厳守させてください。

1. **Canvas2Dの完全禁止**: タイムラインのプレビューやオニオンスキン描画においても、Canvas2Dコンテキスト（`getContext('2d')`）は絶対に使用せず、PixiJSの `RenderTexture` もしくはWebGL2のシェーダーを徹底して使い回すこと。
2. **既存EventBus契約の尊重**: フレーム切り替えイベント（例: `TIMELINE_FRAME_CHANGED`）を新設する際は、既存のレイヤー選択やサムネイル更新のイベントと循環依存を起こさないよう、単方向のデータフロー（State → UI）を維持すること。
3. **UIの肥大化防止**: 最初からDreamsのような美麗なカスタムドラッグUIを作ろうとせず、まずはHTMLの標準要素（`input type="range"` やシンプルなDOM要素の羅列）でロジックの安定性を検証（ドッグフーディング）すること。
