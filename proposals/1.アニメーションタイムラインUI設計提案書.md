# TEGAKI アニメーションタイムライン UI 選定・実装設計書

本文書は、ブラウザお絵かきツール『TEGAKI』のタイムラインUI設計に関する Gemini、GPT、Claude の各提案を統合・整理した設計決定書です。

---

## 1. タイムラインUI 選定結論

**ToonSquid 2 型の「レイヤー直結タイムライン」を基礎アーキテクチャとし、Procreate Dreams 型の「操作思想（モード分離・再生ヘッドスクラブ・折りたたみ）」を組み合わせたハイブリッド設計を採用する。**

### 各製品の分析と評価
*   **Procreate Dreams (トラック型)**
    *   *特徴*: 動画編集ソフト（Premiere等）に近いアセットトラック構造。Perform（実演記録）機能により直感的なキーフレーム生成が可能。
    *   *TEGAKIでの難点*: 既存の `LayerSystem` と構造が大きく乖離しており、データ構造が複雑化して破綻するリスクが高い。
*   **ToonSquid 2 (レイヤー直結型)**
    *   *特徴*: レイヤーパネルがそのまま右側の時間軸（フレーム）に伸びた 1レイヤー=1タイムライン行 の構造。セル長さ調整（リタイミングハンドル）が非常に直感的に操作可能。
    *   *TEGAKIでの利点*: 既存の `LayerSystem` / `data-models.js` の拡張で安全に実装でき、階層・ボーン・物理演算への拡張路をエフェクト方式で後から追加しやすい。

## 2. コア採用機能と設計方針

### ① ベース構造：レイヤー直結型（ToonSquidリスペクト）
*   既存の `Layer` オブジェクトに、フレームごとの画像データを保持する「タイムシート配列（`cells`）」と「変形（`transformKeys`）」を追加して拡張する。二重実装のリスクを最小化する。

### ② 操作思想とジェスチャー（Dreamsリスペクト）
*   **タイムライン折りたたみ**: 通常描画中はコンパクトな再生バーのみを表示し、キャンバス描画スペースを削らない「はっちゃん」の潔さを継承する。
*   **再生ヘッドのスクラブ**: ルーラー（時間軸）のドラッグによる滑らかな再生フレーム移動をサポートする。
*   **モードの分離**: 誤操作防止のため、「描画」「構成（セル操作）」「Key（キーフレーム）」「Perform」のモード切り替えをサポートする。

### ③ Perform（実演記録）の早期シード化
*   TEGAKI の既存の `Vキー変形`（`layer-transform.js`）に「フレーム記録オプション」を付与する形で Perform を設計する。これにより、新しい入力ハンドラを別実装せず、最小コストで「動かして記録する」機能を実装可能にする。

### ④ 物理演算の UI 置き場
*   物理演算はタイムライン行を展開した時に表示されるエフェクト（バネ、遅れ追従、振り子、呼吸）として設計する。メインタイムラインの構造を汚さず、物理がOFFのレイヤーに対してはゼロコストであることを保証する。

## 3. 推奨レイアウト構成

### 通常描画中（タイムライン最小化時）
```
+----------------------------------------------------------------+
|  [左ツールバー]        【 キャンバス 】          [レイヤーパネル]  |
|                                                                 |
+----------------------------------------------------------------+
| ▶  F:12/24  12fps  [onion]  [loop]          [アニメ ▲ 開く]    |
+----------------------------------------------------------------+
```

### アニメテーブル展開時
```
+----------------------------------------------------------------+
|  [左ツールバー]        【 キャンバス 】          [レイヤーパネル]  |
+----------------------------------------------------------------+
| [描画] [構成] [Key] [▶●Perform]  F:12/24  12fps  [onion][loop] |
+----------------------------------------------------------------+
| レイヤー名     | 01 02 03 04 05 06 07 08 09 10 11 12 ...        |
|────────────────|──────────────────────────────────────────────|
| ▶ 髪           | [──────髪A──────][髪B][────────髪C────────]     |
|   ≈ physics    | ~~preset:バネ ~~ (展開時のみ表示)               |
| ▶ 顔           | [────────────顔A────────────][顔B]               |
| ▶ 体           | [──────────────────体A──────────────────]        |
| ▶ 背景         | [──────────────────背景────────────────────]     |
+----------------------------------------------------------------+
```

## 4. データモデル設計（最小構成）

```javascript
AnimationProject = {
  fps: 12,
  frameCount: 24,
  currentFrame: 1,
  layers: [
    {
      layerId: "layer-abc",       // 既存 LayerSystem の ID と同期
      cells: [
        {
          cellId: "cell-001",
          startFrame: 1,
          endFrame: 8,
          drawingId: "drawing-001"
        }
      ],
      transformKeys: [],          // 後続フェーズでキーフレームを格納
      physics: null               // 後続フェーズで物理パラメータを格納
    }
  ],
  drawings: {
    "drawing-001": {
      type: "renderTextureSnapshot",
      sourceLayerId: "layer-abc"
    }
  }
}
```

## 5. 段階的実装ロードマップ

*   **Phase 4g — アニメ入口の正式化**: 通常描画状態を壊さずにサイドバーから新アニメテーブルを開閉できる接続。
*   **Phase 4h — Flipbook MVP**: 現在フレームに対応するセルを表示する、パラパラマンガの基本表示プレビュー。
*   **Phase 4i — オニオンスキン MVP**: 前後1枚ずつを薄く表示する、RenderTextureを汚さない専用合成経路。
*   **Phase 4j — セル操作**: リタイミングハンドルによる長さ変更、セル追加・分割・複製・削除。
*   **Phase 4k — 保存・ロード**: アニメ情報のプロジェクト保存統合と旧バージョン互換。
*   **Phase 5a — Transformキーフレーム**: Vキー変形の時間変化（Linear/Ease）記録開始。
*   **Phase 5b — Perform MVP**: 再生中のVキー移動自動キーフレーム記録。
*   **Phase 5c — 物理演算プレビュー**: 髪・服向けの遅れ追従物理のプレビュー実装。
