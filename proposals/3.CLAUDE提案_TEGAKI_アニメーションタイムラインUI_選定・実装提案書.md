## 1. 参照製品の独立分析

### 1-1. Procreate Dreams ── 「映像制作ソフトの UI を指で使えるようにした」製品

#### 構造上の特徴

Dreams のタイムラインは、トラックに「コンテンツブロック（クリップ）」を並べる映像編集ソフト型だ。
再生ヘッドをルーラーでスクラブし、クリップを左右に動かしてタイミングを調整する。
キーフレームはコンテンツブロックの下に折りたたまれており、展開して確認・編集する。

特筆すべきは **Perform（実演記録）** 機能。再生中にアセットを指で動かすと、その軌跡が自動的にキーフレーム化される。「打ち込む」から「演じる」へのパラダイムシフトが最大の個性だ。

もう一つは **Flipbook モード**。セルアニメ的なコマ描きに特化した表示に切り替えられる。Dreams の中でもこれが最も TEGAKI の現行フェーズと相性がよい。

#### TEGAKI 視点での評価

| 観点 | 評価 | 理由 |
|------|------|------|
| 採用したい操作感 | ◎ | スクラブ、折りたたみ、モード切替の思想が明快 |
| 採用したいデータ構造 | △ | トラック型は既存の LayerSystem と乖離が大きすぎる |
| AI 実装難易度 | × | Track / Clip / Content の概念追加が循環依存を招きやすい |
| スマホ・タブレット操作感 | ◎ | ジェスチャー中心設計で親指操作に向く |
| 液タブ精密操作 | △ | キーフレームの数値管理が弱い |

---

### 1-2. ToonSquid 2 ── 「After Effects を iPad で正しく実装した」製品

#### 構造上の特徴

ToonSquid のタイムラインは、左のレイヤーリストが右の時間軸へそのまま延びた、正統派の 2D アニメーション設計だ。1 レイヤー = 1 タイムライン行。レイヤーを追加すれば自動的にタイムライン行が増える。

**リタイミングハンドル**（セルの端をドラッグして表示長さを変更）は使い心地が特に優れており、ダイアログを開かずにセル長さを変えられる。

キーフレームは別のモードで表示する。セル描画とキーフレーム編集の画面が分離されており、「描いていたら誤ってキーフレームを打った」という事故が起きにくい。

イージングカーブは独立したグラフビューで編集できる。ボーン（Bones）はエフェクトとして各レイヤーに付与する設計で、独立したリグエディタを持たない。これは TEGAKI の将来の物理演算設計への示唆を含む（後述）。

#### TEGAKI 視点での評価

| 観点 | 評価 | 理由 |
|------|------|------|
| 採用したいデータ構造 | ◎ | LayerSystem の直接拡張で破綻しにくい |
| セルアニメ操作性 | ◎ | リタイミングハンドルが直感的で学習コスト低 |
| 物理演算への拡張路 | ◎ | ボーン/メッシュをエフェクトとして扱う設計が TEGAKI に合う |
| AI 実装難易度 | ○ | 既存 Layer オブジェクトへのオプショナル拡張で済む |
| 情報密度 | △ | 機能が多く、TEGAKI では段階的に解放しないと圧迫感が出る |

## 4. Claude 独自提案：三層モデルのタイムライン

GPT・Gemini 提案と方向性は共通しているが、以下の点で差別化する。

### 4-1. 核心的な差別化：「V キー変形 → Perform の連続体」として設計する

TEGAKI はすでに `layer-transform.js` に V キー変形（位置・スケール・回転・反転）を持っている。
これは Dreams の Perform の「手動版」に過ぎない。

**提案：V キー変形モードに「フレームを意識した記録」オプションを足す**ことで、Perform を「全く新しい機能」として作り直さずに済む。

```text
現在:  V キー → ドラッグ → bake（1枚の変形）
拡張:  V キー → アニメモード ON → ドラッグ → フレームごとに自動キーフレーム記録
```

これにより：
- Phase 5 以降の「Perform 実装」ではなく、**Phase 4g〜4h の段階から Perform の種を埋め込める**
- Perform 専用の入力ハンドラを別途作らずに済むため、AI 実装の破綻リスクを下げられる
- ユーザーは「V キーを押してドラッグするだけ」という既存の操作感でそのままアニメを作れる

### 4-2. 物理演算の UI 置き場：ToonSquid の「エフェクト付与」設計を採用

ToonSquid ではボーン・メッシュはエフェクトとしてレイヤーへ付与する。
これは `LayerSystem` への侵襲が最も少ない設計だ。

TEGAKI では物理演算も同じアーキテクチャで扱う：

```text
[レイヤー行] ▶ 展開
  ├─ cell        [───セルA───][セルB]
  ├─ transform   ◆────────◆────────◆  （Phase 5a）
  └─ physics     preset: バネ / strength: 40 / damping: 70  （Phase 5c〜）
```

物理のパラメータは別ウィンドウではなく、**タイムライン行の展開メニュー内に収める**。
常時表示しない。物理 ON のレイヤーだけ行の右端に「≈」アイコンを出す。

### 4-3. Flipbook モードの重要性（両提案で軽視されている点）

Dreams の Flipbook はセルアニメ的な「1 フレーム 1 枚」描きに特化した切り替えモードだ。
TEGAKI のユーザーがアニメに入門するとき、最初に使うのはこのモードに近い体験になるはずだ。

**Phase 4h の「セル配置に応じたキャンバス表示プレビュー」は、事実上の Flipbook MVP だ。**
このフェーズを Dreams の Flipbook と意識的に対応づけて設計することで、エントリーレベルのアニメ UX が格段に改善する。

## 5. UI レイアウト案

### 5-1. 通常描画中（タイムライン最小化）

```
+----------------------------------------------------------------+
|  [左ツールバー]        【 キャンバス 】          [レイヤーパネル]  |
|                                                                 |
+----------------------------------------------------------------+
| ▶  F:12/24  12fps  [onion]  [loop]          [アニメ ▲ 開く]    |
+----------------------------------------------------------------+
```

再生バー 1 本のみ。キャンバス面積を削らない。
「はっちゃん」の潔さを維持する。

### 5-2. アニメテーブル展開時

```
+----------------------------------------------------------------+
|  [左ツールバー]        【 キャンバス 】          [レイヤーパネル]  |
+----------------------------------------------------------------+
| [描画] [構成] [Key] [▶●Perform]  F:12/24  12fps  [onion][loop] |
+----------------------------------------------------------------+
| レイヤー名     | 01 02 03 04 05 06 07 08 09 10 11 12 ...        |
|────────────────|──────────────────────────────────────────────|
| ▶ 髪           | [──────髪A──────][髪B][────────髪C────────]     |
|   ≈ physics    | ~~preset:バネ ~~ (展開時のみ表示)               |
| ▶ 顔           | [────────────顔A────────────][顔B]               |
| ▶ 体           | [──────────────────体A──────────────────]        |
| ▶ 背景         | [──────────────────背景────────────────────]     |
+----------------------------------------------------------------+
```

**モードバー（4 種）の役割：**

| モード | 目的 | 初期実装 |
|--------|------|----------|
| 描画 | 現在フレームのセルへ通常描画 | 必須 |
| 構成 | セルの配置・長さ・分割・移動 | 必須 |
| Key | キーフレーム表示・編集 | 中期 |
| ● Perform | V キー変形のフレーム記録 | TEGAKI 独自の早期導入候補 |

## 7. 物理演算 UI の設計指針

物理演算は「最初から実装しないが、UI の置き場は今決めておく」が正しい姿勢だ。
これは GPT・Gemini 両提案と一致する。

加えて Claude が強調したいのは以下の点：

**ToonSquid の「ボーン = エフェクト」設計を物理演算にも適用せよ。**

ToonSquid ではボーンは専用タイムライン行ではなく、レイヤーのエフェクトリストに追加する形をとる。
これにより：
- メインタイムラインのデータ構造を汚さない
- ボーンを持たないレイヤーには何も増えない
- 段階的にオプションを足せる

TEGAKI での実装方針：

```javascript
// Layer オブジェクトへの最小拡張（物理演算は null がデフォルト）
{
  layerId: "layer-abc",
  cells: [...],
  transformKeys: [],
  physics: null  // または { preset: "spring", strength: 0.4, damping: 0.7 }
}
```

UI 上は、タイムライン行の展開メニューに「Physics」トグルを置く。
ON のレイヤーだけ行端に「≈」アイコンを表示する。常時全表示はしない。

物理プリセットの初期案：

| プリセット | 用途 | パラメータ |
|-----------|------|-----------|
| なし | 通常レイヤー | — |
| 遅れ追従 | 髪・服 | 遅れフレーム数、戻り強さ |
| バネ | アホ毛・アクセサリ | 強さ、減衰 |
| 振り子 | しっぽ・リボン | 支点位置、長さ |
| 呼吸 | 体・胸 | 周期、振幅 |

数値スライダーは最小限（2〜3 本）から始める。詳細設定は後続で足す。

## 8. データモデル案

GPT 提案のモデルが良くできているため、それをベースに最小限の修正を加える。

### 用語定義（GPT 提案に準拠）

| 用語 | 意味 |
|------|------|
| Frame | 時間上の 1 コマ（1〜N） |
| Layer | 既存の描画レイヤー |
| Cell | あるレイヤーの特定期間に表示される描画単位 |
| Drawing | Cell が参照する画像データ（RenderTexture スナップショット） |
| Keyframe | あるフレームにおける Transform 等の値 |
| Track | UI 上の行（初期は 1 レイヤー = 1 トラック） |
| Physics | レイヤーに付与するエフェクト型の物理パラメータ（null = 無効） |

### 最小データモデル（Phase 4 系）

```javascript
AnimationProject = {
  fps: 12,
  frameCount: 24,
  currentFrame: 1,
  layers: [
    {
      layerId: "layer-abc",       // 既存 LayerSystem の ID と同期
      cells: [
        {
          cellId: "cell-001",
          startFrame: 1,
          endFrame: 8,
          drawingId: "drawing-001"
        }
      ],
      transformKeys: [],          // Phase 5a で埋める
      physics: null               // Phase 5c で埋める
    }
  ],
  drawings: {
    "drawing-001": {
      type: "renderTextureSnapshot",
      sourceLayerId: "layer-abc",
      frameIndex: 0
    }
  }
}
```

## 9. 実装ロードマップ

GPT 提案のロードマップを参考にしつつ、Claude 独自視点を追加する。

### Phase 4g：アニメ入口の正式化

左サイドバーのアニメアイコンから新アニメテーブルを開く。
受け入れ条件：通常描画への副作用なし、テーブルを閉じても描画状態が戻る。

### Phase 4h：Flipbook MVP（Dreams リスペクト）

現在フレームに対応するセルをキャンバスへ表示する。
**Dreams の Flipbook と同等の体験**を最初に届ける。
受け入れ条件：レイヤー表示状態を破壊しない、プレビュー終了時に元に戻る。

### Phase 4i：オニオンスキン MVP

前後 1 枚ずつを薄く表示。RenderTexture を汚さない専用合成経路を作る。

### Phase 4j：セル操作（リタイミングハンドル含む）

セルの端ドラッグでの長さ変更（ToonSquid のリタイミングハンドル）が最優先。
次に追加・分割・複製・削除。

### Phase 4k：保存・ロード

アニメ情報をプロジェクト保存へ統合。旧データとの互換を維持する。

### Phase 4l：APNG / GIF / WebP 連番出力

既存 `apng-exporter.js`・`gif-exporter.js` を活用する。

### Phase 5a：Transform キーフレーム

V キー変形の行列値をフレームごとに保存・補間する。
**Perform の種を同フェーズで埋め込む**（V キー操作をフレームに記録するオプションのみ先行実装）。

### Phase 5b：Perform MVP

再生中の V キー変形を自動キーフレーム化。位置のみから始める。

### Phase 5c：物理演算プレビュー

遅れ追従プリセットから。保存前のプレビューのみで開始する。

## 10. AI 実装規律（TEGAKI.md に追加すべき事項）

1. **Cell と Layer を混同しない**: Cell は Layer の時間スライスであり、Layer ではない。EventBus イベント名に明示する（例: `animation:cell-selected` ≠ `layer:activated`）。
2. **オニオンスキンは描画経路に乗せない**: RenderTexture への焼き込み経路と分離した合成専用パスで処理する。
3. **Perform は V キー変形の延長として実装する**: `layer-transform.js` の既存ハンドラに「フレーム記録モード」フラグを足す。新ファイルで別実装しない。
4. **物理演算は `physics: null` をデフォルトにし、null チェックを全経路に入れる**: 物理が OFF のレイヤーにゼロコストであることを保証する。
5. **タイムライン展開・折りたたみは GSAP で行う**: 既存の UI アニメーション資産を使い、新規アニメーションエンジンを混入させない。

## 11. まとめ

| 結論 | 詳細 |
|------|------|
| **基礎構造** | ToonSquid 2 型レイヤー直結タイムラインを採用 |
| **操作感** | Procreate Dreams 型のモード分離・スクラブ・折りたたみを採用 |
| **独自強化① Perform** | V キー変形の延長として早期に設計の軸へ置く |
| **独自強化② 物理演算** | ToonSquid のエフェクト型付与設計で `physics: null` デフォルト |
| **独自強化③ Flipbook** | Phase 4h を Dreams Flipbook と意識的に対応づけてエントリー UX を磨く |
| **データモデル** | GPT 提案の用語定義をベースに採用 |
| **アーキテクチャ統合** | Gemini 提案の LayerSystem 拡張方針を採用 |

TEGAKI が目指すのは「Dreams の気持ちよさで操作し、ToonSquid の正確さで管理できる、はっちゃんの潔さを持ったブラウザアニメツール」だ。
