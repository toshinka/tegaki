# WebGL2化実装に関する質問・提案書

## 🚨 重要な問題点と確認事項

### 1. シェーダーコード未定義の問題

**問題**: 指示書でシェーダーコードが `const vs = '#version 300 es ... (前回の指示書通り)'` のように省略されており、実際のシェーダーコードが不明です。

**質問**: 
- ベジェストローク描画用の頂点シェーダーとフラグメントシェーダーの具体的なコードを提供していただけますか？
- 筆圧による線幅変化をどのようにシェーダーで実装すべきでしょうか？

### 2. ベジェ曲線のWebGL描画手法

**問題**: 現在のCanvas2D実装では `ctx.bezierCurveTo()` を使用していますが、WebGLではベジェ曲線の直接的な描画メソッドがありません。

**質問**:
- ベジェ曲線をWebGLで描画する手法として、以下のどちらを推奨しますか？
  1. **テッセレーション方式**: CPU側でベジェ曲線を直線セグメントに分割
  2. **ジオメトリシェーダー方式**: GPU側で曲線補間（WebGL2で可能）
- 筆圧による動的線幅変化をどのように頂点データに組み込むべきでしょうか？

### 3. `twgl.js`の頂点データ形式

**問題**: `_convertStrokeToVertexArray`メソッドで返すべき配列形式が不明確です。

**質問**:
- `twgl.createBufferInfoFromArrays`に渡すべき配列の具体的な形式を示していただけますか？
- 位置、色、筆圧情報をどのような属性名で渡すべきでしょうか？

### 4. 段階的移行戦略の提案

**問題**: 既存のCanvas2D実装から一気にWebGL専用に変更すると、デバッグが困難になる可能性があります。

**提案**:
以下の段階的実装アプローチはいかがでしょうか？

1. **Phase 1**: WebGL初期化とフォールバック機能の実装
2. **Phase 2**: 基本的な直線描画のWebGL化
3. **Phase 3**: ベジェ曲線描画のWebGL化
4. **Phase 4**: 筆圧・動的線幅のWebGL実装
5. **Phase 5**: Canvas2D依存の完全除去

### 5. エラーハンドリングの改善提案

**現在の指示**:
```javascript
if (!gl) {
    document.getElementById('error-overlay').style.display = 'flex';
    return; // 処理を完全に中断
}
```

**提案**: より詳細なエラー情報を提供できませんか？
- WebGL2サポート状況の詳細診断
- 代替手段の提示（WebGL1への降格など）
- ユーザーフレンドリーな解決策の提示

### 6. パフォーマンス最適化戦略

**質問**:
- 大量のストローク描画時のバッファ管理戦略はありますか？
- リアルタイム描画中のフレームレート維持のための最適化指針を提供していただけますか？

### 7. 既存機能の互換性確保

**懸念**: 現在の `VectorStroke`, `BezierStrokeOptimizer` などのクラスがCanvas2D前提で設計されています。

**質問**:
- これらのクラスの内部実装もWebGL対応に変更すべきでしょうか？
- それとも、インターフェースのみ維持してWebGL描画に対応させるべきでしょうか？

## 🎯 実装方針の明確化要求

### 優先度1: シェーダーコードの提供
具体的なシェーダーコードなしでは実装を開始できません。

### 優先度2: ベジェ曲線描画手法の決定
テッセレーション vs ジオメトリシェーダーの方針決定が必要です。

### 優先度3: 頂点データ形式の標準化
`twgl.js`との連携部分の具体的な実装方法が必要です。

## 💡 代替実装案

もし上記の詳細仕様が決定困難な場合、以下の最小実装から開始することを提案します：

1. **WebGL初期化部分のみ実装** - エラーハンドリング含む
2. **基本的な点描画のWebGL化** - シンプルな頂点・フラグメントシェーダー
3. **段階的な機能拡張** - ベジェ曲線、筆圧対応を順次追加

この方針であれば、具体的な技術仕様を段階的に決定しながら実装を進めることが可能です。
