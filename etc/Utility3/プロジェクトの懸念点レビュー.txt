プロジェクトの懸念点レビュー
現在のコードベース全体をレビューし、将来的に問題を引き起こす可能性のある、特に注意すべき点を3つのカテゴリに分けて整理しました。

🔥【最重要懸念】責務の重複と曖昧さ
現在、複数の「マネージャー」が似たような責務を持っており、どこが何を管理しているのかが非常に分かりにくい状態です。これは、機能追加や修正の際に「どこを直せばいいのか分からない」という混乱の最大の原因となります。

懸念1：描画と表示の責務が分散している
現状の問題点:

CanvasManager: 主にキャンバスの表示（拡大縮小・回転）を管理しています。

DrawingEngine: ユーザーの入力（ポインタイベント）を処理しています。

RenderingBridge: 実際の描画命令を発行しています。

WebGLEngine: レイヤーを合成して最終表示を行っています。

なぜ危ういか:
「線を描画する」という一つの機能が、4つのクラスにまたがって実装されています。例えば、描画処理のパフォーマンスを改善したい場合、これら全てのファイルを調査・修正する必要があり、非常に困難です。

改善の方向性:

CanvasManager は、ユーザー入力の受付と、表示上の変換（ズーム・回転）に責務を特化させる。

DrawingEngine という名前をやめ、RenderingEngine のような名前に統一し、レイヤーへの描画と最終的な画面合成の全責任を負わせる。RenderingBridge と WebGLEngine の機能はこの新しいエンジンに統合します。

🔧【高重要懸念】命名規則とデータフローの不一致
クラスや関数の名前がその役割を正確に表していない、またはデータの流れが直感的でない箇所が見受けられます。

懸念2：「Manager」と名の付くクラスの役割が広すぎる
現状の問題点:

LayerManager はレイヤーのリストを管理していますが、同時に「アクティブレイヤー」という状態も管理しています。

ToolManager は現在のツールを管理していますが、ツールの実体（PenToolなど）は toolset.js にあります。

なぜ危ういか:

「アクティブレイヤー」という状態は、UIの状態なのか、描画エンジンの状態なのかが曖昧です。LayerManager がこれを持つと、UIの都合で LayerManager を修正する必要が出てくるなど、責務が混在し始めます。

データの流れが一方通行でなく、相互に参照しあう「循環参照」に陥りやすくなります。（例: LayerManager が UIManager を呼び、UIManager が LayerManager を呼ぶ）

改善の方向性:

LayerManager は純粋にレイヤーのリスト（配列）の管理（追加、削除、順序変更）だけに特化させます。「アクティブレイヤー」という状態は、UIの状態を管理する UIManager や、実際に描画を行う RenderingEngine が持つべきです。

データの流れを UI → Core Engine → Manager のように、できるだけ一方向に限定するルールを設けます。

🧹【中重要懸念】将来の拡張性の問題
現在の実装は、今の機能には十分でも、少し複雑な機能を追加しようとすると破綻する可能性を秘めています。

懸念3：座標変換のロジックが点在している
現状の問題点:

スクリーン座標とキャンバス座標の変換は transform-utils.js にまとめられています。これは良い点です。

しかし、実際にその変換を使うのは DrawingEngine であり、変換に必要な情報（scale, translation）を持っているのは CanvasManager です。

なぜ危ういか:

DrawingEngine が座標変換を行うたびに、CanvasManager から変換情報を取得する必要があります。これはクラス間の結合度を不必要に高めてしまいます。

将来、例えば「レイヤーごとの拡大縮小」のような機能を追加したくなった場合、この構造では対応が非常に困難になります。

改善の方向性:

座標変換の責務は、変換情報を持っている CanvasManager に集約します。

CanvasManager に screenToCanvas(x, y) のようなメソッドを持たせ、他のモジュールはそれを呼び出すだけで済むようにします。これにより、変換ロジックの変更が必要になっても、修正は CanvasManager だけで完結します。

✅ まとめと次のステップ
まずは、最もクリティカルな**懸念1「描画と表示の責務分散」**から手をつけることをお勧めします。

具体的には、RenderingBridge と WebGLEngine の機能を DrawingEngine に統合し、名前を RenderingEngine に変更するリファクタリングから始めると、プロジェクト全体の構造が大きく改善され、他の問題にも取り組みやすくなります。
