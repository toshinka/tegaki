v4.4策定後、Phase1に特化した憲章（v5.1）の設計過程で、いくつかの重要な発見と、それに伴うアーキテクチャの進化がありました。この知見をマスター憲章に統合し、プロジェクト全体の矛盾を解消することが今回のタスクです。

【最重要の変更哲学】 AIの「誘惑」を断ち切り、曖昧さを排除する
v4.4の設計には、AI（あるいは人間）に解釈の余地を与えてしまう「曖昧さ」が残っていました。それが**「描画モード」という概念**です。

この「モード」という抽象的な概念が存在する限り、AIは「ペンを実装して」という指示に対し、最も安易な手段であるCanvas2Dを使ってしまう「誘惑」に駆られます。これをルールで縛っても、いずれ解釈のミスが発生します。

v5.1で得られた結論は、「ルールで縛る」のではなく「アーキテクチャで強制する」ことです。

新しい哲学は**「ツール選択とエンジン起動の厳格な連動」**です。利用者が「ペンツール」を選択した瞬間、システムは物理的にBezier.js用のエンジンしか起動できなくなります。この「曖昧さの排除」が、これから行う全ての改訂の根幹にあることを強く意識してください。

【具体的な改訂指示】
1. 「描画モード」概念の完全撤廃
v4.4に存在する「描画モード」に関連する全ての記述と設計を削除します。

削除対象:


enum DrawingMode 


DrawingModeController 




TOOL_MODE_MAPPING 

フォルダ 

src/engine/modes/ および src/features/modes/ 



AnimatedDrawingModeToggle.js のようなモード切替UI 

理由: これらの存在 자체가、AIに「モード」という誤解を招く余地を与えるためです。

2. 新原則「ツール・エンジン厳格連動」の導入
「描画モード」に代わり、v5.1で確立された「ツール選択がエンジン起動を一意に決定する」という原則を導入します。

導入対象:

v5.1の ToolEngineController の設計思想を、v4.4のアーキテクチャに統合します。

コントローラーは、どのツールがどのエンジンクラスとどのCanvas/Div要素に紐づくかを定義した設定オブジェクト（Binding Map）を保持します。


ToolActions  から呼び出される 

selectTool メソッドが、エンジンの切り替えを行う唯一のトリガーとなります。

3. 課題①：ツールの拡張性への対応
v5.1の設計では「1ツール＝1エンジン」でしたが、これでは将来「同じ

Canvas2DRenderer  を使うが設定が異なるツール（例：ブラシと塗りつぶし）」に対応できません。この問題をマスター憲章で解決します。

解決策:

ToolEngineController が持つ設定オブジェクトを拡張し、ツールの初期設定（initialConfig） を含められるようにします。

実装例:

JavaScript

const TOOL_ENGINE_BINDING = {
    'pen': {
        engineClass: 'BezierStrokeRenderer',
        // このツールには固有の設定は不要
    },
    'brush': {
        engineClass: 'Canvas2DRenderer',
        initialConfig: { brushType: 'normal', blendingMode: 'source-over' }
    },
    'airbrush': {
        engineClass: 'Canvas2DRenderer',
        initialConfig: { brushType: 'airbrush', blendingMode: 'source-over' }
    }
};
ToolEngineController は、ツール選択時にこの initialConfig を取得し、対応するエンジンの initialize(config) メソッドに渡して起動します。

4. 課題②：エンジン横断Undo（取り消し）への対応
v5.1では未解決だったUndo問題は、マスター憲章で対応方針を定める必要があります。エンジンごとに独立した履歴を持つと、システム全体でのUndo/Redoが破綻します。

解決策:


コマンドパターンを全面的に採用します。

各エンジン（

BezierStrokeRenderer, Canvas2DRenderer  等）は、直接Canvasに描画する代わりに、

アクション（描画、移動など）を実行するためのコマンドオブジェクトを生成します。

各コマンドオブジェクトは、execute() と undo() の2つのメソッドを持ちます。


HistoryActions  は、このコマンドオブジェクトを受け取って 

execute() を実行し、そのオブジェクトを単一の履歴スタック（Undoスタック）に積みます。

ユーザーが「取り消し」を押すと、HistoryActions はスタックからコマンドを1つ取り出し、その .undo() メソッドを実行します。

これにより、どのエンジンから発行されたアクションであっても、統一された方法で取り消しが可能になります。

5. フォルダ構成とファイル名の更新
上記の変更に伴い、v4.4のフォルダ構成  を更新します。

src/features/history/  配下に、コマンドオブジェクトのクラス群を格納する 

commands/ ディレクトリを新設することを推奨します。
