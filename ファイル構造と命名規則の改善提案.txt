ファイル構造と命名規則の改善提案現在のプロジェクトが抱える問題の多くは、ファイルとディレクトリの構成に起因しています。ここでは、より堅牢で将来の機能追加に対応しやすい構造への改善案を提案します。Part 1: 現状のファイル構造と命名の問題点現在の構造は「機能別」のようでいて、実際には「役割別 (manager, core)」の分類が混在しています。これが混乱の根本原因です。問題点①：「Manager」という曖昧な命名canvas-manager, layer-manager, tool-manager など、多くのファイルが Manager という名前を持っています。なぜ問題か？: 「管理」という言葉は非常に曖昧で、何（データ）を、どのように（ロジック）管理するのかがファイル名から読み取れません。 結果として、1つのManagerファイルに状態管理、UI操作、コアロジックなど、複数の責務が詰め込まれがちになります。問題点②：core と ui の境界が曖昧core/canvas-manager.js はキャンバスの表示（UIに近い）を扱い、ui/tool-manager.js は現在のツールという状態（コアロジックに近い）を管理しています。なぜ問題か？: UIとコアロジックの依存関係が複雑に絡み合ってしまいます。例えば、UIのボタンを追加しただけなのに、core内部のファイルを修正する必要が出てくるなど、変更の影響範囲が予測しづらくなります。問題点③：アプリケーションの「状態」が分散している「現在選択中のレイヤー」は LayerManager に、「現在選択中のツール」は ToolManager に、「現在の色」は ColorManager にあります。なぜ問題か？: アプリケーションの状態が各マネージャーに散らばっているため、状態の全体像を把握するのが困難です。ある操作が他の機能にどう影響するかを追跡するのが難しく、バグの温床となります。Part 2: 改善提案： 「関心事の分離」に基づく新構造これらの問題を解決するため、**「何をするか」でファイルを分ける「関心事の分離」**という原則に基づいた新しいファイル構造を提案します。提案する新しいディレクトリ構造src/
│
├─ app/
│   ├─ main.js             # アプリケーションの起動と全体の統括 (旧 core-engine.js)
│   └─ state.js            # アプリケーション全体の状態を一元管理する場所
│
├─ engine/
│   ├─ renderer.js         # WebGLでのレイヤー合成と最終描画 (旧 webgl-engine, rendering-bridge)
│   └─ viewport.js         # 表示領域の管理（ズーム、パン、回転）(旧 canvas-manager)
│
├─ features/
│   ├─ layers/
│   │   ├─ layer.js          # Layerクラスの定義
│   │   └─ layer-store.js    # レイヤーリストの状態管理 (旧 layer-manager)
│   │
│   ├─ tools/
│   │   ├─ pen-tool.js       # 各ツールの実装
│   │   └─ tool-store.js     # 現在のツールや設定の状態管理 (旧 tool-manager, pen-settings-manager)
│   │
│   └─ history/
│       └─ history-store.js  # Undo/Redoのための状態管理
│
├─ ui/
│   ├─ LayerPanel.js       # レイヤーパネルのUIコンポーネント
│   ├─ Toolbar.js          # ツールバーのUIコンポーネント
│   └─ ui-controller.js    # UI全体のイベントと状態の同期
│
├─ data/
│   └─ storage.js          # IndexedDBとのやり取り (旧 db-indexed.js)
│
└─ utils/
    └─ transform.js        # 汎用的な座標変換など

新しい命名規則のポイントManager の廃止 → 具体的な役割名へ曖昧な Manager は使わず、Renderer (描画するもの), Viewport (表示領域), Controller (制御するもの), Store (状態を保存するもの) のように、役割が明確な単語を使います。Store (ストア) の導入アプリケーションの状態（どのツールが選ばれているか、レイヤーのリストなど）は、すべて Store という名前のファイルに集約します。これにより、データの流れが「UI → Storeを更新 → Storeの変更を各部が検知して再描画」という一方向になり、非常に見通しが良くなります。features ディレクトリの新設「レイヤー機能」「ツール機能」のように、アプリケーションの主要な機能ごとにディレクトリを分けます。関連するファイルが近くにまとまるため、開発効率が向上します。Part 3: この改善によるメリットこの構造に変更することで、以下のような大きなメリットが生まれます。予測可能性の向上: 「レイヤーのリストを操作したいなら features/layers/layer-store.js を見ればいい」というように、ファイルやディレクトリの役割が明確になり、コードを探す時間が激減します。テストの容易化: UIとロジック、状態が綺麗に分離されるため、それぞれの機能を単体でテストすることが容易になります。AIとの連携強化: 「tool-store.js にブラシサイズの状態を追加して」のように、AIに対して非常に具体的で誤解のない指示を出せるようになり、生成されるコードの精度が格段に向上します。これは大規模な変更となりますが、プロジェクトを一度リセットするつもりでこの構造を導入すれば、今後の開発速度と品質は飛躍的に向上するはずです。
