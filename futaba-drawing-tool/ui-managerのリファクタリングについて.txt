ご指摘の通り、現在の ui-manager.js は複数の独立したクラス定義と、それらを統括するマネージャーのロジックが混在しており、責務過多な状態にあります。これは、アプリケーションの規模が大きくなるにつれて、可読性やメンテナンス性の低下を招く可能性があります。


Rulebook.txt  で掲げられている「UIロジックと描画ロジックは分離する」「1関数1責務」という原則に基づき、ui-manager.js の責務を分割・整理することを強く推奨します。

◆ 現状の ui-manager.js の問題点
現在の ui-manager.js は、以下の責務をすべて担っています。

UIコンポーネントクラスの定義:

SliderController

PopupManager

StatusBarManager

PresetDisplayManager

SettingsPanelManager

UI全体の初期化と統括:

UIManager クラスが、上記の全コンポーネントを初期化し、互いを連携させている。

DOMイベントの広範な管理:

ツールボタン、スライダー、プリセット、リサイズ、チェックボックスなど、多数のDOMイベントリスナーを設定している。

他マネージャーとの密な連携:

SettingsManager と連携して設定パネルを構築・管理している。

DrawingToolsSystem と連携してプリセットやブラシ設定のUIを更新している。

◆ 解決策：UIコンポーネントのモジュール化（推奨）
最もクリーンで拡張性の高い方法は、ui-manager.js 内で定義されている各UIコンポーネントクラスを、それぞれ独立したファイルに分割することです。これにより、ui-manager.js は純粋な「UIの統括役（Orchestrator）」に特化できます。

理想的なファイル構成案
/js
|-- ui/  (新規作成)
|   |-- slider-controller.js
|   |-- popup-manager.js
|   |-- status-bar-manager.js
|   |-- preset-display-manager.js
|   |-- settings-panel-manager.js
|
|-- app-core.js
|-- drawing-tools.js
|-- history-manager.js
|-- settings-manager.js
|-- ui-manager.js  (リファクタリング後)
|-- main.js
リファクタリング後の ui-manager.js の役割
リファクタリング後の ui-manager.js は、各UIコンポーネントのクラス定義を持たず、それらをインポートして初期化し、協調させる役割に集中します。

JavaScript

// ui-manager.js (リファクタリング後のイメージ)

// 各UIコンポーネントをインポート (将来的にはESMで)
// import { SliderController } from './ui/slider-controller.js';
// import { PopupManager } from './ui/popup-manager.js';
// ...

class UIManager {
    constructor(app, toolsSystem, settingsManager) {
        this.app = app;
        this.toolsSystem = toolsSystem;
        this.settingsManager = settingsManager;

        // 各UIコンポーネントのインスタンスを生成
        this.popupManager = new PopupManager();
        this.statusBar = new StatusBarManager();
        this.presetDisplay = new PresetDisplayManager(toolsSystem);
        this.settingsPanel = new SettingsPanelManager(settingsManager, this);
        // ... 他のUIコンポーネントも同様に
    }

    async init() {
        // 各UIコンポーネントのセットアップメソッドを呼び出す
        this.setupPopups();
        this.setupSliders();
        // ...

        // 統括マネージャーとしてのイベントリスナーを設定
        this.setupAppEventListeners();
    }

    // 各コンポーネントを連携させるロジック
    // (例: プリセットがクリックされたら、スライダーの値を更新する)
}
◆ 具体的なリファクタリング手順
ui/ ディレクトリの作成:
プロジェクト内に js/ui/ のようなディレクトリを作成します。

クラスのファイル分割:
ui-manager.js から SliderController、PopupManager、StatusBarManager、PresetDisplayManager、SettingsPanelManager のクラス定義を、それぞれ対応する新しいファイル（例: js/ui/slider-controller.js）にカット＆ペーストします。

index.html の修正:
新しく作成したUIコンポーネントのJavaScriptファイルを index.html に追加します。ui-manager.js よりも前に読み込む必要があります。

HTML

<script src="https://cdnjs.cloudflare.com/ajax/libs/pixi.js/7.4.2/pixi.min.js"></script>

<script src="app-core.js"></script>

<script src="settings-manager.js"></script>

<script src="history-manager.js"></script>

<script src="drawing-tools.js"></script>

<script src="ui/slider-controller.js"></script>
<script src="ui/popup-manager.js"></script>
<script src="ui/status-bar-manager.js"></script>
<script src="ui/preset-display-manager.js"></script>
<script src="ui/settings-panel-manager.js"></script>

<script src="ui-manager.js"></script>

<script src="main.js"></script>
ui-manager.js のリファクタリング:
ファイル分割後の ui-manager.js から、移動したクラス定義を削除し、純粋な統括ロジックのみを残します。

◆ このリファクタリングによるメリット

責務の明確化: 各ファイルが単一の責任を持つようになり、Rulebook.txt  の設計思想に沿います。

可読性とメンテナンス性の向上: 特定のUIパーツ（例: スライダー）に修正が必要な場合、ui/slider-controller.js を見ればよいため、影響範囲の特定が容易になります。

再利用性の向上: 各UIコンポーネントが独立しているため、将来的に他のプロジェクトで再利用しやすくなります。

TypeScript/Vite移行への布石: ファイルが適切にモジュール化されている状態は、import/export を使用するモダンなビルド環境への移行を非常にスムーズにします。

settings-manager.js に SettingsPanelManager を移すことも考えられますが、SettingsPanelManager はDOM要素の操作というUIの責務を強く持つため、UI関連のファイル群にまとめておく方が、UIとロジックの分離がより明確になり、良い設計と言えるでしょう。
