改修計画書1. 検査概要検査対象提供されたエラーログとファイルURLに基づき、以下のファイルを検査しました：ui-manager.js (エラー発生元)
main.js (システムチェックと初期化ロジック)
utils.js (エラーハンドリングとユーティリティ)
index.html (スクリプト読み込み順序確認)
その他の関連ファイル (依存関係確認のため、app-core.js, drawing-tools.js など参照)

検査方法:各ファイルのコンテンツをブラウズツールで取得し、コードを分析。
エラーログと照合して原因を特定。
DRY (Don't Repeat Yourself: コード重複回避) と SOLID 原則 (Single responsibility, Open-closed, Liskov substitution, Interface segregation, Dependency inversion) を基準に問題点を評価。

検査結果主な原因: ui-manager.js の line 500 に SyntaxError (Invalid or unexpected token) が発生。これによりスクリプト全体が解析失敗し、UIManager オブジェクトがグローバルスコープに定義されない。結果、main.js の checkIntegratedSystems 関数で uiManager が不足として検知され、Phase2統合システムチェックが失敗。想定される具体的なエラー要因: 文字列内の無効な文字 (例: UTF-8 BOM, スマートクォート, 予期せぬ日本語文字のエスケープ不足)、欠損した終端記号 (;, ", }, ) )、または不正なトークン (例: 予約語の誤用)。

依存関係の問題: index.html でスクリプトの読み込み順序が不適切な可能性。ui-manager.js が main.js より前にロードされていない場合、UIManager が利用不可になる。
エラーハンドリングの状態: utils.js で createApplicationError, logError, handleGracefulDegradation が定義されており、main.js で使用されているが、重複したエラーチェックコードが存在 (DRY違反)。
システム全体の問題:main.js で複数システム (Pixi, DrawingToolsSystem, HistoryManager など) のチェックが集中 (Single responsibility 原則違反: 初期化とチェックの責務が混在)。
依存がハードコード (window.UIManager などグローバル参照) されており、Dependency inversion 原則に違反。テストしにくく、エラー回復力が低い。
他のファイル (例: app-core.js, drawing-tools.js) では UI関連の依存が散見されるが、ui-manager.js の失敗で連鎖エラー発生。

パフォーマンス/安定性: Phase2最適化 (main.js に記載) が部分的に適用されているが、ui-manager.js のエラーで無効化。エラー回復 (graceful degradation) は utils.js に存在するが、syntax error レベルでは適用不可。

2. 改修方針原則遵守:DRY: エラーハンドリングや設定取得などの共通ロジックを utils.js に集約。重複コードを削除。
SOLID:Single responsibility: 各モジュールに1つの責務のみ割り当て (例: main.js は初期化フロー制御のみ、ui-manager.js はUI構築のみ)。
Open-closed: 拡張可能にするため、クラス/関数をインターフェースベースに変更。新機能追加時、既存コードを変更せず拡張。
Liskov substitution: 継承使用時、サブクラスがベースクラスを完全に置換可能に (現在未使用のため、導入時に遵守)。
Interface segregation: 大きなインターフェースを小さく分割 (例: UIマネージャのAPIを細分化)。
Dependency inversion: 高レベルモジュール (main.js) が低レベル (ui-manager.js) に直接依存せず、抽象 (インターフェース) を介す。依存注入 (DI) を導入。

全体目標: エラー耐性を高め、アプリケーションの安定初期化を実現。Phase2最適化を拡張し、モジュール化を推進。
優先度: 高 (即時修正: syntax error)、中 (原則準拠リファクタリング)、低 (追加機能)。

3. 具体的な改修計画3.1 即時修正 (エラー解消)ui-manager.js の syntax error 修正:line 500 付近を検査: トークン解析エラーの原因を特定 (例: 文字列内の無効文字をエスケープ、欠損記号を追加)。
修正例: もし console.log("テスト”); (スマートクォート) なら console.log("テスト"); に変更。
全ファイルを通し、UTF-8 エンコーディングを確認 (BOM除去)。
修正後: JSHint/ESLint で静的解析を実行し、他の潜在エラーを検知。

スクリプト読み込み順序の最適化 (index.html):<script> タグの順序を調整: config.js → utils.js → ui-manager.js → main.js の順にロード。
修正例:

<script src="config.js"></script>
<script src="utils.js"></script>
<script src="ui-manager.js"></script>
<script src="main.js"></script>

async/defer 属性を検討: 非同期ロード時は依存を module (ES6 import) に移行。

3.2 リファクタリング (DRY・SOLID準拠)DRY適用:エラーチェックコードの集約: main.js の checkIntegratedSystems を utils.js に移動。main.js からは呼び出しのみ。例: utils.js に checkAllSystems() 関数を追加し、systemsStatus オブジェクトを返す。

重複ログ関数: console.log/error を utils.logError に統一。

SOLID適用:Single responsibility: main.js を初期化専用に。システムチェックを別モジュール (system-monitor.js) に分離。
Open-closed/Interface segregation: UIManager をクラス化し、インターフェース定義 (例: IUIManager { initUI(); updateUI(); })。拡張時、実装クラスを追加。
Dependency inversion: 依存注入導入。main.js で UIManager をインスタンス化せず、コンストラクタ引数で渡す。例 (ES6クラス化):

class MainInitializer {
  constructor(uiManager, toolsSystem) {
    this.uiManager = uiManager;
    this.toolsSystem = toolsSystem;
  }
  initialize() { /* 初期化ロジック */ }
}
// 使用: const initializer = new MainInitializer(new UIManager(), new DrawingToolsSystem());

Liskov substitution: UIコンポーネントの継承 (例: PresetManager extends BaseManager) で、置換性を確保。

エラー回復力強化:utils.handleGracefulDegradation を拡張: uiManager 失敗時、フォールバックUI (シンプルモード) を提供。
try-catch を初期化周りに追加 (ただし syntax error はキャッチ不可のため、モジュール化で回避)。

3.3 追加対策テスト導入: Unitテスト (Jest) で各モジュールを検証。例: ui-manager.js の syntax をテスト。
モジュール化移行: 将来的に ES modules (import/export) に変換。ブラウザ互換のため Babel 使用。
パフォーマンス監視: performance-monitor.js を統合し、初期化時間をログ。

4. 実施スケジュール日1: syntax error 修正 + 読み込み順序調整 (テスト実行)。
日2-3: リファクタリング (DRY/SOLID適用) + ユニットテスト。
日4: 全体統合テスト + デプロイ。

5. 期待効果エラー解消: UIManager が正しくロードされ、アプリケーション起動成功。
保守性向上: DRYでコード量削減、SOLIDで拡張しやすく。
安定性: エラー回復力が強化され、Phase2目標達成。
