# JavaScript + PixiJS v7 Rulebook v8（Phase3統合システム対応版）

## 1. 基本理念

### 1.1 核心方針
- **安定動作・可読性・拡張性を優先**し、将来的なTypeScript + Vite + ES6移行を意識する
- **PixiJSエコシステム活用**により、車輪の再発明を避け、標準実装を採用する
- **統合システム設計**により、座標変換・履歴管理・デバッグの一元化を推進する
- **段階的品質向上**と既存システムとの互換性維持を両立する
- **AI協働最適化**を考慮した標準パターンの採用を推奨する

### 1.2 実装原則
- 現在はJavaScript+fetch API分割でありグローバル変数形式で実装する
- ファイル間の依存は明確化し、循環参照を作らない
- **SOLID・DRY原則準拠**: 単一責任・責務分離・重複コード排除を徹底する
- **統合システム設計**: 座標変換・履歴管理・監視機能の統合による効率化
- **モジュール分割設計**: 機能ごとの専門ファイル分離による保守性向上
- PixiJS v7の公式API仕様に沿い、非推奨機能は使用しない
- ES2023構文使用OK、OffscreenCanvasや最新Canvas APIも使用OK
- 最新Chrome+デスクトップPC環境を想定、後方互換は意識しない
- **ファイルサイズ制限**: 単一ファイルは800行以下を目安とし、それを超える場合は責務に応じて分割する
- 必ずdebug/またはmonitoring/ディレクトリの既存モジュールを確認し、重複を避ける事

## 4. コードスタイル規約（Phase3統合システム対応）

### 4.1 命名規則（統合システム拡張版）
- **変数 / 関数**: `camelCase`（例：`drawLine`, `pointerPosition`）
- **定数**: `SCREAMING_SNAKE_CASE`（例：`CANVAS_WIDTH`, `BG_COLOR`）
- **クラス**: `PascalCase`（例：`TransformManager`, `LogBasedHistoryManager`）
- **ファイル名**: `kebab-case`（例：`transform-manager.js`, `debug-manager.js`）
- **統合システム関数**: `integrated{Feature}`（例：`integratedTransform`, `integratedHistory`）
- **統合システム設定**: `{SYSTEM}_INTEGRATION_CONFIG`（例：`TRANSFORM_INTEGRATION_CONFIG`）

### 4.2 統合システム使用パターン
```javascript
// 推奨: 統合システム経由の操作
const transform = app.transformManager.transformLayer(layerId, { dx: 5, dy: 5 });

// 推奨: 統合履歴との連携
app.historyManager.recordAction({
    type: 'transform', 
    layerId, 
    transform, 
    timestamp: Date.now()
});

// 推奨: 統合デバッグ監視
const stats = app.debugManager.monitorIntegratedPerformance();

// 禁止: 直接的な分散操作
// layer.x += 5; // NG - TransformManager経由必須
// customHistoryArray.push(data); // NG - HistoryManager経由必須
```

## 10. 禁止事項（Phase3統合システム対応）

### 10.1 基本禁止事項（継続）
- DOM直接操作（Pixi外の`querySelector`等）
- Pixi内部プロパティへの直接アクセス
- 複数責務を1関数にまとめる
- **単一ファイル800行超過**（分割必須）

### 10.2 統合システム固有禁止事項
- **統合システム迂回**: TransformManager・HistoryManager・DebugManagerを経由しない直接操作
- **分散状態管理**: 統合システムと並行した独自状態管理の新規作成
- **履歴システム重複**: 独自履歴機能の新規実装（LogHistoryManager使用必須）
- **座標変換分散**: 統合システム外での座標・変形処理の実装
- **監視機能重複**: 独自デバッグ・監視機能の新規実装

### 10.3 Phase3移行期禁止事項
- **段階的移行無視**: 一度に複数システム統合変更禁止
- **互換性破壊**: 既存API削除による破壊的変更禁止
- **性能劣化放置**: 統合前後の性能測定・改善義務
- **エラーハンドリング省略**: 統合システム障害対応必須
